I Built a Slot-Filling Onboarding Bot. It Leaked a Validation Error Into the Chat翻訳スキップ

原文説明（英語）

<p>I was building an onboarding flow: a new user lands, a chat window asks them three things - what problem they're solving, what their idea is, who the user is - and once all three are answered, the app needs <code>{ problem, idea, targetUser }</code> as a clean, validated object. Not a transcript. Not "probably in there somewhere." An actual object my code could pass to the next step without re-reading prose.</p> <p>My first pass was the obvious one: after every user reply, try to extract the object, check if it validated, and if it didn't, ask the model for a follow-up question. It worked right up until the extraction step threw partway through a conversation - a <code>SchemaViolationError</code> with actual validation details in it - and I hadn't wrapped that path carefully enough. The raw error text got forwarded straight into the chat window as if it were the bot's next question. A user saw <code>"idea": expected string, received undefined</code> where a friendly follow-up question should have been.</p> <p>I fixed that specific leak with a try/catch. Then found a second one a day later in a different code path. That's when I stopped patching leaks one at a time and looked at whether the tool I was already using (<a href="https://github.com/aviasoletechnologies/shapecraft" rel="noopener noreferrer">shapecraft</a>) had a mode built for exactly this shape of problem, instead of me re-deriving "never let internal errors reach the user" by hand, one bug at a time.</p> <h2> The actual design problem </h2> <p>It's not really "extract after every message and retry." It's two completely different concerns that I'd tangled into one loop:</p> <ol> <li> <strong>Having a conversation</strong> - a model asking natural questions, probing vague answers, deciding when it has enough. That's chat. It should never be schema-constrained, because forcing JSON-shaped output mid-conversation is exactly how you get a bot that "helpfully" outputs a stray <code>{...}</code> into the chat.</li> <li> <strong>Producing a validated object, once, at the end</strong> - a completely separate concern from the conversation itself.</li> </ol> <p>Conflating them means every turn is a place validation <em>could</em> fail and leak. Separating them means there's exactly <strong>one</strong> place validation happens - after the conversation is over - and exactly one rule to enforce: nothing generated internally at that step is ever routed back into the chat.</p> <h2> <code>turnaround</code> mode </h2> <p><code>generate(..., { turnaround: true })</code> is built around that split. The model drives the conversation entirely through <code>systemPrompt</code> - what to ask, when to probe a vague answer, when it's satisfied. Shapecraft's only job during collection is to relay the model's reply and carry the transcript forward:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="k">import</span> <span class="p">{</span> <span class="nx">generate</span> <span class="p">}</span> <span class="k">from</span> <span class="dl">"</span><span class="s2">@aviasole/shapecraft</span><span class="dl">"</span><span class="p">;</span> <span class="kd">const</span> <span class="nx">FACILITATOR</span> <span class="o">=</span> <span class="dl">"</span><span class="s2">Ask exactly one focused question at a time to learn: (1) what problem are we solving, </span><span class="dl">"</span> <span class="o">+</span> <span class="dl">"</span><span class="s2">(2) what the idea is, (3) who the user is. That's the trigger for a single, separate <code>generate()</code> call - a normal structured extraction pass over the whole transcript, not part of the conversation the user sees. I liked this over having the model just emit JSON directly once it thinks it's done, for the same reason my original leak happened: raw or premature JSON has no business anywhere near the user-facing chat, and "is this the sentinel string" is a trivial check instead of a fragile "does this look like JSON" guess.</p> <h2> The rule that actually fixes my bug </h2> <p>Every response from a <code>turnaround</code> call is one of two shapes: <code>{ status: "collecting", message }</code> where <code>message</code> is <em>only ever the model's own conversational text</em>, or <code>{ status: "complete", data }</code> where <code>data</code> is only ever produced by the end-of-conversation extraction pass. There is no third path where something shapecraft generated internally ends up in <code>message</code>. If that end-of-conversation extraction fails validation, it throws a real, terminal error out of <code>generate()</code> - it does not get wrapped into a fake follow-up question, and the user is never re-interrogated for answers already sitting in the transcript.</p> <p>That last part surprised me at first - I expected "validation failed, so ask again." But if the user's answers are already in the transcript, a validation failure at the end means <em>my</em> extraction step is broken, not that the user needs to repeat themselves. Re-asking would be asking the wrong party to fix the wrong problem.</p> <h2> Where that leaves it </h2> <p>My onboarding bot now has exactly one place structured data gets produced, and exactly zero places an internal error can masquerade as a chat message:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="k">import</span> <span class="p">{</span> <span class="nx">generate</span> <span class="p">}</span> <span class="k">from</span> <span class="dl">"</span><span class="s2">@aviasole/shapecraft</span><span class="dl">"</span><span class="p">;</span> </code></pre> </div> <p>Repo's at <a href="https://github.com/aviasoletechnologies/shapecraft" rel="noopener noreferrer">github.com/aviasoletechnologies/shapecraft</a>, package is <code>@aviasole/shapecraft</code> on npm. If you've ever had a slot-filling bot leak something internal into the chat window - curious how you caught it.</p>
