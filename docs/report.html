Choosing an RDP Client on macOS:What I Learned After Managing Windows Machines from a Mac
原文説明
原文説明（英語）

<p>If you move from Windows to macOS and still need to access Windows servers or desktops, one of the first problems you will hit is remote access.</p> <p>Many developers are familiar with Microsoft's Remote Desktop client (<code>mstsc</code>) on Windows and expect a similar built-in workflow on Mac. macOS does not include a native RDP client, so you need a third-party solution.</p> <p>After trying different approaches, I found that the choice depends heavily on how you use remote connections: connecting to one Windows machine occasionally is simple, while managing multiple servers and protocols requires a different workflow.</p> <h2> Microsoft's Remote Desktop Client: The Obvious Starting Point </h2> <p>Microsoft provides an official RDP client for macOS through Windows App.</p> <p>For pure RDP access, it works well:</p> <ul> <li>Connect to Windows desktops and servers</li> <li>Support common RDP features</li> <li>Provide a familiar Microsoft ecosystem experience</li> </ul> <p>If your only requirement is "connect my Mac to one Windows machine", this may already solve the problem.</p> <p>However, real development and operations workflows often become more complicated.</p> <h2> Where a Dedicated RDP Client Workflow Starts Showing Limits </h2> <p>When managing multiple remote machines, a few issues become noticeable.</p> <h3> Server management becomes difficult at scale </h3> <p>A few saved connections are easy to handle. Once you have many Windows servers, development machines, test environments, and customer systems, organizing them becomes less convenient.</p> <p>Remote connection tools are often used by developers who need dozens of saved sessions, not just one personal computer.</p> <h3> Hardware compatibility can become a concern </h3> <p>Some older Mac devices using Intel processors may face compatibility limitations depending on the software version and future support direction.</p> <p>The Apple Silicon transition has changed the macOS application landscape significantly, and long-term compatibility matters for tools you rely on every day.</p> <h3> RDP is only one part of the workflow </h3> <p>Developers rarely use only RDP.</p> <p>A typical remote workflow may include:</p> <ul> <li>SSH into Linux servers</li> <li>RDP into Windows environments</li> <li>VNC for remote desktops</li> <li>SFTP for file transfer</li> <li>Serial connections for hardware devices</li> <li>Database management</li> </ul> <p>Using separate applications for every protocol increases context switching and makes daily operations more fragmented.</p> <h2> Why Some Developers Prefer Unified Remote Management Tools </h2> <p>For users managing multiple environments, a unified client can simplify the workflow.</p> <p>This was one of the reasons I built DartShell. The goal was to combine common remote protocols into one macOS application instead of forcing developers to maintain a collection of separate tools.</p> <p>For example, a typical day might involve:</p> <ul> <li>SSH into a Linux server to check logs</li> <li>Open an RDP session to a Windows build machine</li> <li>Transfer files through SFTP</li> <li>Manage another device through VNC</li> </ul> <p>Keeping these workflows together reduces the friction of switching between different applications.</p> <h2> RDP Performance Still Matters </h2> <p>Remote desktop experience depends on more than just connection support.</p> <p>Things that affect daily usability include:</p> <ul> <li>Image quality</li> <li>Input latency</li> <li>Clipboard behavior</li> <li>Connection stability</li> <li>Multi-monitor support</li> </ul> <p>A remote desktop tool can technically connect successfully but still feel uncomfortable during long development sessions.</p> <p>For Windows users moving to Mac, choosing an RDP client is less about finding a single "best" application and more about matching the tool to the way you work.</p> <p>If you only need occasional Windows access, Microsoft's official client may be enough. If you manage multiple machines and multiple remote protocols every day, a unified remote management workflow can save a lot of time.</p> <p>What remote access workflow do you use on macOS: a dedicated RDP client, or one tool that handles multiple protocols?</p>

buildline: merging cargo and ninja's build profiling into one timeline
原文説明
原文説明（英語）

<p>Every build tool profiles its own silo. <code>cargo build --timings</code> will tell you cargo is fine. Ninja's build log will tell you linking is fine. And yet a CI build takes 22 minutes and nobody can point at where the time actually went, because none of these tools know the others exist.</p> <p>I built <strong>buildline</strong> to fix that: it merges the profiling output your build tools already produce into one wall-clock timeline, emitted as standard <strong>Chrome Trace Event Format</strong> so it opens directly in Perfetto - no UI to build or maintain.</p> <h2> The gap nobody profiles </h2> <p>Here's the part I didn't expect going in. A tiny <code>ninja</code> build finishes in under a second. A real <code>cargo build</code> starts about 2.7 seconds later. Neither tool's own profiler records that gap — ninja doesn't know cargo exists, and cargo's own timer starts from zero the moment it launches. On a merged timeline, it's just visibly empty space, exactly where the surprise usually is in a real CI log: toolchain downloads, dependency resolution, environment setup, all invisible to every tool's own instrumentation because it happens around them, not inside them.</p> <p>That's the actual thesis of the project: the interesting bottleneck in a multi-tool build is often the join between tools, not inside any one of them.</p> <h2> How it avoids becoming a fragile process-tree hack </h2> <p>The obvious design for "profile a build made of several tools" is an orchestrator that wraps the whole build and somehow intercepts every child process it spawns. I didn't want that — process interception is<br /> platform-specific, fragile against anything that isn't a direct child<br /> (containers, remote shells, build systems that re-exec themselves), and it turns a normalization tool into a ptrace project.</p> <p><code>buildline</code> wraps each tool invocation instead, correlated through a shared session file:</p> <p><code>BUILDLINE_SESSION=./build.trace buildline -- ninja<br /> BUILDLINE_SESSION=./build.trace buildline -- cargo build</code><br /> <code># open build.trace in &lt;https://ui.perfetto.dev&gt;</code><br /> ​<br /> Each invocation is transparent — inherited stdout/stderr, the tool's real exit code passed through unchanged. <code>buildline</code> stamps the wall-clock instant right before it execs the wrapped command, then reads that tool's own profiling artifact after it exits (<code>.ninja_log</code>, cargo's <code>--timings</code> report), and appends the result to the shared trace file. There's no orchestration, no process-tree walking, and no finalize step — <code>build.trace</code> is a valid, openable Chrome Trace file after every single invocation.</p> <h2> The span schema, and the one design decision that matters most </h2> <p><code>pub struct Span {<br /> pub name: String, // "serde v1.0", "obj/parser.o"<br /> pub category: Category, // Compile | Link | Configure | Resolve | Download | Test | Other(String)<br /> pub status: Status, // Success | Failed | Skipped | Incomplete<br /> pub track: String, // "cargo", "ninja" — groups rows in Perfetto<br /> pub lane: u32, // sub-row for parallel work within a track<br /> pub start_us: i64, // relative to that tool's own start — no wall-clock here<br /> pub dur_us: i64,<br /> pub args: BTreeMap&lt;String, String&gt;,<br /> }</code><br /> ​<br /> Every adapter is a pure function: native tool output in, a <code>Vec&lt;Span&gt;</code> out, timestamps relative to that tool's own start. The wrapper is the only thing that knows about wall-clock time — it's a thin layer that offsets each adapter's relative spans onto the shared axis. Keeping wall-clock entirely out of the adapters is what makes golden-file testing possible at all: an adapter's output is 100% deterministic, so a fixture (a real <code>.ninja_log</code> or <code>--timings</code>report) always produces the exact same normalized JSON, with nothing timing-dependent to make the test flaky.</p> <p><code>Category</code> is a closed enum, not a free-form string, on purpose. <br /> A golden test only diffs an adapter against itself — nothing stops one adapter emitting <code>"compile"</code> and another emitting <code>"Compile"</code>, each passing its own test while the "unified" timeline is silently incoherent underneath. The enum is the actual contract that makes it one timeline instead of N timelines that happen to render next to each other. <code>Other(String)</code> is the escape hatch for genuinely tool-specific states — cargo's <code>run-custom-build</code> (running an already-compiled build script, as opposed to compiling one) isn't a compile step, so it stays <code>Other</code> rather than being folded into <code>Compile</code> and losing that distinction.</p> <p><code>Status</code> includes <code>Incomplete</code> for a reason I only appreciated after hitting it myself elsewhere: a span that started and never finished — the build that hung or got killed. That's not a "duration" problem, it's a "the last thing that happened before everything went sideways" problem, and no other Status variant models it.</p> <h2> Contributing a new adapter is a fixture pair, not a code review </h2> <p>Supporting a new build system (Bazel, Gradle, MSBuild, whatever isn't<br /> covered) is one adapter file plus a fixture pair: a real trace from that tool, and the exact normalized JSON it should produce. CI diffs the two — no subjective judgment call needed to review it. The best fixtures come from people running build systems I don't have access to myself (a real Bazel profile from a remote-execution setup, a real MSBuild trace on Windows) — that coverage isn't something I can produce alone.</p> <h2> Where it's at </h2> <p>Early: <strong>ninja</strong> and <strong>cargo</strong> adapters today, both covered by golden-file tests.<br /> Single-machine only for now — distributed/remote-execution builds are out of scope until there's a sane way to align clocks I don't control, and I'd rather say that upfront than have it be a surprise.<br /> The cargo adapter has one real fragility worth naming: stable cargo has no <code>--timings=json output</code>, so it reads the <code>UNIT_DATA</code> array embedded in the <code>--timings</code> HTML report — cargo's own dashboard data, not a documented or versioned format. It could change without notice in a future cargo release. If that golden test ever starts failing after a <code>rustup update</code>, that's why.</p> <p><code>cargo install buildline</code>​</p> <ul> <li>Repo: <a href="https://github.com/nabsei/buildline" rel="noopener noreferrer">https://github.com/nabsei/buildline</a> </li> <li>Crate: <a href="https://crates.io/crates/buildline" rel="noopener noreferrer">https://crates.io/crates/buildline</a> </li> </ul> <p><strong>Disclosure:</strong> built with the assistance of Claude Code. I made the design decisions described above, reviewed the implementation, and tested the result myself.</p> <p>Feedback very welcome — especially "this already exists, here's why it's<br /> better," since I'd rather know that now.</p>

<p>各ビルドツールはそれぞれ独自のプロファイリングを行っています。`cargo build --timings` を実行すれば Cargo が正常であることが分かります。Ninja のビルドログを見ればリンクが正常であることが分かります。しかし、CI ビルドに 22 分もかかっているのに、その時間がどこに費やされたのか誰も特定できません。なぜなら、これらのツールは互いの存在を認識していないからです。</p> <p>この問題を解決するために、私は `buildline` を開発しました。ビルドツールが既に生成しているプロファイリング出力を、標準の `Chrome Trace Event Format` として出力される単一のタイムラインに統合します。

要約
