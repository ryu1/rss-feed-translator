Local-first RAG for privileged legal documents: why citations need verification
原文説明
原文説明（英語）

<p>Lawyers want the same thing everyone else wants from AI: ask a question, get a useful answer, move faster.</p> <p>The difference is that legal documents often come with privilege, confidentiality, protective orders, client expectations, and professional responsibility duties. Uploading a contract set, deposition transcript, pleading file, or discovery bundle into a generic cloud chatbot is not a neutral implementation detail. A vendor sits in the data path.</p> <p>That is the design problem behind <a href="https://github.com/janderswag/docuchat.app" rel="noopener noreferrer">DocuChat</a>: an open-source, local-first document chat app for attorneys.</p> <p>The core idea is simple:</p> <blockquote> <p>Put the AI next to the documents, not the documents next to the AI vendor.</p> </blockquote> <h2> The architecture </h2> <p>DocuChat is a self-hosted legal document RAG stack:</p> <ul> <li> <strong>Local models</strong> through Ollama</li> <li> <strong>FastAPI</strong> backend bound to loopback</li> <li> <strong>LanceDB</strong> embedded vector store</li> <li> <strong>PyMuPDF / Docling / Tesseract</strong> for PDFs, tables, and OCR</li> <li> <strong>Matter-scoped retrieval</strong> so one client/matter cannot bleed into another</li> <li> <strong>Verified page/span citations</strong> before answers are displayed</li> </ul> <p>The query path does not require OpenAI, Anthropic, Google, or any cloud AI API key. After local models are downloaded, the workflow can run offline.</p> <h2> Why ordinary citations are not enough </h2> <p>A citation-looking string is easy for an LLM to generate. That does not mean the cited text actually supports the claim.</p> <p>DocuChat treats citations as a mechanical verification problem. The model can propose an answer and source quote, but the app checks whether that quote actually appears in the retrieved source text at the claimed page/span. If a claim cannot be verified against the source, it is dropped instead of shown.</p> <p>That creates a more useful failure mode for legal work:</p> <ul> <li>Better: “I could not find that in the documents.”</li> <li>Worse: a confident answer with a fake or sloppy citation.</li> </ul> <h2> What this is not </h2> <p>DocuChat is not an AI lawyer. It does not give legal advice, draw legal conclusions, or act outside the user’s machine.</p> <p>It is a cited-retrieval assistant for document-heavy work: contracts, transcripts, scanned PDFs, exhibits, pleadings, and matter files.</p> <h2> Why open source matters here </h2> <p>For sensitive workflows, “trust us” is not enough. Legal technologists should be able to inspect the data path, citation logic, storage layer, and model boundaries.</p> <p>The repo is here:</p> <p><a href="https://github.com/janderswag/docuchat.app" rel="noopener noreferrer">https://github.com/janderswag/docuchat.app</a></p> <p>If you care about local-first AI for private legal documents, a GitHub star/watch helps validate that this is worth building further.</p>

A Folder of Docs Is Not a Knowledge Base
原文説明
原文説明（英語）

<blockquote> <p>Disclosure up front: I build <a href="https://agentproto.sh" rel="noopener noreferrer">agentproto</a> and its corpus<br /> tooling, which is what the walkthrough uses. The commands are real and<br /> checkable; the problem in the first half is one every RAG setup hits.<br /> Corrections welcome — file an issue.</p> </blockquote> <p>The advice is everywhere and it sounds complete: <em>give your agent your knowledge —<br /> just do RAG over your docs folder.</em> So you point a retriever at <code>docs/</code>, embed the<br /> chunks, and wire it up. The agent starts answering questions about your systems.</p> <p>Then you ask it something that matters, and it gives you a fluent, confident<br /> paragraph — and you have no idea whether it's true, because you have no idea where<br /> it came from. Was that from the current runbook or a design doc you abandoned in<br /> 2024? Is that retry number the real one, or a chunk from a stale README that<br /> happened to score well on cosine similarity? You can't tell. Neither can the<br /> agent.</p> <p>That's the tell that you don't have a knowledge base. You have a search index over<br /> a pile of documents, which is a different and much weaker thing.</p> <blockquote> <p><strong>The one idea, if you remember nothing else:</strong><br /> A folder of docs is not a knowledge base. What makes it one is <em>provenance</em> —<br /> every fact knowing which source it came from, so the agent can cite it and you<br /> can check it.</p> </blockquote> <h2> The two things a doc dump can't do </h2> <p>A folder of markdown fails as a knowledge base for two structural reasons, and no<br /> amount of better embeddings fixes either.</p> <p><strong>It retrieves prose, not facts.</strong> A document is a wall of sentences about many<br /> things at once. Chunk it and you get arbitrary slices — half a paragraph about<br /> retries glued to the start of a paragraph about logging. The agent gets <em>near</em> the<br /> answer and reconstructs the rest, which is exactly where confident-but-wrong comes<br /> from. The unit of knowledge should be a <em>claim</em>, not a 512-token window.</p> <p><strong>It forgets where everything came from.</strong> Once a chunk is embedded, it's just a<br /> floating string. There's no thread back to "this is from <code>runbooks/payments.md</code>,<br /> last touched in the April incident." So the agent can't cite, and — worse — <em>you</em><br /> can't audit. A knowledge base you can't audit is a knowledge base you can't trust,<br /> which means it's a liability the first time it's confidently wrong.</p> <p>Both problems have the same fix, and it's not a fancier retriever. It's a<br /> <strong>pipeline</strong>: turn sources into atomic, sourced claims <em>before</em> anything queries<br /> them. That pipeline is what separates a corpus from a folder.</p> <h2> What's your "knowledge base" actually made of? </h2> <p>Place yourself before you build. Four honest answers.</p> <p><strong>Nothing.</strong> The agent runs on pre-training and whatever you paste. Generic by<br /> default. <em>Ceiling: it never knows one thing that isn't already public.</em></p> <p><strong>A docs folder + RAG.</strong> You retrieve chunks by similarity. Better than nothing,<br /> and it's the trap this piece is about: no provenance, no dedup, no notion of a<br /> claim — it answers from sources it can't name. <em>Ceiling: confident, unverifiable,<br /> and stale the day a doc rots.</em></p> <p><strong>A hand-curated wiki.</strong> Real, human-maintained, often good — but it's written for<br /> <em>people</em> to read, not for an agent to query claim-by-claim, and it goes stale<br /> between the edits nobody makes. <em>Ceiling: high quality, low queryability, manual<br /> upkeep.</em></p> <p><strong>A distilled, sourced corpus.</strong> Sources imported with their origin attached,<br /> distilled to claim-level entries, each carrying its provenance, queryable by topic.<br /> <em>This is the one an agent can cite and you can audit — and it's four commands.</em></p> <blockquote> <p><strong>Where are you?</strong> If your honest answer is <em>"RAG over a docs folder,"</em> you have<br /> the confident-but-unverifiable setup — the most dangerous kind, because it <em>feels</em><br /> like it works right up until it states a stale fact with a straight face. The<br /> rest of this turns the pile into a corpus.</p> </blockquote> <h2> Step 1: import, and keep the receipt </h2> <p>Start by pulling your sources into a corpus workspace — and the non-negotiable is<br /> that each source keeps a thread back to where it came from.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>npm i <span class="nt">-g</span> @agentproto/corpus-cli <span class="c"># scaffold an AIP-10 workspace (generic research preset, no seed noise)</span> corpus init house-knowledge knowledge/house <span class="nt">--preset</span> research <span class="c"># import your sources; --tags is what makes the corpus queryable by topic later</span> corpus import-web knowledge/house <span class="nt">--urls-file</span> runbooks.txt <span class="nt">--tags</span> runbooks corpus import-web knowledge/house <span class="nt">--urls-file</span> incidents.txt <span class="nt">--tags</span> incidents </code></pre> </div> <p>Every imported source lands as an AIP-10 record carrying its <code>originalUrl</code> — the<br /> provenance thread. An article comes in via readability; a walled page routes<br /> through a browser; <strong>a recorded incident review or an architecture talk on video<br /> comes in via transcription</strong> — same workspace, same provenance, so a decision your<br /> team only ever said out loud becomes a citable source. Import is resumable: re-run<br /> it and it skips what's already in, so you grow the corpus instead of rebuilding it.</p> <h2> Step 2: distill — turn documents into claims </h2> <p>This is the step a folder-and-RAG setup skips entirely, and it's the one that<br /> matters. Distillation reads each source and emits <strong>atomic, claim-level entries</strong>,<br /> roughly seven per source, each one carrying the source it came from.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code><span class="c"># cheap + strong: distill on an open model via OpenRouter (saves frontier credits)</span> corpus distill knowledge/house <span class="nt">--engine</span> opencode <span class="nt">-m</span> openrouter/deepseek/deepseek-v4-pro </code></pre> </div> <p>Now the unit of knowledge is a claim, not a chunk — <em>"payment retries are capped at<br /> 3 with an idempotency key, per incident 2026-04"</em> is one entry with one source, not<br /> a slice of a paragraph that happened to embed nearby. When an agent retrieves this,<br /> it retrieves the fact <strong>and</strong> its origin, which is what lets it answer <em>"per<br /> incident 2026-04…"</em> instead of <em>"a common approach is…"</em>. Provenance stops being<br /> metadata and becomes the answer.</p> <blockquote> <p><strong>Why claims beat chunks, grounded.</strong> Anthropic's context-engineering guidance is<br /> to keep the working context small and retrieve high-signal facts <strong>just in time</strong><br /> rather than pre-loading walls of text. You can't do just-in-time retrieval on a<br /> document — it's too big and says too much. You can do it on a claim. Distillation<br /> is what makes your knowledge <em>retrievable at the right grain.</em></p> </blockquote> <h2> Step 3: gate the corpus — a knowledge base is maintained, not filled </h2> <p>A pile of docs is fill-once-and-forget. It isn't.</p> <p>Geoffrey Huntley named the mechanism precisely when he described the Ralph<br /> technique as <em>a deterministic loop wrapped around a non-deterministic model</em> —<br /> the bet being that repeated attempts converge on a working solution. It will never give up on the wrong path, either.</p> <h2> The wrong fix: a watchdog that restarts you into the same groove </h2> <p>The reflex, when a loop misbehaves, is to make it <em>more</em> persistent: auto-restart<br /> on stall, replay the last message in a fresh session, keep it alive through<br /> context overflow and API errors. That's a watchdog, and it's genuinely necessary<br /> plumbing for unattended runs — but it is the exact wrong tool for this problem.</p> <p>A watchdog keeps the process breathing and says nothing about whether the process<br /> is right. <strong>It will cheerfully restart a wrong loop into producing the same wrong<br /> answer until sunrise</strong> — the <a href="https://dev.to/agentiknet/your-agent-says-the-tests-passed-it-didnt-run-them-15j">supervision ladder</a><br /> calls this L1, and the reason it's the lowest automated rung is that "alive" and<br /> "correct" are different axes. It refuels it.</p> <blockquote> <p><strong>Alive ≠ on-track.</strong> Health-monitoring harnesses track context usage, idle<br /> time, and error counts, then act: clear context below a threshold, restart<br /> agents that stall, disable them after N consecutive errors (Claude Code Agent<br /> Farm, 2026). Every one of those signals is about <em>liveness</em>. None of them asks<br /> the only question that matters mid-flight: is this still the right thing to be<br /> building?</p> </blockquote> <p>So the fix can't be "keep it going better." They're about a different job than the one it's built for.</p> <p><strong>1. Supervision means you, faster — not you, removed.</strong> The relay moves the<br /> human decision to your pocket; it doesn't take the human out of the loop. There<br /> is no policy engine that fires <em>at the agent's turn-end</em> — nothing that says<br /> "when the agent claims done, run lint + tests, have a second model try to refute<br /> the diff, and stage the commit until a human acks." Your tools don't ride along.</strong> Paseo injects its own control tools into the<br /> agents it runs — that's how permit/chat/terminal reach the model — but there's<br /> no mechanism to register <em>your</em> tool once and have every agent (Claude, Codex,<br /> OpenCode) see it. Each provider keeps its own MCP config; the daemon isn't a<br /> tool bus. If you want one capability visible to every agent brand, that's still<br /> manual, per-provider labor with your name on it.</p> <p><strong>3. The daemon's own MCP surface is off by default.</strong> My first probe of<br /> <code>127.0.0.1:6767</code> returned 404s on the obvious MCP paths — because the MCP server<br /> is config-gated. But what it<br /> exposes is Paseo's <em>own</em> control tools, not a bus for distributing <em>your</em><br /> tools into the agents it runs. Point 2 still stands.</p> </blockquote> <p>None of that makes Paseo worse at its job. It makes it a different rung than the<br /> one I build for — which is the fair way to end a review a competitor wrote.</p> <h2> The verdict, and where it genuinely beats us </h2> <p><strong>Use Paseo if your bottleneck is *being physically present.</strong>* You run two to<br /> five Claude Code / Codex sessions, permission prompts are what shreds your day,<br /> and you want to approve them from the couch — with worktrees, schedules,<br /> structured output, and a phone app that, honestly, is nicer than most.
