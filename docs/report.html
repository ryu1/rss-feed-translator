Here's how I got there, what broke along the way, and how the architecture ended up.</p> </blockquote> <h2> Where I Started </h2> <p>I have a CLI tool called <strong>AppPulse</strong> that monitors my app's reviews and crash data. Every morning it:</p> <ol> <li><p>Pulls new reviews from Google Play / App Store</p></li> <li><p>Classifies each one with an LLM — bug, crash, feature request, performance, praise</p></li> <li><p>Correlates reviews with crash data from Sentry/Firebase</p></li> <li><p>Sends me a digest<br /> </p></li> </ol> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>apppulse run <span class="c"># → "12 new reviews: 3 bugs (1 critical), 2 feature requests, 7 praise"</span> apppulse reviews <span class="nt">--category</span> bug <span class="c"># ID │ Rating │ Summary │ Severity</span> <span class="c"># 42 │ ★☆☆☆☆ │ Photo upload crashes for &gt;10MB images │ critical</span> </code></pre> </div> <p>This was genuinely useful. I knew <em>what</em> users were complaining about and which crashes were affecting the most people. But the pipeline stopped at classification. When I saw "Photo upload crashes for &gt;10MB images," I still had to:</p> <ol> <li><p>Open the project in my IDE</p></li> <li><p>Search for upload-related code</p></li> <li><p>Cross-reference the stacktrace from Sentry</p></li> <li><p>Mentally map the review to the actual code</p></li> <li><p>Figure out what to change</p></li> </ol> <p>For one bug, that's fine. For a batch of 5 critical bugs on a Monday morning, it's a lot of manual work before I've even started fixing anything.</p> <p>I wanted the pipeline to go further — to take the classified review, point to the specific files and lines causing the issue, and hand me a brief I could act on immediately.</p> <h2> The First Version: A PydanticAI Agent with Codebase Tools </h2> <p>The core idea was simple: give an LLM agent access to my codebase through read-only tools and let it investigate the bug, just like I would.</p> <h3> Why PydanticAI </h3> <p>I considered LangGraph, LangChain, and rolling my own agent loop. 原文説明（英語）

<p><em>Originally published at <a href="https://nlqdb.com/blog/sitemap-advertising-redirects/?utm_source=devto" rel="noopener noreferrer">nlqdb.com/blog</a></em></p> <p>Run <code>curl -sI</code> over the URLs in your own sitemap. We did, and 27 of ours answered <strong>307 Temporary Redirect</strong> — every route on the site. The canonical tag was worse: each page declared a canonical URL that redirected to the page itself. We were telling crawlers, on every single page, that the authoritative copy lives one hop away.</p> <h2> How a static host quietly forks every URL in two </h2> <p>The mechanics are mundane, which is why this survives review. A static build writes each route as <code>route/index.html</code>. The host serves <code>/route/</code> directly — and answers the bare <code>/route</code> with a 307 to the slash form. So every route on the site has two URLs: one real, one a redirect. Which one your HTML advertises depends on whoever typed the link.</p> <p>Our templates typed the bare form everywhere it matters most: the <code>canonical</code> tag, <code>og:url</code>, the sitemap generator, and <code>llms.txt</code> — the four surfaces whose entire job is to name the authoritative URL. All four advertised the redirect.</p> <h2> Why a self-referential redirecting canonical is worth fixing </h2> <ul> <li> <strong>A canonical pointing at a redirect is a mixed signal.</strong> The tag exists to end URL ambiguity; making crawlers resolve a hop to find the 'canonical' copy reintroduces exactly the ambiguity it was meant to close.</li> <li> <strong>Sitemap URLs are supposed to be final.</strong> Search engines tolerate redirects there, but every hop is a chance for a URL to get indexed in the form you didn't pick — and split whatever authority the page earns across two spellings.</li> <li> <strong>AI crawlers fetch exactly what you wrote.</strong> <code>llms.txt</code> consumers and answer-engine bots are literal; an extra 307 on every fetch is latency you chose, on the surface built for machines.</li> </ul> <h2> The fix is a policy plus one helper, not a link hunt </h2> <p>Chasing individual links is the wrong altitude. The durable fix was three moves: pick the slash form as policy (<code>trailingSlash: "always"</code>, so the build itself rejects the other spelling); route every emitted URL — canonical, <code>og:url</code>, sitemap, <code>llms.txt</code> — through one path-normalize helper in the head layout and URL generators; and re-audit with <code>curl -sI</code> over every sitemap URL, expecting nothing but 200s.</p> <p>The one-helper rule is what made the follow-up cheap: once advertised URLs were normalized in one place, sweeping the remaining 1,100+ bare-path hrefs inside page bodies was a mechanical pass, and a link check in the build now fails on any regression. (<a href="https://nlqdb.com" rel="noopener noreferrer">nlqdb</a> is the database you talk to; this is one of the engineering notes from building it in the open.)</p> <p>The general lesson: every surface that names a URL — canonical, <code>og:url</code>, sitemap, <code>llms.txt</code>, internal hrefs — should call the same normalize function. If two of them can disagree about your own address, they eventually will, and you'll advertise the disagreement to every crawler that visits.</p>

<p>Venho usando agentes de código (Claude Code, Cursor) e bati numa parede que todo mundo conhece: o <strong>vibe coding</strong> — descrever a tarefa e deixar a IA cuspir código — escala mal. Funciona até a terceira mudança, quando ninguém mais lembra qual era a intenção original. O código virou a única fonte de verdade, e ele é péssimo pra contar <em>por que</em> algo foi feito.</p> <p>Foi aí que caí no <strong>Spec-Driven Development (SDD)</strong>: colocar a especificação <em>antes</em> do código. Na prática, a "spec" é um markdown versionado no repo — requisitos + critérios de aceite — que a IA lê antes de gerar qualquer linha.
