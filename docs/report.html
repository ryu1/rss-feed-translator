Build a Feedback Channel That Does Not Become Another Abandoned Inbox

原文説明（英語）

<p>A feedback button is easy to ship. A feedback system is harder.</p> <p>The difference is not the input form. It is what happens after someone presses <strong>Send</strong>:</p> <ul> <li>Was the report stored before the API responded?</li> <li>Did it include enough context to investigate?</li> <li>Who owns the next action?</li> <li>Can the team reply through the original channel?</li> <li>What happens when notifications fail?</li> <li>How do old reports become visible before they are forgotten?</li> </ul> <p>This tutorial builds a small but operational feedback workflow with TypeScript, Express, and PostgreSQL. The same design works for bug reports, feature requests, support questions, and in-product conversations.</p> <h2> Start with the workflow, not the widget </h2> <p>A useful feedback lifecycle can be modeled as:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>captured -&gt; new -&gt; acknowledged -&gt; planned/resolved/closed | +-&gt; needs_more_information </code></pre> </div> <p>Every active item should have three properties:</p> <ol> <li> <strong>An owner</strong> responsible for the next decision.</li> <li> <strong>A next-action timestamp</strong> that makes inactivity queryable.</li> <li> <strong>A reply path</strong> when the reporter expects a response.</li> </ol> <p>Without those properties, a unified inbox merely creates a unified place to ignore.</p> <p>The system we will build has four parts:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Browser or app | v Feedback API -----&gt; PostgreSQL | v Transactional outbox | v Notification worker | v Chat, email, or issue tracker </code></pre> </div> <p>PostgreSQL is the source of truth. Use a strict schema, field length limits, and an allowlist.</p> <h3> Generating a new ID on every retry </h3> <p>This produces duplicates during temporary network failures. Reuse one idempotency key for all attempts of the same submission.</p> <h3> Promising real-time support without staffing it </h3> <p>A live widget changes user expectations. Display expected availability or use asynchronous wording when immediate responses are not guaranteed.</p> <h3> Mixing feedback directly into the roadmap </h3> <p>Reports are evidence, not automatically requirements. Keep capture, triage, and product prioritization as separate steps.</p> <h3> Closing an item without responding </h3> <p>If the user expects a reply, an internal status update is insufficient. <p>Image optimization often fails because a team treats it as a final cleanup task. Someone notices that a page feels slow, compresses a few visible files, and moves on. New screenshots and product images then arrive with no shared limit, so the page becomes heavy again.</p> <p>A more durable approach is to create an image budget: a small set of measurable rules for file size, dimensions, format, and visual acceptance. The budget turns compression from an occasional rescue operation into a repeatable publishing decision. It also gives writers, designers, and developers the same definition of “ready.”</p> <p>This guide explains how to create that workflow for documentation, landing pages, release notes, and other content where images must stay useful while loading efficiently.</p> <h2> Start with the job each image must do </h2> <p>Do not begin by choosing one universal size limit. First classify the image by its purpose. A hero photograph, a product screenshot, a small logo, and a diagram lose value in different ways.</p> <p>The <a href="https://www.w3.org/WAI/tutorials/images/decision-tree/" rel="noopener noreferrer">W3C image decision tree</a> separates informative, decorative, functional, and text-containing images because their roles affect how they should be presented. A decorative background can usually tolerate more aggressive reduction because it does not carry essential meaning.</p> <p>Create a short inventory before setting limits:</p> <ul> <li>Hero and marketing images support visual identity and first impressions.</li> <li>Interface screenshots explain a procedure or prove that a feature exists.</li> <li>Diagrams communicate relationships that may be difficult to express in prose.</li> <li>Thumbnails help people scan a collection but are not usually inspected in detail.</li> <li>Logos and icons depend on clean edges, transparency, and consistent dimensions.</li> </ul> <p>This inventory prevents a common mistake: applying the same quality setting to every file. The target is not identical compression. The target is consistent usefulness within a predictable transfer budget.</p> <h2> Define budgets at the page level first </h2> <p>An individual image can appear reasonably small while the complete page remains expensive. Ten files at 300 KB each still create roughly 3 MB of image transfer before scripts, styles, fonts, or video are counted. Start with the total page experience, then divide that allowance among assets.</p> <p>For a documentation article, you might decide that all initially visible images should stay below a combined amount, while screenshots further down the page can load later. Use those characteristics as constraints rather than treating the newest format as an automatic winner.</p> <p>PNG remains useful for lossless graphics, transparency, and sharp synthetic content. JPEG is widely supported and often effective for photographs, but it can introduce visible artifacts around hard edges. WebP can provide lossy or lossless output and works well for many mixed-content images.
