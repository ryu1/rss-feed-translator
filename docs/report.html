<div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code> **Python Selenium Architecture** </code></pre> </div> <p><strong>Introduction:</strong><br /> Selenium automates web browsers. The Python Selenium architecture consists of four main components that work together to control a browser.</p> <p><strong>1.Selenium Client Library (Python Language Binding)</strong></p> <ul> <li><p>Python developers write automation scripts using the standard Selenium API.</p></li> <li><p>This library converts your Python code into a standardized format.</p></li> <li><p>It sends these commands as programmatic requests to the browser driver.</p></li> </ul> <p><strong>2.W3C WebDriver Protocol/JSON Wire Protocol</strong></p> <ul> <li><p>This is the communication channel between the code and the driver.</p></li> <li><p>Historically, Selenium used the JSON Wire Protocol over HTTP.</p></li> <li><p>Modern Selenium (Version 4+) uses the standardized W3CWebDriver Protocol.</p></li> <li><p>Commands and responses are transferred directly without any middle translation.</p></li> </ul> <p><strong>3.Browser Drivers</strong></p> <ul> <li><p>Every web browser has its own specific executable driver.</p></li> <li><p>Examples include ChromeDriver for Chrome and GeckoDriver for Firefox.</p></li> <li><p>The driver acts as a secure HTTP server that receives commands.</p></li> <li><p>It passes these requests directly to the browser and returns results.</p></li> </ul> <p><strong>4.Web Browsers</strong></p> <ul> <li><p>This is the final layer where execution happens physically.</p></li> <li><p>Supported browsers include Google Chrome, Mozilla Firefox, Microsoft Edge, and Safari.</p></li> <li><p>The browser receives commands through its native OS-level API.</p></li> <li><p>It executes actions like clicking, typing, or fetching text.</p></li> </ul> <p><strong>Significance of Python Virtual Environments:</strong></p> <p>A Python Virtual Environment is an isolated directory containing its own Python installation and independent packages. It prevents dependency conflicts across different software projects.</p> <p><strong>Conclusion: Why It Matters?</strong></p> <ul> <li><p>Avoids Dependency Hell: Different projects can use different versions of the same library.</p></li> <li><p>Protects System Python: Prevents breaking system-wide packages required by your operating system.</p></li> <li><p>Ensures Reproducibility: Allows developers to easily recreate the exact environment on other machines.</p></li> <li><p>No Admin Privileges Needed: Allows installing packages without sudo or administrator rights.</p></li> </ul> <p><strong>Real-World Example:</strong><br /> Example 1: The Version Conflict Scenario</p> <p>Imagine you are managing two automation projects on the same computer:</p> <ul> <li><p>Project A (Legacy App): Built years ago. It requires selenium==3.141.0 to work with older tests.</p></li> <li><p>Project B (Modern App): Needs selenium==4.20.0 to use new relative locators.</p></li> </ul> <p>Without a virtual environment, installing Selenium 4 globally will overwrite Selenium 3, completely breaking Project A. Virtual environments keep them safely separated.</p>

EFS is like a shared Google Drive folder that your entire team can open simultaneously from different computers, see the same files, and edit them in real time.</p> <p><strong>Key characteristics:</strong></p> <ul> <li>Uses the <strong>NFS 4.1 protocol</strong> (Network File System — standard Linux file sharing)</li> <li> <strong>Serverless</strong> — no capacity to provision, no servers to manage</li> <li> <strong>Auto-scales</strong> — grows from KB to PB automatically, shrinks when files are deleted</li> <li> <strong>Highly available</strong> — data replicated across multiple AZs in a Region</li> <li> <strong>Pay-as-you-go</strong> — billed per GB actually stored, no pre-provisioning</li> </ul> <h2> EFS vs EBS vs S3 — The Big Comparison </h2> <p>This is one of the most commonly tested comparisons in AWS certifications.</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Feature</th> <th>EBS</th> <th>EFS</th> <th>S3</th> </tr> </thead> <tbody> <tr> <td><strong>Access</strong></td> <td>Single EC2 instance only</td> <td>Multiple EC2 instances simultaneously</td> <td>Internet / API, unlimited clients</td> </tr> <tr> <td><strong>Protocol</strong></td> <td>Block storage</td> <td>NFS v4.1</td> <td>HTTP/REST (object storage)</td> </tr> <tr> <td><strong>Scope</strong></td> <td>AZ-specific</td> <td>Regional (multi-AZ)</td> <td>Region-wide, globally accessible</td> </tr> <tr> <td><strong>Scaling</strong></td> <td>Manual (resize volume)</td> <td>Automatic</td> <td>Unlimited, automatic</td> </tr> <tr> <td><strong>Performance</strong></td> <td>Very high, low latency</td> <td>High (shared)</td> <td>Variable</td> </tr> <tr> <td><strong>Use Case</strong></td> <td>OS disks, databases</td> <td>Shared content, CMS, ML training</td> <td>Backups, static assets, objects</td> </tr> <tr> <td><strong>Cost Model</strong></td> <td>Provisioned (pay for allocated size)</td> <td>Usage-based (pay for what's stored)</td> <td>Usage-based</td> </tr> </tbody> </table></div> <p><strong>Simple decision rule:</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Need a disk for ONE EC2 instance (like an OS drive)? → EBS Need a shared folder MULTIPLE EC2 instances can access? → EFS Need to store files accessible from anywhere via internet? → S3 </code></pre> </div> <blockquote> <p>🎯 <strong>Interview Tip:</strong> "Can two EC2 instances share an EBS volume?" — Only with io1/io2 Multi-Attach, and even then both must be in the same AZ. EFS is the natural answer when the question is about shared access across multiple instances in different AZs.</p> </blockquote> <h2> EFS Architecture </h2> <p>EFS spans <strong>multiple Availability Zones</strong> within a Region. Here's how it's structured:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code> Amazon EFS File System (stored redundantly across AZs) │ ┌─────────────────────┼─────────────────────┐ │ │ │ AZ 1 AZ 2 AZ 3 Mount Target Mount Target Mount Target (ENI + IP) (ENI + IP) (ENI + IP) │ │ │ EC2 Instance EC2 Instance EC2 Instance /mnt/efs /mnt/efs /mnt/efs All instances access the SAME file system via NFS (port 2049) </code></pre> </div> <p><strong>Mount Target</strong> — this is the key piece. It's an <strong>ENI (Elastic Network Interface)</strong> that EFS creates inside each AZ of your VPC. It's the "doorway" through which EC2 instances in that AZ connect to EFS over NFS.</p> <ul> <li>One mount target per AZ (auto-created when you set up EFS)</li> <li>Has its own private IP and security group</li> <li>EC2 instances in that AZ connect to the <strong>local mount target</strong> for best performance</li> </ul> <h2> Mounting — What It Actually Means </h2> <p><strong>Mounting</strong> is the process of attaching a file system to a directory on your OS so it behaves like a normal local folder.</p> <p><strong>Analogy:</strong> Mounting EFS is like plugging in a USB drive and having it appear as a folder you can open, save files into, and read from — except this "USB drive" lives in AWS and multiple computers can plug into it at once.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code><span class="c"># Before mounting: /mnt/efs is just an empty folder</span> <span class="c"># After mounting: /mnt/efs IS the EFS file system</span> <span class="nb">sudo </span>mount <span class="nt">-t</span> nfs4 <span class="nt">-o</span> <span class="nv">nfsvers</span><span class="o">=</span>4.1 &lt;EFS-DNS-Name&gt;:/ /mnt/efs </code></pre> </div> <p>Once mounted, every <code>ls</code>, <code>cat</code>, <code>echo &gt;</code>, <code>cp</code> command on <code>/mnt/efs</code> is actually reading/writing to the shared EFS file system — not local disk.</p> <h2> Mount Targets &amp; Security Group Configuration </h2> <p>This is the part that trips people up in labs — <strong>two separate security groups working together.</strong></p> <h3> The Two Security Groups </h3> <p><strong>EC2 Security Group (<code>lab-ec2-sg</code>)</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Inbound: SSH (22) from My IP Outbound: NFS (2049) to EFS-SG [allows EC2 to reach EFS] </code></pre> </div> <p><strong>EFS Security Group (<code>lab-efs-sg</code>)</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Inbound: NFS (2049) from lab-ec2-sg [only EC2 instances can reach EFS] Outbound: Allow all (default) </code></pre> </div> <p><strong>Why two security groups referencing each other?</strong></p> <p>This is a common AWS pattern — instead of allowing an IP range, you allow <strong>traffic from a specific security group</strong>. This means: "Only instances that have the EC2 security group attached are allowed to talk to EFS on port 2049." Clean, scalable, and secure — you never have to know or update IP addresses.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>EC2 Instance (has lab-ec2-sg) │ │ NFS traffic, port 2049 ▼ EFS Mount Target (has lab-efs-sg, allows inbound from lab-ec2-sg) │ ▼ Access Granted ✅ </code></pre> </div> <blockquote> <p>⚠️ <strong>Common lab mistake:</strong> Creating the EFS file system and trying to mount immediately — it fails because the EFS security group doesn't yet allow NFS traffic from the EC2 security group. You must add that inbound rule first.</p> </blockquote> <h3> Mount Target Requirements </h3> <ul> <li>One mount target per AZ in your VPC</li> <li>Requires a subnet assignment</li> <li>Has a private IP auto-assigned</li> <li>DNS-based access — EFS gives you one DNS name that automatically resolves to the correct local mount target for whichever AZ you're connecting from</li> </ul> <h3> Persistence Across Reboots </h3> <p>By default, if your EC2 instance restarts, the EFS mount is <strong>gone</strong> — you'd have to mount it manually again. To make it automatic:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code><span class="c"># Add to /etc/fstab</span> &lt;EFS-DNS-Name&gt;:/ /mnt/efs nfs4 defaults,_netdev 0 0 </code></pre> </div> <p>The <code>_netdev</code> option tells the OS to wait for networking to be ready before attempting the mount — critical since EFS is a network resource, not local disk.</p> <h2> EFS Storage Classes </h2> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Storage Class</th> <th>Use Case</th> <th>Cost</th> </tr> </thead> <tbody> <tr> <td><strong>Standard</strong></td> <td>Active data, accessed frequently</td> <td>Higher per GB</td> </tr> <tr> <td><strong>Infrequent Access (IA)</strong></td> <td>Files not accessed often</td> <td>~50% cheaper per GB + a per-access fee</td> </tr> </tbody> </table></div> <p><strong>How it works:</strong> Standard is the default — best for files you're actively using. IA is for files that just sit there most of the time (old logs, archived reports) — cheaper to store, but costs a small fee each time you access them. The trade-off only makes sense if access is genuinely rare.</p> <h2> EFS Lifecycle Management &amp; Policies </h2> <p>This is <strong>automatic tiering</strong> — EFS moves files between Standard and IA based on actual usage, without you lifting a finger.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>File created → Standard storage (full price) │ │ Not accessed for X days (your chosen threshold) ▼ Auto-moved to Infrequent Access (cheaper) │ │ Someone accesses the file again ▼ Auto-moved back to Standard </code></pre> </div> <p><strong>Configurable transition periods:</strong> 7, 14, 30, 60, or 90 days of inactivity (or disabled entirely).</p> <p><strong>Why this matters:</strong> You don't have to manually audit and move files — EFS handles cost optimization automatically based on real access patterns. Set it once during file system creation and forget about it.</p> <blockquote> <p>🎯 <strong>Cert Perspective:</strong> "How do you reduce EFS storage cost for rarely accessed files without manual intervention?" → Enable Lifecycle Management with a transition policy to IA.</p> </blockquote> <h2> EFS Performance Modes &amp; Throughput Modes </h2> <p>These are two <strong>separate</strong> settings that often get confused — performance mode is about latency/concurrency, throughput mode is about data transfer speed.</p> <h3> Performance Modes (set once, cannot change later) </h3> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Mode</th> <th>Optimized For</th> <th>Use Case</th> </tr> </thead> <tbody> <tr> <td> <strong>General Purpose</strong> (default)</td> <td>Lowest latency</td> <td>Web serving, CMS, most workloads — use this unless you have a specific reason not to</td> </tr> <tr> <td><strong>Max I/O</strong></td> <td>Maximum throughput, more concurrent operations</td> <td>Big data, media processing, highly parallel workloads</td> </tr> </tbody> </table></div> <h3> Throughput Modes (can change later) </h3> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Mode</th> <th>How it Works</th> <th>Use Case</th> </tr> </thead> <tbody> <tr> <td> <strong>Bursting</strong> (default)</td> <td>Throughput scales with file system size, bursts higher when needed</td> <td>Default choice, cost-effective for most workloads</td> </tr> <tr> <td><strong>Provisioned</strong></td> <td>You specify a guaranteed throughput, independent of storage size</td> <td>Predictable high-throughput needs regardless of how much is stored</td> </tr> <tr> <td><strong>Max I/O</strong></td> <td>Optimized for highest throughput and ops/second</td> <td>Heavy parallel processing</td> </tr> </tbody> </table></div> <blockquote> <p>🎯 <strong>Cert Perspective:</strong> Know that Performance Mode is locked at creation, but Throughput Mode can be changed afterward. A scenario like "small file system but needs guaranteed high throughput" points to Provisioned Throughput.</p> </blockquote> <h2> Benefits of EFS </h2> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Benefit</th> <th>What It Means</th> </tr> </thead> <tbody> <tr> <td><strong>Scalability</strong></td> <td>Grows from KB to PB automatically, zero downtime</td> </tr> <tr> <td><strong>High Availability</strong></td> <td>Data redundantly stored across multiple AZs</td> </tr> <tr> <td><strong>Durability</strong></td> <td>Multiple copies protect against component failure</td> </tr> <tr> <td><strong>Simple Integration</strong></td> <td>Standard Linux NFS interface — no app code changes needed</td> </tr> <tr> <td><strong>Cost-Optimized</strong></td> <td>IA storage class cuts costs ~50% for inactive files</td> </tr> </tbody> </table></div> <h2> Security &amp; Encryption </h2> <p><strong>Network Security:</strong></p> <ul> <li>Security groups control all access (as covered above)</li> <li>VPC isolation — EFS only accessible within your VPC by default</li> <li>NFS traffic always on port 2049</li> </ul> <p><strong>Data Encryption:</strong></p> <ul> <li> <strong>At rest</strong> — via AWS KMS, enable when creating the file system</li> <li> <strong>In transit</strong> — via TLS, add <code>-o tls</code> to your mount command</li> </ul> <p><strong>Access Control (3 layers):</strong></p> <ol> <li> <strong>File System Policy</strong> — IAM-style resource policy on the EFS itself (grant/deny access)</li> <li> <strong>POSIX file permissions</strong> — standard Linux permissions (<code>chmod</code>, <code>chown</code>) work exactly as expected</li> <li> <strong>IAM integration</strong> — controls who can call EFS API operations (create, delete, modify)</li> </ol> <p><strong>Best practices:</strong></p> <ul> <li>Always enable encryption when creating the file system (can't add it retroactively)</li> <li>Restrict security groups to required ports only — never open NFS to 0.0.0.0/0</li> <li>Use VPC security groups for isolation, exactly like the EC2-SG ↔ EFS-SG pattern above</li> </ul> <h2> EFS Pricing &amp; Cost Optimization </h2> <p><strong>What you pay for:</strong></p> <ul> <li>Standard Storage — per GB stored</li> <li>Infrequent Access — ~50% lower per GB, plus a small fee each time data is retrieved</li> </ul> <p><strong>Cost optimization checklist:</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>✅ Enable Lifecycle Management — auto-move inactive files to IA ✅ Choose sensible transition days based on actual access patterns ✅ Use Bursting Throughput (default) unless you specifically need Provisioned ✅ Monitor usage via CloudWatch metrics ✅ Clean up unused files regularly </code></pre> </div> <h2> 🧪 LAB — Full Hands-On Walkthrough </h2> <h3> Lab Architecture </h3> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code> VPC (Default) ┌─────────────────────────┐ │ Availability Zone │ │ ┌──────────┐ ┌──────────┐ │ │ EC2-1 │ │ EC2-2 │ │ │ /mnt/efs │ │ /mnt/efs │ │ └─────┬────┘ └─────┬────┘ │ │ NFS (2049) │ │ └──────┬───────┘ │ ▼ │ Amazon EFS │ (Mount Target) └─────────────────────────┘ Security Groups: EC2-SG → allows SSH (22) inbound from My IP EFS-SG → allows NFS (2049) inbound from EC2-SG only </code></pre> </div> <h3> Lab Step 1 — Launch 2 EC2 Instances </h3> <p>Repeat this entire process twice (<code>EC2-Instance-1</code> and <code>EC2-Instance-2</code>):<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>1. <code>--sandbox workspace-write</code> 與 <code>exec resume</code>（可多輪、可寫檔），但我刻意把它限縮成「出題者」，讓出題與施工不同源。誤把單次 read-only 模式當施工者用，會得到看似對、實際編不過的檔案。</li> <li> <strong>Grok</strong>：我用 headless、可寫檔模式跑 write→test→fix。踩到一個<strong>參數相容性</strong>的坑：在 <code>grok 0.2.77</code> 用 <code>grok-composer-2.5-fast</code> 傳 <code>--effort</code> 時，經該 CLI 的 API 路徑回 <code>400 Bad Request：invalid-argument: Model grok-composer-2.5-fast does not support parameter reasoningEffort</code>，整輪空轉、零檔案寫入。這是<strong>該 model 不吃這個參數</strong>，不是 Grok CLI 的限制——<code>grok 0.2.77</code> 本身有 <code>--effort</code>；換支援的 model 就沒事。</li> <li> <strong>Claude</strong>：負責整合與驗收，因為它能在同一個 session 內持續持有上下文、跑工具、比對 diff。</li> </ul> <p>integration 成本比模型能力更早成為瓶頸：真正卡我的不是智力，而是一個 Zig 細節——單純 <code>pub const x = @import("x.zig")</code> 的 re-export，若沒被任何 test path 參照，Zig 的 lazy discovery <strong>不會</strong> discover 該檔的測試；要在 root 的 <code>test {}</code> 區塊加 <code>_ = x;</code> 強制 discovery。這種 integration 細節，才是 pipeline 真正的 latency 來源。</p> <h3> 為什麼「獨立驗收」不是形式：三個測試沒抓到、但驗收抓到的錯 </h3> <p>「Claude 驗收」聽起來像蓋章，但它擋下的都是綠燈下的暗傷：</p> <ul> <li> <strong>施工者謊報成功。</strong> 有一次 Grok 回報「成功」，實際上它跑在錯的目錄、測試本來就綠，它一行沒寫。只看回報就會提交一個沒改的 commit。</li> <li> <strong>測試全綠、但 migration 靜默跳過。</strong> 在 Rust 那輪，一段 schema migration 的 idempotent 守衛檢查「DDL 是否含 <code>destination</code>」就跳過——但線上 DB 早就有 <code>destination</code>、缺的是新欄位，於是守衛誤判、整段擴充被<strong>無聲略過</strong>。測試在本地全綠；我是讀 diff、再對線上 DDL 才抓到。</li> <li> <strong>seed 檔只載入一半、無錯誤。</strong> 一個 seed 的值字面含分號，而 seed splitter 正是以分號切句——三列只進了一列，沒有任何報錯。不是測試抓到的，是我核對「實際列數」才發現。</li> </ul> <p>這三個都會通過「測試全綠 + 施工者自述成功」。<strong>驗收要做的不是相信綠燈，是去證明綠燈為真。</strong></p> <h3> 一個要先講的安全坑：別把密鑰餵進 argv </h3> <p>把工作交給施工 agent 時，我一度讓 runtime 把 <code>.env</code> 的 DB token 直接展開進 <code>grok -p '…'</code> 的命令列——結果那顆 JWT 就明擺在 process list 與本地 log 裡。沒有進 git、token 可重簽，實際暴露面很小；但<strong>密鑰進 argv／log 這個「類別」本身就是硬傷</strong>。交派時要給的是「去載入憑證」的指令（<code>source .env</code>／讓 CLI 自己 mint），永遠不要把字面值塞進命令列。</p> <h3> 成本與失敗場景（正面之外） </h3> <ul> <li> <strong>成本是每個角色的來回，不是一次 LLM 呼叫。</strong> n=2 個功能我花了十幾次 agent 呼叫在「出題→核對→RED→施工→驗收」，其中 1 次失敗交接（Grok 空轉），人工驗收每個功能約幾分鐘。對小功能，這比單 agent 直接寫慢。划算場景是：<strong>功能有明確測試 oracle，且你在意正確性甚於速度。</strong>（wall-clock 未逐步計時，屬已知量測缺口。）</li> <li> <strong>紅燈要「可診斷」。</strong> 若 stub 全部回傳 <code>error.NotImplemented</code>，所有測試會用同一種方式失敗——那是無資訊的 RED。每個測試必須因自己的原因失敗，施工者才知道往哪修。</li> <li> <strong>共享 DB 要 panic-safe 清理。</strong> 若整合測試打的是共享／線上 DB，teardown 必須用 RAII 守衛掛住——否則一次 RED（斷言 panic）就會把測試列洩進正式庫。這是 live-DB 專案的額外適用條件。</li> </ul> <h3> 換一個 stack 再驗一次 </h3> <p>為了看它是不是只在 Zig 成立，我在一個 Rust + Turso（雲端共享 DB）專案上用<strong>同一條</strong>「Codex 出測試→Grok 實作→Claude 驗收」獨立重跑了三個功能。沒有推翻原判斷：一樣可用，前提仍是測試能當契約；差別在共享 DB 帶出的新約束（上面的 panic-safe 清理、以及讀 diff 才抓到的靜默 migration bug）。<strong>換個語言、換個測試框架，卡點依舊在 integration 與驗收，不在模型智力。</strong> 這讓我更有信心，但仍只是第二個小樣本案例，不是 benchmark。</p> <h3> 適用與不適用 </h3> <p><strong>適用</strong>：有靜態型別 + 嚴格測試框架的專案、功能可切成小塊、每塊有明確斷言。測試當契約，跨 agent 交接才有意義。</p> <p><strong>不適用</strong>：探索性、規格未定、或「測試本身就是要設計的東西」的工作。這條 pipeline 假設測試是可先寫死的規格；當規格還在流動，強行分工只會把來回成本放大。</p> <h3> 判斷 </h3> <p>這不是新範式，是把單 agent 的一個內部步驟（自己出測試自己實作）拆成跨 agent 的外部契約。買到的是<strong>更早、更獨立的錯誤發現點</strong>——任何一環的自欺都會在下一環被抓到。代價是 orchestration 來回，和你必須真的去驗收、而不是相信回報。</p> <p>對決定要不要把多 agent 導入實際 workflow 的人：先確認你的專案有「測試能當契約」的體質。沒有這個前提，多 agent 只是把一個 agent 的不可靠乘以三。</p> <p><em>文中失敗場景（Grok <code>grok-composer-2.5-fast</code> 的 reasoningEffort 400、假成功回報、Zig lazy discovery 需 `</em> = x;`、Rust 那輪的靜默 migration 與 seed 半載入、密鑰進 argv）均為本人實測，工具版本與命令見「實驗條件」。_</p>
