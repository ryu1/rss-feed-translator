<blockquote> <p>Every "what's the cheapest model?" thread online is people trading vibes. I got tired of it, so I built a pipeline that pulls <em>live, cited</em> prices and runs the numbers through an <strong>exact-rational math kernel</strong> — no floating-point drift, no LLM hallucinating a multiplication. Then I pointed it at eight of the cost questions every agent builder actually faces. Here's everything it found, and the repo so you can re-run all of it.</p> </blockquote> <p>There are two kinds of LLM cost advice. The first is a benchmark leaderboard with a price column, which tells you nothing about <em>your</em> workload. The second is a confident tweet that's quietly wrong because someone multiplied a per-million price by the wrong token count in their head.</p> <p>I wanted a third kind: a model you can <strong>audit</strong>. Re-run it and you get bit-identical numbers, with the source of every input price one file away. This article is the result — a tour through eight cost decisions, each answered with real money math, and each one teaching something that intuition gets wrong.</p> <p>Let's start with the question that kicked it off.</p> <h2> Cold open: what's the most cost-effective model to run an agent on? </h2> <p>The setup: I run <a href="https://github.com/NousResearch/hermes-agent" rel="noopener noreferrer">Hermes Agent</a>, a model-agnostic agent framework, and I wanted the cheapest token provider to drive it. Not cheapest <em>per token</em> — cheapest <strong>per unit of quality</strong>, because a model that flubs tool calls and retries can cost more than a pricier reliable one.</p> <p>So the metric is <strong>blended cost ÷ agentic-quality-score</strong>. Blended cost uses a production-agentic token mix (more on that below); quality is a normalized agentic score (mean of BFCL / τ²-bench / SWE-bench Verified). Filters: open-weights, prompt caching, no-train.</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Model @ Provider</th> <th>Blended $/1M</th> <th>Quality</th> <th>$/quality (×1000)</th> </tr> </thead> <tbody> <tr> <td><strong>DeepSeek V3.2 @ OpenRouter</strong></td> <td><strong>0.1145</strong></td> <td>77</td> <td><strong>1.49</strong></td> </tr> <tr> <td>DeepSeek V3.2 @ DeepInfra</td> <td>0.1951</td> <td>77</td> <td>2.53</td> </tr> <tr> <td>MiniMax M2 @ MiniMax</td> <td>0.2629</td> <td>73</td> <td>3.60</td> </tr> <tr> <td>GLM-4.6 @ z.ai</td> <td>0.5276</td> <td>71</td> <td>7.43</td> </tr> <tr> <td>Kimi K2 @ DeepInfra</td> <td>0.6613</td> <td>66</td> <td>10.02</td> </tr> <tr> <td>DeepSeek R1 @ DeepInfra</td> <td>0.6519</td> <td>54</td> <td>12.07</td> </tr> </tbody> </table></div> <p><strong>DeepSeek V3.2 wins decisively</strong> — it's both the highest-quality open model in the set <em>and</em> near the cheapest, because its output token price is absurdly low for its tier. The emerging best deal (pending a quality test) is <strong>DeepSeek V4 Flash on Fireworks</strong> at $0.0896/1M blended — cheapest in the field, with ZDR-by-default and a 50% batch discount.</p> <p>That's a useful answer. But the <em>interesting</em> part is everything that question opens up. If you can split traffic across providers, what's the optimal mix? Should you self-host instead? Is the "smart expensive model" actually cheaper once you count its thinking tokens? Each of those is a chapter.</p> <p>First, how this is built.</p> <h2> The toolchain: research agents + an exact-math kernel </h2> <p>The pipeline has two halves, and the whole point is that <strong>the LLM never touches the arithmetic.</strong></p> <p><strong>1. Research agents gather live, cited inputs.</strong> Token prices in 2026 move monthly — half the models in my training data are already legacy. So every price and benchmark in this guide came from a research agent doing live web search/fetch, returning a structured table with source URLs and observation dates. Those land in <code>data/</code> (<code>pricing.md</code>, <code>quality.md</code>, <code>gpu.md</code>, <code>frontier.md</code>, <code>decline.md</code>), each one auditable.</p> <p><strong>2. An exact-rational kernel does the math.</strong> I used <a href="https://github.com/copyleftdev/agent-calc" rel="noopener noreferrer"><code>agent-calc</code></a>, an AI-native computation kernel that works in exact rationals — <code>3/5 × 1,000,000</code> returns <code>600000</code>, not <code>599999.9999998</code>. It exposes typed domains: <code>rational</code>, <code>linear</code> (LP), <code>finance</code> (NPV/amortization), <code>solve</code> (algebra), <code>stats</code> (distributions), <code>polynomial</code>, <code>interval</code> (uncertainty arithmetic). Every chapter below drives a different one.</p> <p>Why bother? Because cost models are exactly where floating-point lies and LLMs fumble: tiny per-token prices, huge token counts, compounding rates. When an answer is "self-hosting breaks even at 87.2% utilization," you want that to be <em>computed</em>, not vibed. And the secret structure of this guide is that each chapter answers a money question <strong>and</strong> exercises one kernel domain — it's a cost-modeling handbook that doubles as a tour of exact computation.</p> <h3> The shared workload model </h3> <p>Every chapter uses one production-agentic token profile, because agent tool-loops are wildly input-heavy:</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Component</th> <th>Share of tokens</th> </tr> </thead> <tbody> <tr> <td>Fresh input</td> <td>21.25%</td> </tr> <tr> <td>Cached input</td> <td>63.75%</td> </tr> <tr> <td>Output</td> <td>15%</td> </tr> </tbody> </table></div> <p>That's an 85/15 input/output split with 75% of input being a cache hit (the system prompt, tools, and skills get re-sent every tool-loop step). Blended price = <code>0.2125·in + 0.6375·cached + 0.15·out</code>. Chapter 7 explains <em>why</em> this profile is forced on you; Chapter 6 explains why the cache hit rate is so high. For now, take it as the workload.</p> <h2> Chapter 1 — Optimal multi-provider routing (a linear program) </h2> <p><a class="article-body-image-wrapper" href="https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.us-east-2.amazonaws.com%2Fuploads%2Farticles%2Fvvob8hq7fsafn5kvk3sn.png"><img alt="The cost-of-privacy frontier: cost stays flat until 40% US-jurisdiction traffic, rises steeply after 60%, and hits an infeasible wall at 87.2%" height="486" src="https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.us-east-2.amazonaws.com%2Fuploads%2Farticles%2Fvvob8hq7fsafn5kvk3sn.png" width="800" /></a></p> <p>You don't have to pick one provider. Run a cheap one for the bulk, escalate elsewhere when constraints demand. The optimal split — minimize spend subject to a quality floor, per-endpoint capacity caps, and a <strong>US-jurisdiction floor θ</strong> — is a literal linear program. <code>agent-calc</code> solves it (<code>solve_lp</code>), then I re-verify the winning allocation's cost in exact rationals (the LP solver is f64; the headline number shouldn't be).</p> <p>Sweeping the privacy knob θ traces a <strong>cost-of-privacy frontier</strong>:</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>US floor θ</th> <th>min $/1M</th> <th>$/mo @ 1B</th> <th>allocation</th> </tr> </thead> <tbody> <tr> <td>0.00–0.40</td> <td>0.1468</td> <td>$146.77</td> <td>A=0.60, B=0.40</td> </tr> <tr> <td>0.60</td> <td>0.1629</td> <td>$162.89</td> <td>A=0.40, B=0.60</td> </tr> <tr> <td>0.80</td> <td>0.2716</td> <td>$271.61</td> <td>A=0.20, B=0.60, F=0.13, H=0.07</td> </tr> <tr> <td><strong>0.872</strong></td> <td>0.3116</td> <td>$311.59</td> <td>A=0.13, B=0.60, F=0.27</td> </tr> <tr> <td>≥ 0.873</td> <td>—</td> <td><strong>INFEASIBLE</strong></td> <td>the privacy wall</td> </tr> </tbody> </table></div> <p>Three things intuition misses:</p> <ol> <li> <strong>The first 40% of US-jurisdiction traffic is free</strong> — the cost-optimal blend already lands there, because the cheapest provider is capped at 60% and the overflow falls on a US endpoint anyway.</li> <li> <strong>The marginal cost of privacy is convex</strong> — 40→60% is cheap; past 60% the quality floor forces in weaker fillers and the bill nearly doubles by 87%.</li> <li> <strong>There's a hard wall at 87.2%.</strong> Only <em>one</em> high-quality US endpoint exists in this menu; beyond its capacity the math is infeasible. To break the wall you need a second high-quality US provider, or you accept single-point-of-failure risk.</li> </ol> <h2> Chapter 2 — Self-host vs. serverless: it's a utilization question </h2> <p><a class="article-body-image-wrapper" href="https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.us-east-2.amazonaws.com%2Fuploads%2Farticles%2Fwcvw2d7zq4z6pknvnvkd.png"><img alt="Break-even utilization by hardware and procurement: every rented and Hopper config needs &gt;100% utilization (impossible); only owned 8×B200 wins at 72%" height="459" src="https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.us-east-2.amazonaws.com%2Fuploads%2Farticles%2Fwcvw2d7zq4z6pknvnvkd.png" width="800" /></a></p> <p>Everyone's instinct: "at some volume, renting GPUs beats per-token APIs." The reframe that changes everything: <strong>a self-hosted node costs the same at 5% load or 95% load</strong>, while the API scales linearly with tokens. So it's not a <em>dollar</em> break-even — it's a <strong>break-even utilization</strong>: how busy must the box stay to win?<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>u* = node_$/mo × 1e6 / (capacity_tokens/mo × API_$/1M) </code></pre> </div> <p>If <code>u* &gt; 100%</code>, self-hosting loses <em>even at full tilt</em>. Modeling DeepSeek V3.2 (8-GPU node) against DeepInfra's $0.195/1M, with <code>agent-calc</code> handling the owned-hardware amortization (<code>finance</code>) and the exact <code>u*</code> (<code>eval</code>):</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Hardware / procurement</th> <th>break-even u*</th> <th>verdict</th> </tr> </thead> <tbody> <tr> <td>8×H200, rent on-demand</td> <td>531%</td> <td>never</td> </tr> <tr> <td>8×H200, own (amortized)</td> <td>220%</td> <td>never</td> </tr> <tr> <td>8×B200, rent reserved</td> <td>174%</td> <td>never</td> </tr> <tr> <td><strong>8×B200, own (amortized)</strong></td> <td><strong>72%</strong></td> <td><strong>the only winner</strong></td> </tr> </tbody> </table></div> <p>For a commodity model this cheap on the API, <strong>renting GPUs loses in every configuration</strong> — the serverless provider batches across thousands of tenants to hit a utilization you can't. The <em>only</em> config that wins is <strong>owned Blackwell silicon kept above ~72% utilization 24/7</strong> (~56B tokens/month of steady load). And against the falling API floor ($0.0896/1M), even that roughly doubles its break-even. The takeaway: <strong>self-host for privacy, control, or latency — not to cut the token bill.</strong></p> <h2> Chapter 3 — The reasoning-token tax </h2> <p><a class="article-body-image-wrapper" href="https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.us-east-2.amazonaws.com%2Fuploads%2Farticles%2F8xxq79ayjh348gzugle6.png"><img alt="DeepSeek R1's cost-per-solved climbs steeply with reasoning burn, always far above the DeepSeek V3.2 champion line — even at zero thinking" height="485" src="https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.us-east-2.amazonaws.com%2Fuploads%2Farticles%2F8xxq79ayjh348gzugle6.png" width="799" /></a></p> <p>Reasoning models emit hidden "thinking" tokens, billed at the output rate. A 600-token answer can bill 7,800. Does the higher success rate justify the burn? The honest unit isn't $/token — it's <strong>cost per <em>solved</em> task</strong> = attempt cost ÷ success rate.</p> <p>With <code>agent-calc</code> computing exact costs (<code>eval</code>) and the break-even success rate (<code>solve</code>):</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Model</th> <th>k (think mult.)</th> <th>$/solved</th> <th>×champion</th> </tr> </thead> <tbody> <tr> <td><strong>DeepSeek V3.2</strong></td> <td>4</td> <td><strong>3.17m</strong></td> <td><strong>1.00×</strong></td> </tr> <tr> <td>MiniMax M2</td> <td>2</td> <td>4.03m</td> <td>1.27×</td> </tr> <tr> <td>Kimi K2</td> <td>1</td> <td>8.79m</td> <td>2.77×</td> </tr> <tr> <td>GLM-4.6</td> <td>5</td> <td>13.77m</td> <td>4.35×</td> </tr> <tr> <td>DeepSeek R1</td> <td>12</td> <td>36.80m</td> <td>11.61×</td> </tr> </tbody> </table></div> <p>(<code>m</code> = milli-dollars = $0.001/task.)</p> <p>The killer result: <strong>DeepSeek R1 would need a 627% success rate</strong> to match V3.2's cost-per-solved — mathematically impossible. Even a <em>perfect</em> R1 stays far more expensive, because its per-attempt token burn alone exceeds V3.2's entire cost-per-solved. The sweep confirms it: even at k=0 (zero thinking), R1 is still 2.6× the champion.</p> <p>The rule: <strong>the reasoning tax only pays when output tokens are cheap.</strong> Thinking is a multiplier on the output price. Pay $2.15/M and ×12 and you've built the most expensive possible way to solve a task; pay $0.38/M and the same multiplier barely registers.</p> <h2> Chapter 4 — The retry cascade </h2> <p><a class="article-body-image-wrapper" href="https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.us-east-2.amazonaws.com%2Fuploads%2Farticles%2Fxrhx8db0avuhf5ybz3o4.png"><img alt="Cost-per-solved vs coverage: cascades sit down-and-right of both single-model strategies — higher coverage at a fraction of premium-only cost" height="472" src="https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.us-east-2.amazonaws.com%2Fuploads%2Farticles%2Fxrhx8db0avuhf5ybz3o4.png" width="800" /></a></p> <p>You don't need one model for every task. Run a <strong>cheap</strong> model on everything, <strong>escalate</strong> to a reliable premium model only on the residual failures. <code>agent-calc</code> computes the escalation probability (<code>stats/binomial_pmf</code>) and the exact expected cost (<code>eval</code>).</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Strategy</th> <th>cost/solved</th> <th>coverage</th> <th>% → premium</th> </tr> </thead> <tbody> <tr> <td>Cheap-only (DeepSeek V3.2)</td> <td>3.17m</td> <td>77.0%</td> <td>0%</td> </tr> <tr> <td>Premium-only (Claude Opus 4.8)</td> <td>82.96m</td> <td>88.0%</td> <td>100%</td> </tr> <tr> <td><strong>Cascade V3.2 → Opus</strong></td> <td><strong>19.78m</strong></td> <td><strong>97.2%</strong></td> <td>23%</td> </tr> <tr> <td><strong>Cascade V3.2 → Gemini → Opus</strong></td> <td><strong>13.52m</strong></td> <td><strong>99.5%</strong></td> <td>4.4%</td> </tr> </tbody> </table></div> <p>The cascade beats premium-only on <strong>both</strong> axes: higher coverage (97% vs 88%) at a quarter the cost, because only 23% of tasks reach the expensive model. Adding a mid-tier pushes coverage to 99.5% while <em>cutting</em> cost further — the cheap+mid tiers absorb 95.6% of traffic, so the $73m Opus call fires on 4.4% of tasks.</p> <p>And the trap to avoid: <strong>retrying the <em>same</em> model never changes cost-per-solved</strong> (it stays C/p), and for deterministic failures it doesn't even raise coverage. Only escalating to a <em>different, stronger</em> model moves the needle. "Just add retries" is a no-op; "add a tier" is the win. (The cost advantage survives even strong failure correlation — 3.6× cheaper than premium-only even when the premium rarely rescues what the cheap model botched.)</p> <h2> Chapter 5 — The NPV of waiting </h2> <p><a class="article-body-image-wrapper" href="https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.us-east-2.amazonaws.com%2Fuploads%2Farticles%2Fc4iqu00jzc28s7c4vds3.png"><img alt="12-month NPV: pay-go beats a 30%-off commit for fast-declining commodity prices (−23%) but loses for sticky frontier prices (+22%)" height="469" src="https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.us-east-2.amazonaws.com%2Fuploads%2Farticles%2Fc4iqu00jzc28s7c4vds3.png" width="800" /></a></p> <p>A provider offers a discounted <em>reserved</em> rate if you commit for a year. Locking it in feels prudent — but token prices fall fast, and you might be locking yourself <em>above</em> where pay-go will be in six months. This is a present-value problem: NPV the committed cash-flow stream against the declining pay-go stream (<code>agent-calc finance/net_present_value</code>), and find the <strong>break-even decline rate</strong>.</p> <p>The decisive input is that price decline depends entirely on what you hold fixed:</p> <ul> <li> <strong>Fixed-quality</strong> (same capability getting cheaper): <strong>~80%/yr</strong> (a16z/Epoch).</li> <li> <strong>Frontier</strong> (always run the best): <strong>~30%/yr</strong>, sometimes flat or rising.</li> </ul> <p>At a 30%-off, 12-month commit:</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Workload</th> <th>central decline</th> <th>winner</th> </tr> </thead> <tbody> <tr> <td><strong>Fixed-quality</strong></td> <td>80%/yr</td> <td><strong>pay-go (−23%)</strong></td> </tr> <tr> <td><strong>Frontier</strong></td> <td>30%/yr</td> <td><strong>commit (+22%)</strong></td> </tr> </tbody> </table></div> <p>And the break-even decline grid shows committing <strong>never</strong> wins for commodity models (80%/yr exceeds every reservable discount's break-even), while longer terms always make it worse: a 30% discount breaks even at 57.8%/yr over 12 months but only <strong>24.5%/yr over 36 months</strong>. The rule: <strong>reserve where prices are sticky (frontier); ride the curve where they fall fast (commodity). Never sign a 3-year inference commit in a market falling &gt;25%/yr.</strong></p> <h2> Chapter 6 — Prompt-cache ROI </h2> <p>Caching looks free, but a cache <em>write</em> often costs more than a normal input token (Anthropic charges up to 2×), while reads are cheap. So it's a bet: pay the write premium to save on reads, and it only pays if you reuse enough before the cache expires. The break-even reuse count is <code>N* = (write − read)/(in − read)</code>.</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Provider</th> <th>break-even N*</th> <th>savings ceiling</th> </tr> </thead> <tbody> <tr> <td>DeepInfra / DeepSeek / OpenAI / Fireworks (no write fee)</td> <td>1.00</td> <td>50–90%</td> </tr> <tr> <td>Anthropic Opus 5-min TTL (1.25× write)</td> <td>1.28</td> <td>90%</td> </tr> <tr> <td>Anthropic Opus 1-hr TTL (2× write)</td> <td>2.11</td> <td>90%</td> </tr> </tbody> </table></div> <p><strong>Caching pays almost immediately</strong> — most providers from the second call, even the harshest write premium by the third. The <em>read</em> multiplier sets the long-run ceiling (0.1×-read → 90% savings; 0.5×-read → 50%), so for cache-heavy loops that's the number that compounds. Using <code>agent-calc</code>'s <code>interval</code> domain, an 8k-token agent prefix reused an uncertain 3–20 times costs $0.088–0.156 cached vs $0.12–0.80 uncached — <strong>80% off at the high end</strong>, even with a 2× write premium. This is why the guide's 75% cache-hit assumption holds: agent loops reuse a large prefix 10–20× per task.</p> <h2> Chapter 7 — Agent-loop compounding (the O(K²) tax) </h2> <p><a class="article-body-image-wrapper" href="https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.us-east-2.amazonaws.com%2Fuploads%2Farticles%2F21w6ks9gq0t46u62gijj.png"><img alt="Cumulative input cost vs loop length: naive grows quadratically, caching halves the constant but stays quadratic, compaction turns it linear" height="486" src="https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.us-east-2.amazonaws.com%2Fuploads%2Farticles%2F21w6ks9gq0t46u62gijj.png" width="800" /></a></p> <p>Why is that reused prefix so large? Because a multi-turn agent re-sends its growing context every step — at step <em>k</em>, the input is the base prompt plus <em>every prior tool call and result</em>. Per-step input grows linearly, so <strong>cumulative input over a K-step task grows quadratically</strong>:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>cumulative over K = (δ/2)·K² + (B − δ/2)·K </code></pre> </div> <p><code>agent-calc</code> evaluates that polynomial exactly (<code>polynomial/evaluate</code>):</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>K steps</th> <th>cum tokens</th> <th>naive $</th> <th>cached $</th> <th>compacted $</th> </tr> </thead> <tbody> <tr> <td>10</td> <td>147,500</td> <td>0.0384</td> <td>0.0220</td> <td>0.0384</td> </tr> <tr> <td>50</td> <td>2,237,500</td> <td>0.5817</td> <td>0.3015</td> <td>0.3267</td> </tr> <tr> <td>100</td> <td>8,225,000</td> <td>2.1385</td> <td>1.0896</td> <td>0.6907</td> </tr> </tbody> </table></div> <p>A 100-step task isn't 10× a 10-step task — it's <strong>56×</strong>. Your 50th tool call's input costs 10× your first. <strong>Caching bends the curve (lowers the constant); only compaction breaks it</strong> — capping the context window turns the quadratic linear, and the gap widens without bound as loops get longer. They compose: cached <em>and</em> compacted is read-rate pricing on a linear token count. For any loop past ~20 steps, do both.</p> <p>This closes the loop on the whole guide: the quadratic re-send is exactly why agent workloads are so input-heavy and so cache-dependent — the entire shared workload profile falls out of this one chapter.</p> <h2> The cheat sheet </h2> <p>Everything above, compressed to decisions:</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Decision</th> <th>Verdict</th> </tr> </thead> <tbody> <tr> <td><strong>Which model?</strong></td> <td>Cheap efficient hybrid (DeepSeek V3.2-class). Cheap output + high success beats everything on cost-per-solved.</td> </tr> <tr> <td><strong>One provider or many?</strong></td> <td>Blend via LP. <p>Here's something you'll notice after running AI agents in production for a few weeks: a fresh conversation with your agent is sharp. Give that same agent 40 messages of history and it starts contradicting earlier decisions, forgetting constraints, and producing worse output than it did at the start of the session.</p> <p>It's not random. It's structural. The context window is a fixed-size working memory, and you're filling it with noise.</p> <p>I call this context rot — the gradual degradation of agent performance as accumulated context crowds out the signal with stale data, repeated boilerplate, and irrelevant turns. Here's what causes it, how to measure it, and three patterns that genuinely fix it.</p> <h2> What's actually happening </h2> <p>Language models have no persistent memory between calls. Every request is a fresh inference over the entire sequence of tokens you provide. The "memory" is entirely the context window.</p> <p>This creates a few failure modes as conversations grow:</p> <p><strong>1. <p>En algún lugar de la página de precios de tu proveedor de autenticación hay una casilla que cuesta, al mes, más que el portátil de un ingeniero, y cuya provisión no cuesta nada. Este texto trata sobre esa casilla.</p> <p>Existe un sitio web llamado sso.tax. Existe porque tantas empresas de software cobran un recargo tan alto por el inicio de sesión único, una función cuya operación no les cuesta prácticamente nada, que alguien tuvo que crear una lista pública de escarnio solo para llevar la cuenta. SAML es un estándar consolidado desde hace dos décadas. El código está escrito. Los bytes son los mismos bytes, tanto si un usuario inicia sesión con una contraseña como a través de un proveedor de identidad corporativo. Y aun así, «activar el SSO» es de forma fiable la línea que duplica tu factura, o la que finalmente obliga a la temida llamada a ventas. Eso no es un precio. Es un peaje en una carretera que ya estaba construida y pagada.</p> <p>Aquí está la distinción que la industria preferiría que nunca trazaras con claridad. Hay una diferencia entre cobrar el <em>coste</em> y cobrar el <em>permiso</em>. Cobrar el coste es honesto: si envío más tráfico, almaceno más datos o mantengo activos a más usuarios activos mensuales, atenderme cuesta realmente más, y deberías cobrarlo. Cobrar el permiso es algo completamente distinto. Es cobrarme por accionar un booleano que ya has escrito, sobre una infraestructura que ya está en marcha, por una función cuyo coste marginal, para ti, se redondea a cero. La mayoría de las páginas de precios de autenticación son sobre todo lo segundo disfrazado de lo primero.</p> <p>Una vez que lo ves, no puedes dejar de verlo. El inicio de sesión único encerrado tras el nivel «Enterprise», o cobrado por conexión a más de cien dólares al mes cada una, y a veces cobrado <em>dos veces</em>: una por la conexión SSO y otra por el aprovisionamiento SCIM que viaja sobre esa misma integración. La autenticación multifactor vendida como una mejora de pago, es decir, un recargo por no dejar que tus usuarios caigan en el phishing. Los registros de auditoría troceados en ventanas de retención según el nivel, como si una fila del trimestre pasado costara más almacenar que una de esta mañana. Los tokens de máquina a máquina limitados y cobrados por uso, como si firmar un JWT fuera un mineral escaso.
