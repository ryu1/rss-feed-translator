同期ターボモジュールは自動的に高速になるわけではない
Synchronous TurboModules Are Not Automatically Fast
原文説明
原文説明（英語）

<p>React Native’s New Architecture gives JavaScript more direct access to native functionality.</p> <p>One important capability is synchronous communication through TurboModules.</p> <p>That can be useful.</p> <p>It can also introduce a performance problem that is easy to miss:</p> <blockquote> <p>A native method being synchronous does not mean the native work is fast.</p> </blockquote> <p>The communication mechanism may be efficient, but the JavaScript thread must still wait for the native operation to finish.</p> <p>That difference matters in production applications.</p> <h2> The misleading simplicity of a synchronous API </h2> <p>Consider a native method exposed like this:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">token</span> <span class="o">=</span> <span class="nx">SecureStorage</span><span class="p">.</span><span class="nf">getAccessToken</span><span class="p">();</span> </code></pre> </div> <p>From the JavaScript side, it looks harmless.</p> <p>There is no <code>await</code>, no callback and no visible asynchronous workflow.</p> <p>The value is returned immediately from the developer’s point of view.</p> <p>But the native implementation may perform several operations before returning:</p> <ul> <li>access Android Keystore or iOS Keychain</li> <li>decrypt stored data</li> <li>read from disk</li> <li>parse JSON</li> <li>convert native objects into JavaScript-compatible values</li> <li>communicate with another system service</li> </ul> <p>If this work takes time, JavaScript cannot continue until the method returns.</p> <p>The API looks simple.</p> <p>The execution may not be.</p> <h2> What synchronous actually means </h2> <p>A synchronous call means the caller waits for the result.</p> <p>For example:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">result</span> <span class="o">=</span> <span class="nx">NativeModule</span><span class="p">.</span><span class="nf">getValue</span><span class="p">();</span> <span class="nf">renderScreen</span><span class="p">(</span><span class="nx">result</span><span class="p">);</span> </code></pre> </div> <p><code>renderScreen</code> cannot begin until <code>getValue()</code> finishes.</p> <p>That behaviour is acceptable when the native operation is extremely small and predictable.</p> <p>It becomes dangerous when the operation has variable latency.</p> <p>The same method may take:</p> <ul> <li>less than one millisecond when data is cached</li> <li>several milliseconds when data requires conversion</li> <li>much longer when storage, cryptography or system services are involved</li> </ul> <p>The JavaScript thread does not care why the native method is delayed.</p> <p>It simply remains blocked.</p> <h2> Why this affects React Native performance </h2> <p>React Native applications perform important work on the JavaScript thread.</p> <p>This can include:</p> <ul> <li>processing state updates</li> <li>running application logic</li> <li>handling user interactions</li> <li>preparing React renders</li> <li>coordinating navigation</li> <li>executing animation-related JavaScript</li> <li>responding to events from native modules</li> </ul> <p>When a synchronous native method blocks that thread, other JavaScript work must wait.</p> <p>A single call may not create an obvious problem.</p> <p>Repeated calls can.</p> <p>Imagine that a component reads a secure value during rendering:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight tsx"><code><span class="kd">function</span> <span class="nf">AccountScreen</span><span class="p">()</span> <span class="p">{</span> <span class="kd">const</span> <span class="nx">token</span> <span class="o">=</span> <span class="nx">SecureStorage</span><span class="p">.</span><span class="nf">getAccessToken</span><span class="p">();</span> <span class="k">return</span> <span class="p">&lt;</span><span class="nc">AccountDetails</span> <span class="na">token</span><span class="p">=</span><span class="si">{</span><span class="nx">token</span><span class="si">}</span> <span class="p">/&gt;;</span> <span class="p">}</span> </code></pre> </div> <p>Now imagine that the component renders multiple times during navigation or state updates.</p> <p>Every render may trigger another native operation.</p> <p>If the method accesses storage and performs decryption, the application is repeatedly blocking the JavaScript thread inside the render path.</p> <p>The visible result may be:</p> <ul> <li>slow navigation</li> <li>delayed button responses</li> <li>dropped animation frames</li> <li>inconsistent screen transitions</li> <li>temporary UI freezes</li> <li>performance that becomes worse on lower-end devices</li> </ul> <p>The developer may initially investigate React re-renders.</p> <p>The real problem is the synchronous native work performed during those renders.</p> <h2> TurboModules reduce overhead, not execution cost </h2> <p>This is the central distinction.</p> <p>TurboModules improve how JavaScript and native code communicate.</p> <p>They can reduce some of the overhead associated with the older bridge architecture.</p> <p>But TurboModules do not make the native operation itself free.</p> <p>Suppose a native method performs three tasks:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Read secure storage ↓ Decrypt the value ↓ Convert it into a JavaScript object </code></pre> </div> <p>The communication layer may be efficient.</p> <p>The application must still pay for:</p> <ul> <li>storage access</li> <li>cryptographic processing</li> <li>object allocation</li> <li>data conversion</li> <li>thread coordination</li> </ul> <p>Optimising the path between JavaScript and native code does not eliminate the work happening at the destination.</p> <p>A faster doorway does not make the room behind it smaller.</p> <h2> When synchronous native methods are appropriate </h2> <p>Synchronous methods are useful when the operation is:</p> <ul> <li>tiny</li> <li>deterministic</li> <li>memory-resident</li> <li>frequently required</li> <li>guaranteed not to perform slow I/O</li> <li>predictable across supported devices</li> </ul> <p>Examples may include reading a cached capability:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">supportsBiometrics</span> <span class="o">=</span> <span class="nx">DeviceCapabilities</span><span class="p">.</span><span class="nf">hasBiometricHardware</span><span class="p">();</span> </code></pre> </div> <p>Or checking a native constant:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">environment</span> <span class="o">=</span> <span class="nx">AppConfiguration</span><span class="p">.</span><span class="nf">getEnvironment</span><span class="p">();</span> </code></pre> </div> <p>Or performing a very small calculation:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">pixels</span> <span class="o">=</span> <span class="nx">DisplayUtils</span><span class="p">.</span><span class="nf">dpToPixels</span><span class="p">(</span><span class="mi">16</span><span class="p">);</span> </code></pre> </div> <p>These operations are generally suitable only when their native implementation remains lightweight.</p> <p>The important question is not whether the return type is small.</p> <p>The important question is how the value is produced.</p> <p>Returning one boolean can still be expensive if the implementation queries a system service every time.</p> <h2> Operations that should usually remain asynchronous </h2> <p>Native methods should normally be asynchronous when they involve:</p> <h3> Storage </h3> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">profile</span> <span class="o">=</span> <span class="k">await</span> <span class="nx">SecureStorage</span><span class="p">.</span><span class="nf">getProfile</span><span class="p">();</span> </code></pre> </div> <p>Disk and secure-storage access may have unpredictable latency.</p> <h3> Cryptography </h3> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">decryptedPayload</span> <span class="o">=</span> <span class="k">await</span> <span class="nx">CryptoModule</span><span class="p">.</span><span class="nf">decrypt</span><span class="p">(</span><span class="nx">payload</span><span class="p">);</span> </code></pre> </div> <p>Cryptographic work can become expensive depending on the algorithm and input size.</p> <h3> Networking </h3> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">response</span> <span class="o">=</span> <span class="k">await</span> <span class="nx">NativeNetworkClient</span><span class="p">.</span><span class="nf">fetchConfiguration</span><span class="p">();</span> </code></pre> </div> <p>Network operations are naturally unpredictable and should never block JavaScript synchronously.</p> <h3> Large data conversion </h3> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">records</span> <span class="o">=</span> <span class="k">await</span> <span class="nx">NativeDatabase</span><span class="p">.</span><span class="nf">getRecords</span><span class="p">();</span> </code></pre> </div> <p>Converting thousands of native records into JavaScript objects can consume considerable time.</p> <h3> Image and file processing </h3> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">result</span> <span class="o">=</span> <span class="k">await</span> <span class="nx">ImageProcessor</span><span class="p">.</span><span class="nf">compress</span><span class="p">(</span><span class="nx">filePath</span><span class="p">);</span> </code></pre> </div> <p>Media processing is computationally expensive and should normally run away from latency-sensitive application work.</p> <h3> System services </h3> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">location</span> <span class="o">=</span> <span class="k">await</span> <span class="nx">LocationModule</span><span class="p">.</span><span class="nf">getCurrentPosition</span><span class="p">();</span> </code></pre> </div> <p>The response time may depend on permissions, sensors, device state and operating-system behaviour.</p> <h2> Avoid synchronous work inside rendering </h2> <p>One of the strongest review rules is:</p> <blockquote> <p>Do not perform synchronous native work inside a React render path unless the operation is proven to be trivial.</p> </blockquote> <p>This is risky:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight tsx"><code><span class="kd">function</span> <span class="nf">Dashboard</span><span class="p">()</span> <span class="p">{</span> <span class="kd">const</span> <span class="nx">settings</span> <span class="o">=</span> <span class="nx">NativeSettings</span><span class="p">.</span><span class="nf">getSettings</span><span class="p">();</span> <span class="k">return</span> <span class="p">&lt;</span><span class="nc">DashboardContent</span> <span class="na">settings</span><span class="p">=</span><span class="si">{</span><span class="nx">settings</span><span class="si">}</span> <span class="p">/&gt;;</span> <span class="p">}</span> </code></pre> </div> <p>A better design is to load the value asynchronously and store it in application state:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight tsx"><code><span class="kd">function</span> <span class="nf">Dashboard</span><span class="p">()</span> <span class="p">{</span> <span class="kd">const</span> <span class="p">[</span><span class="nx">settings</span><span class="p">,</span> <span class="nx">setSettings</span><span class="p">]</span> <span class="o">=</span> <span class="nx">useState</span><span class="o">&lt;</span><span class="nx">Settings</span> <span class="o">|</span> <span class="kc">null</span><span class="o">&gt;</span><span class="p">(</span><span class="kc">null</span><span class="p">);</span> <span class="nf">useEffect</span><span class="p">(()</span> <span class="o">=&gt;</span> <span class="p">{</span> <span class="kd">let</span> <span class="nx">active</span> <span class="o">=</span> <span class="kc">true</span><span class="p">;</span> <span class="k">async</span> <span class="kd">function</span> <span class="nf">loadSettings</span><span class="p">()</span> <span class="p">{</span> <span class="kd">const</span> <span class="nx">value</span> <span class="o">=</span> <span class="k">await</span> <span class="nx">NativeSettings</span><span class="p">.</span><span class="nf">getSettings</span><span class="p">();</span> <span class="k">if </span><span class="p">(</span><span class="nx">active</span><span class="p">)</span> <span class="p">{</span> <span class="nf">setSettings</span><span class="p">(</span><span class="nx">value</span><span class="p">);</span> <span class="p">}</span> <span class="p">}</span> <span class="nf">loadSettings</span><span class="p">();</span> <span class="k">return </span><span class="p">()</span> <span class="o">=&gt;</span> <span class="p">{</span> <span class="nx">active</span> <span class="o">=</span> <span class="kc">false</span><span class="p">;</span> <span class="p">};</span> <span class="p">},</span> <span class="p">[]);</span> <span class="k">if </span><span class="p">(</span><span class="o">!</span><span class="nx">settings</span><span class="p">)</span> <span class="p">{</span> <span class="k">return</span> <span class="p">&lt;</span><span class="nc">LoadingScreen</span> <span class="p">/&gt;;</span> <span class="p">}</span> <span class="k">return</span> <span class="p">&lt;</span><span class="nc">DashboardContent</span> <span class="na">settings</span><span class="p">=</span><span class="si">{</span><span class="nx">settings</span><span class="si">}</span> <span class="p">/&gt;;</span> <span class="p">}</span> </code></pre> </div> <p>The exact state-management approach may differ.</p> <p>The principle remains the same:</p> <p>Do not make rendering wait for native storage, cryptography or system resources.</p> <h2> Cache deliberately, not accidentally </h2> <p>Sometimes a synchronous API is created because the application needs immediate access to a value.</p> <p>The safer solution may be to load the value earlier and maintain an in-memory cache.</p> <p>For example:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">class</span> <span class="nc">SessionStore</span> <span class="p">{</span> <span class="k">private</span> <span class="nx">token</span><span class="p">:</span> <span class="kr">string</span> <span class="o">|</span> <span class="kc">null</span> <span class="o">=</span> <span class="kc">null</span><span class="p">;</span> <span class="k">async</span> <span class="nf">initialise</span><span class="p">():</span> <span class="nb">Promise</span><span class="o">&lt;</span><span class="k">void</span><span class="o">&gt;</span> <span class="p">{</span> <span class="k">this</span><span class="p">.</span><span class="nx">token</span> <span class="o">=</span> <span class="k">await</span> <span class="nx">SecureStorage</span><span class="p">.</span><span class="nf">getAccessToken</span><span class="p">();</span> <span class="p">}</span> <span class="nf">getCachedToken</span><span class="p">():</span> <span class="kr">string</span> <span class="o">|</span> <span class="kc">null</span> <span class="p">{</span> <span class="k">return</span> <span class="k">this</span><span class="p">.</span><span class="nx">token</span><span class="p">;</span> <span class="p">}</span> <span class="p">}</span> </code></pre> </div> <p>The native storage operation remains asynchronous:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="k">await</span> <span class="nx">sessionStore</span><span class="p">.</span><span class="nf">initialise</span><span class="p">();</span> </code></pre> </div> <p>Later reads are synchronous because they access application memory:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">token</span> <span class="o">=</span> <span class="nx">sessionStore</span><span class="p">.</span><span class="nf">getCachedToken</span><span class="p">();</span> </code></pre> </div> <p>This creates a meaningful distinction:</p> <ul> <li>loading from storage is asynchronous</li> <li>reading an already-loaded value is synchronous</li> </ul> <p>That is much safer than hiding storage access behind a synchronous API.</p> <p>However, caching introduces its own responsibilities:</p> <ul> <li>cache invalidation</li> <li>logout handling</li> <li>token refresh</li> <li>memory lifecycle</li> <li>multi-process consistency</li> <li>protection of sensitive values</li> </ul> <p>The goal is not to make everything synchronous.</p> <p>The goal is to place expensive work outside latency-sensitive paths.</p> <h2> Measure the whole native method </h2> <p>A common performance mistake is to measure only JavaScript execution.</p> <p>For example:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">start</span> <span class="o">=</span> <span class="nx">performance</span><span class="p">.</span><span class="nf">now</span><span class="p">();</span> <span class="kd">const</span> <span class="nx">token</span> <span class="o">=</span> <span class="nx">SecureStorage</span><span class="p">.</span><span class="nf">getAccessToken</span><span class="p">();</span> <span class="nx">console</span><span class="p">.</span><span class="nf">log</span><span class="p">(</span><span class="s2">`Completed in </span><span class="p">${</span><span class="nx">performance</span><span class="p">.</span><span class="nf">now</span><span class="p">()</span> <span class="o">-</span> <span class="nx">start</span><span class="p">}</span><span class="s2"> ms`</span><span class="p">);</span> </code></pre> </div> <p>This can reveal blocking time from the JavaScript side.</p> <p>But deeper investigation may require native instrumentation.</p> <p>On Android, developers can measure sections using native timing tools or platform profilers.</p> <p>On iOS, Instruments can help identify time spent in storage, cryptography, object conversion and thread blocking.</p> <p>The useful questions are:</p> <ol> <li>How long does the method take on average?</li> <li>What is the worst observed latency?</li> <li>Does the duration change when the value is not cached?</li> <li>Does the method perform disk access?</li> <li>Does it allocate or convert large objects?</li> <li>Is it called during rendering or navigation?</li> <li>How many times is it called during one user action?</li> <li>How does it behave on lower-end devices?</li> </ol> <p>Average timing alone is not enough.</p> <p>A method that normally completes quickly but occasionally blocks for 40 milliseconds can still create visible UI problems.</p> <h2> Watch for repeated synchronous calls </h2> <p>The cost of one method call may appear acceptable.</p> <p>The call frequency may be the actual problem.</p> <p>Consider:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="nx">items</span><span class="p">.</span><span class="nf">map</span><span class="p">(</span><span class="nx">item</span> <span class="o">=&gt;</span> <span class="nx">NativeFormatter</span><span class="p">.</span><span class="nf">format</span><span class="p">(</span><span class="nx">item</span><span class="p">));</span> </code></pre> </div> <p>If there are 500 items, the application performs 500 synchronous native calls.</p> <p>Even if each call takes only a fraction of a millisecond, the combined cost can become significant.</p> <p>A stronger native API might process the entire collection:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="kd">const</span> <span class="nx">formattedItems</span> <span class="o">=</span> <span class="k">await</span> <span class="nx">NativeFormatter</span><span class="p">.</span><span class="nf">formatBatch</span><span class="p">(</span><span class="nx">items</span><span class="p">);</span> </code></pre> </div> <p>This can reduce:</p> <ul> <li>repeated boundary crossings</li> <li>repeated allocations</li> <li>thread coordination</li> <li>per-call conversion overhead</li> </ul> <p>Batching is especially useful when working with:</p> <ul> <li>database rows</li> <li>image metadata</li> <li>sensor records</li> <li>BLE data</li> <li>file lists</li> <li>cryptographic operations</li> <li>large arrays produced by native SDKs</li> </ul> <p>The correct API shape often matters as much as the implementation.</p> <h2> Synchronous methods create architectural coupling </h2> <p>There is another risk beyond immediate performance.</p> <p>A synchronous API can become slow later.</p> <p>Version one may return an in-memory value:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight typescript"><code><span class="nf">getUserRegion</span><span class="p">():</span> <span class="kr">string</span><span class="p">;</span> </code></pre> </div> <p>A future requirement may change the implementation:</p> <ul> <li>read the region from encrypted storage</li> <li>migrate an old value</li> <li>consult remote configuration</li> <li>query a native SDK</li> <li>support multiple user profiles</li> </ul> <p>The JavaScript interface remains synchronous, but the native responsibility has become heavier.</p> <p>Now the team faces a difficult choice:</p> <ul> <li>keep blocking JavaScript</li> <li>introduce hidden caching</li> <li>change the public API</li> <li>maintain both synchronous and asynchronous versions</li> </ul> <p>For values likely to require I/O in the future, starting with an asynchronous contract is often safer.</p> <p>API design should consider how the operation may evolve, not only how it works today.</p> <h2> A practical decision rule </h2> <p>Before exposing a synchronous TurboModule method, ask:</p> <blockquote> <p>Can this method always complete quickly without storage, networking, cryptography, large conversion or unpredictable system work?</p> </blockquote> <p>If the answer is uncertain, use an asynchronous API.</p> <p>A useful classification is:</p> <h3> Suitable for synchronous execution </h3> <ul> <li>constants</li> <li>cached primitives</li> <li>tiny calculations</li> <li>in-memory flags</li> <li>deterministic capability checks</li> </ul> <h3> Better suited for asynchronous execution </h3> <ul> <li>file access</li> <li>secure storage</li> <li>databases</li> <li>networking</li> <li>encryption and decryption</li> <li>compression</li> <li>image processing</li> <li>large data conversion</li> <li>hardware interaction</li> <li>operating-system services</li> </ul> <p>The method’s name is not enough.</p> <p>Review the complete native execution path.</p> <h2> Code-review checklist </h2> <p>When reviewing a synchronous TurboModule method, check:</p> <ul> <li>Does it perform disk or secure-storage access?</li> <li>Does it call another SDK or system service?</li> <li>Does it perform cryptography?</li> <li>Can it return a large object or array?</li> <li>Does it parse or serialise data?</li> <li>Can its latency change based on device state?</li> <li>Is it called during rendering?</li> <li>Is it called repeatedly inside a loop?</li> <li>Is the result already available in memory?</li> <li>Could the method become more expensive later?</li> </ul> <p>One concerning answer may be enough to reconsider the synchronous design.</p> <h2> Final takeaway </h2> <p>TurboModules can improve React Native’s native integration model.</p> <p>They make direct and synchronous interactions possible where appropriate.</p> <p>But capability should not become default behaviour.</p> <p>The safest rule is:</p> <blockquote> <p>Use synchronous native methods only for tiny, predictable and memory-resident operations.</p> </blockquote> <p>Keep storage, cryptography, networking, large data processing and unpredictable system work asynchronous.</p> <p>TurboModules reduce communication overhead.</p> <p>They do not remove the cost of native execution.</p> <p>Where have you seen a synchronous native method create unexpected UI latency in a React Native application?</p>

翻訳済み説明
