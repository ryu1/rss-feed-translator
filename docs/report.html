</h2> <p>Imagine your <code>/slow</code> endpoint starts timing out under load. Phase 3 — ROBUSTIFY (Steps 9-12) "Now make sure it doesn't fall apart." </code></pre> </div> <p>👨‍🦳 <strong>Uncle:</strong> Three phases, and inside each phase, the steps follow naturally — once you understand the problem, requirements follow. Once you know requirements, scale estimation follows. Once you know scale, your API and database design write themselves, because now they're grounded in real numbers instead of guesses. <strong>Never skip a step, and never skip the order.</strong> Skip scale estimation and jump straight to "I'll use Kafka" — you've just guessed. A senior engineer never guesses when they could calculate.</p> <h2> Part 2: Phase 1 — Understand (Steps 1-4) </h2> <h3> Step 1 — Understand the Problem </h3> <p>👦 <strong>Nephew:</strong> Interviewer says "Design a URL Shortener." What do I actually say first?</p> <p>👨‍🦳 <strong>Uncle:</strong> You do <strong>not</strong> say "I'll use Redis." You ask questions, out loud, before touching architecture:</p> <ul> <li>What's the main use case — is this for individual users, or a marketing team doing bulk campaigns?</li> <li>What features are required versus optional?</li> <li>Any constraints I should know about (must the short URL be a specific length, must it be guessable-resistant, etc.)?</li> </ul> <p>This alone — pausing to ask instead of diving in — signals seniority before you've said a single technical word.</p> <h3> Step 2 — Requirements: Split Functional vs Non-Functional </h3> <p>👨‍🦳 <strong>Uncle:</strong> Always split these into two explicit buckets. Interviewers love this separation because it shows systematic thinking, not just listing random features.</p> <p><strong>Functional Requirements — WHAT the system does:</strong></p> <ul> <li>User submits a long URL</li> <li>System generates a short URL</li> <li>Redirecting the short URL takes you to the original</li> <li>(Optional) Analytics on clicks</li> <li>(Optional) Expiration after 30 days</li> </ul> <p><strong>Non-Functional Requirements — HOW WELL it should behave:</strong></p> <ul> <li>99.99% availability</li> <li>Low latency (&lt;100ms)</li> <li>Scalable to millions of users</li> <li>Secure (no URL guessing)</li> </ul> <p>👦 <strong>Nephew:</strong> Why does splitting them matter so much? Isn't it all just "requirements"?</p> <p>👨‍🦳 <strong>Uncle:</strong> Because they drive <strong>completely different architecture decisions</strong>. Functional requirements shape your API and data model. Non-functional requirements shape your infrastructure choices — availability targets push you toward replication and multi-region thinking; latency targets push you toward caching; scale targets push you toward horizontal scaling and sharding. Mixing them together is how candidates end up designing something that "does the right things" but "does them the wrong way" — technically correct features, but the wrong infrastructure underneath.</p> <h3> Step 3 — Non-Functional Requirements, One Level Deeper </h3> <p>👨‍🦳 <strong>Uncle:</strong> Let's go deeper here, because this is where seniority really shows. There isn't just one non-functional requirement — there's a whole family, and different systems prioritize different members of that family:</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>NFR</th> <th>Question It Answers</th> <th>Typical Tools</th> </tr> </thead> <tbody> <tr> <td>Scalability</td> <td>Can it grow 10x without breaking?</td> <td>Horizontal scaling, sharding, caching</td> </tr> <tr> <td>Availability</td> <td>Is the system up? (99.99% = ~52 min downtime/year)</td> <td>Replicas, failover, load balancing</td> </tr> <tr> <td>Reliability</td> <td>Can users trust the system?</td> <td>Transactions, replication, retry logic</td> </tr> <tr> <td>Consistency</td> <td>Do all users see the same data? (Strong vs Eventual)</td> <td>ACID DB, leader-based writes, quorum</td> </tr> <tr> <td>Latency</td> <td>How fast is the response?</td> <td>Caching, CDN, database indexes</td> </tr> <tr> <td>Throughput</td> <td>How many requests/second?</td> <td>Message queues, load balancing</td> </tr> <tr> <td>Fault Tolerance</td> <td>What happens if a component fails?</td> <td>Replication, backup, failover</td> </tr> <tr> <td>Durability</td> <td>Does data survive crashes?</td> <td>Persistent storage, backups, WAL</td> </tr> <tr> <td>Security</td> <td>Protected from attacks?</td> <td>HTTPS, encryption, rate limiting, auth</td> </tr> <tr> <td>Maintainability</td> <td>Can engineers easily modify it?</td> <td>Microservices, CI/CD, monitoring</td> </tr> </tbody> </table></div> <p><strong>Memory trick — S-A-C-L-R-F-D-S:</strong> Scalability, Availability, Consistency, Latency, Reliability, Fault tolerance, Durability, Security.</p> <p>👦 <strong>Nephew:</strong> Do I need all ten for every system?</p> <p>👨‍🦳 <strong>Uncle:</strong> No — and knowing <em>which ones matter most for this specific system</em> is exactly what separates mid-level from senior. Look at this comparison:</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>System</th> <th>Most Important NFRs</th> <th>Why</th> </tr> </thead> <tbody> <tr> <td>WhatsApp</td> <td>Scalability, Availability, Latency</td> <td>Billions of messages, can't afford downtime, must feel instant</td> </tr> <tr> <td>Netflix</td> <td>Availability, Scalability, Bandwidth</td> <td>Millions streaming simultaneously, video needs huge bandwidth</td> </tr> <tr> <td>Banking</td> <td>Consistency, Durability, Security</td> <td>Money is involved — zero tolerance for data loss or inconsistency</td> </tr> <tr> <td>Trading Platform</td> <td>Consistency, Latency, Reliability</td> <td>Milliseconds matter, accuracy is everything</td> </tr> <tr> <td>URL Shortener</td> <td>Availability, Scalability, Latency</td> <td>Simple functionality, but must handle massive, simple traffic</td> </tr> </tbody> </table></div> <p>👨‍🦳 <strong>Uncle:</strong> Saying "for a banking system, I'd prioritize strong consistency and durability over raw availability, because losing or double-counting money is unacceptable, even if it costs us a bit of uptime" — that one sentence, said out loud, tells the interviewer you actually understand tradeoffs, not just terminology.</p> <h3> Step 4 — Scale Estimation — The Step Everyone Skips (Don't) </h3> <p>👨‍🦳 <strong>Uncle:</strong> This is, without exaggeration, the single most important step in the entire framework. Most candidates skip straight from requirements to architecture. That's the mistake that separates a mid-level answer from a senior one. <strong>You cannot choose the right architecture without knowing the numbers first.</strong></p> <p><strong>Ask these before calculating anything:</strong></p> <ul> <li>How many Daily Active Users (DAU)?</li> <li>How many requests per day?</li> <li>What's the read/write ratio?</li> <li>How much storage per record?</li> </ul> <p><strong>Why this single step tells you almost everything:</strong></p> <ul> <li>Immediately tells you if you need caching</li> <li>Tells you your database size, and whether one server can hold it</li> <li>Tells you your read/write pattern (read-heavy? write-heavy?)</li> <li>Tells you if a single server is even viable, or if you need to design for distribution from day one</li> </ul> <p>We'll do the actual math in the next section — but understand <em>why</em> it comes first: everything after this step (API design, database choice, caching strategy, number of servers) is a direct <em>consequence</em> of these numbers, not a separate creative decision.</p> <h2> Part 3: Back-of-the-Envelope Calculations — The Skill That Makes You Sound Senior Instantly </h2> <p>👨‍🦳 <strong>Uncle:</strong> Before you ever say "Redis," "MongoDB," "Kafka," or "CDN" out loud, you should be able to answer: how many users? How many requests? How much storage? How much bandwidth? How much memory? Otherwise you're designing blind, guessing dressed up as confidence.</p> <h3> Memorize these cold — memory units </h3> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Unit</th> <th>Bytes</th> <th>Everyday Example</th> </tr> </thead> <tbody> <tr> <td>1 KB</td> <td>10³ = 1,000</td> <td>Small text file</td> </tr> <tr> <td>1 MB</td> <td>10⁶ = 1,000,000</td> <td>A song, small image</td> </tr> <tr> <td>1 GB</td> <td>10⁹ = 1,000,000,000</td> <td>A movie, many songs</td> </tr> <tr> <td>1 TB</td> <td>10¹² = 1,000,000,000,000</td> <td>A large database</td> </tr> </tbody> </table></div> <p><strong>And one time constant you'll use in nearly every calculation:</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>1 Day = 86,400 seconds </code></pre> </div> <p>👦 <strong>Nephew:</strong> Why 86,400 specifically? Where does that even come from?</p> <p>👨‍🦳 <strong>Uncle:</strong> 24 hours × 60 minutes × 60 seconds = 86,400. Simple, but if you fumble this number live in an interview trying to do 24×60×60 in your head under pressure, it costs you composure at the worst possible moment. Just memorize it as a fact.</p> <h3> The Five Formulas You Must Know By Heart </h3> <p><strong>Formula 1 — QPS (Queries Per Second), the most important one:</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>QPS = Total Requests Per Day / 86,400 </code></pre> </div> <p><em>Example:</em> 100 million requests/day → 100,000,000 / 86,400 ≈ <strong>1,157 QPS</strong></p> <p><strong>Peak traffic rule — always design for peak, never average:</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Peak QPS = Average QPS × 3 to 5 </code></pre> </div> <p><em>Example:</em> 1,157 × 5 ≈ <strong>5,785 QPS</strong></p> <p>👨‍🦳 <strong>Uncle:</strong> Why multiply by 3-5? Because average traffic hides the truth — real traffic isn't smooth, it spikes around specific hours, campaigns, or events. If you design a system that can <em>only</em> handle the average, the very first traffic spike takes it down. Designing for peak is designing for reality, not for a spreadsheet.</p> <p><strong>Formula 2 — Storage per day:</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Storage = Records Per Day × Size Per Record </code></pre> </div> <p><em>Example:</em> 10 million new URLs/day × 500 bytes = <strong>5 GB/day</strong></p> <p><strong>Formula 3 — Yearly storage:</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Yearly Storage = Daily Storage × 365 </code></pre> </div> <p><em>Example:</em> 5 GB/day × 365 = <strong>1.8 TB/year</strong></p> <p>👦 <strong>Nephew:</strong> Why does yearly storage matter so much? Isn't 5 GB/day tiny?</p> <p>👨‍🦳 <strong>Uncle:</strong> Because a single day's number looks harmless, and that's exactly the trap. 1.8 TB/year tells you something concrete: somewhere around year 2-3, you'll likely need sharding or a distributed database, because a single machine's disk starts becoming a real constraint. This number is what lets you say, confidently, in an interview: "for year one, a single well-provisioned database is fine; by year three, we'll need to shard" — a genuinely senior-sounding sentence, backed by actual math, not vibes.</p> <p><strong>Formula 4 — Bandwidth:</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Bandwidth = Peak QPS × Response Size </code></pre> </div> <p><em>Example:</em> 5,000 QPS × 2 KB = <strong>10 MB/sec</strong></p> <p>👨‍🦳 <strong>Uncle:</strong> This tells you whether your network card, your load balancer, and your CDN strategy can actually keep up — especially critical for anything media-heavy (we'll see this explode in the Netflix example shortly).</p> <p><strong>Formula 5 — Cache memory:</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Cache Memory = Hot Data Size × 1.2 (overhead factor) </code></pre> </div> <p><em>Example:</em> 10 million hot URLs × 500 bytes = 5 GB → with overhead, <strong>5-8 GB Redis needed</strong></p> <p>👦 <strong>Nephew:</strong> Why the 1.2 overhead factor?</p> <p>👨‍🦳 <strong>Uncle:</strong> Redis (and caches generally) don't store <em>just</em> your raw data — there's metadata, key overhead, internal data structure costs. The 1.2 multiplier is a practical safety margin so you don't under-provision and hit unexpected evictions the moment real traffic arrives.</p> <h3> The 7 Numbers You Should Calculate, Every Single Time </h3> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>1. DAU (Daily Active Users) 2. Total Requests/day 3. Read Requests/day 4. Write Requests/day 5. Read QPS (average) 6. Peak QPS (for capacity planning) 7. Storage/year </code></pre> </div> <p>👨‍🦳 <strong>Uncle:</strong> Build the habit of running through exactly these seven, in this order, for <em>every</em> design question, without exception — URL shortener, chat app, notification system, rate limiter, file upload, doesn't matter. This becomes muscle memory, and muscle memory is what survives interview nerves.</p> <h3> Seeing It All Together — Three Full Worked Examples </h3> <p>👨‍🦳 <strong>Uncle:</strong> Numbers in isolation don't teach you anything. Let's see how the <em>same</em> seven-number process produces completely different architectures depending on the system.</p> <h4> Example A — URL Shortener (a Read-Heavy System) </h4> a different system, points you toward a totally different toolset — without me ever having to "guess" which technology sounds impressive?</p> <p>👨‍🦳 <strong>Uncle:</strong> That's the entire point of this discipline. The numbers <em>are</em> the design decision. You're not choosing Kafka because it's trendy — you're choosing it because 57,870 peak QPS of writes, sustained, genuinely requires a queue-and-distribute approach; a single database simply cannot absorb that write rate directly.</p> <h4> Example C — Netflix (a Bandwidth-Heavy System) </h4> <p><strong>Assumptions:</strong> 10 million concurrent users, 5 Mbps per video stream.</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Metric</th> <th>Calculation</th> <th>Result</th> <th>Implication</th> </tr> </thead> <tbody> <tr> <td>Bandwidth</td> <td>10M × 5 Mbps</td> <td>50 Tbps</td> <td>Literally impossible to serve from one origin</td> </tr> </tbody> </table></div> <p><strong>Architecture decision this produces:</strong> this single number rules out "just a bigger server" entirely — it <em>forces</em> a CDN (Akamai, CloudFront) with regional edge caching so videos are served from servers physically near each user, a separate lightweight "control plane" for metadata (small, fast) versus a separate "data plane" for actual video delivery (huge, offloaded to the CDN), and adaptive bitrate streaming to adjust quality per connection.</p> <p>👨‍🦳 <strong>Uncle:</strong> Notice — for Netflix, we didn't even need to discuss database schema first. The <em>bandwidth</em> number alone was so extreme it dictated the entire shape of the architecture before anything else mattered. This is why scale estimation comes third in the framework, right after requirements — sometimes one number single-handedly eliminates 90% of your design options before you've drawn a single box.</p> <h2> Part 4: Phase 2 — Design (Steps 5-8) </h2> <h3> Step 5 — Design APIs Before the Database </h3> <p>👨‍🦳 <strong>Uncle:</strong> Define your API contracts <em>before</em> your database schema. Why this order? Because the API describes what the <em>outside world</em> actually needs from your system — and that's what should shape your data model, not the other way around. Designing the database first often leads to a schema that's convenient to build but awkward for the actual use cases.</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Endpoint</th> <th>Method</th> <th>Request</th> <th>Response</th> </tr> </thead> <tbody> <tr> <td><code>/shorten</code></td> <td>POST</td> <td><code>{"url": "https://google.com"}</code></td> <td><code>{"shortUrl": "abc123"}</code></td> </tr> <tr> <td><code>/:shortCode</code></td> <td>GET</td> <td>—</td> <td> <code>301 Redirect</code> → original URL</td> </tr> <tr> <td><code>/stats/:shortCode</code></td> <td>GET</td> <td>—</td> <td><code>{"clicks": 1000, "created": "..."}</code></td> </tr> </tbody> </table></div> <p>👦 <strong>Nephew:</strong> Why <code>301</code> for the redirect specifically?</p> <p>👨‍🦳 <strong>Uncle:</strong> A <code>301</code> is a <em>permanent</em> redirect — browsers and search engines are allowed to cache it, meaning repeat visits to the same short URL can skip your server entirely after the first hit. This one HTTP status code choice is itself a small but real caching decision, baked directly into your API design.</p> <h3> Step 6 — Design the Database </h3> <p>👨‍🦳 <strong>Uncle:</strong> Now think about what data must actually be stored, and exactly how big each field is — because this feeds directly back into your storage calculations from Step 4.</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Field</th> <th>Type</th> <th>Size</th> <th>Index?</th> </tr> </thead> <tbody> <tr> <td><code>_id</code></td> <td>UUID</td> <td>16 B</td> <td>Primary Key</td> </tr> <tr> <td><code>shortCode</code></td> <td>String(8)</td> <td>8 B</td> <td>Unique Index</td> </tr> <tr> <td><code>originalUrl</code></td> <td>String(2048)</td> <td>~300 B</td> <td>No</td> </tr> <tr> <td><code>userId</code></td> <td>UUID</td> <td>16 B</td> <td>Index</td> </tr> <tr> <td><code>createdAt</code></td> <td>Timestamp</td> <td>8 B</td> <td>No</td> </tr> <tr> <td><code>expiresAt</code></td> <td>Timestamp</td> <td>8 B</td> <td>Index</td> </tr> <tr> <td><strong>Total per record</strong></td> <td></td> <td><strong>~500 B</strong></td> <td></td> </tr> </tbody> </table></div> <p>👦 <strong>Nephew:</strong> Why index <code>expiresAt</code>? Nobody's directly querying "give me the record that expires at this exact millisecond."</p> <p>👨‍🦳 <strong>Uncle:</strong> Good instinct to question it — the real use case is a background cleanup job running periodically, querying "find all records where <code>expiresAt</code> &lt; now." Without an index on that field, that query becomes a full table scan across your entire dataset, every single time it runs, which gets catastrophically slow as your table grows into the millions. This is exactly the kind of quiet, unglamorous indexing decision that separates a schema that scales gracefully from one that silently degrades over a year.</p> <h3> SQL or NoSQL — The Actual Decision Tree </h3> <p>👨‍🦳 <strong>Uncle:</strong> This question comes up in nearly every design interview, and most candidates answer it with a vibe ("SQL feels more reliable") instead of a reason. Here's the actual decision tree:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Need relationships / multi-table joins / transactions across entities? → SQL (PostgreSQL) — ACID guarantees, strong consistency Need massive horizontal scale (billions of records, simple access patterns)? → NoSQL (MongoDB, Cassandra) — flexible schema, scales out easily Need both — some strongly relational data AND some massive-scale data? → Hybrid — e.g. PostgreSQL for core transactional data (users, billing), a NoSQL store for high-volume, simpler data (activity logs, messages) </code></pre> </div> <p><strong>A quick reference decision tree for the broader toolkit, not just databases:</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Need data relationships? → SQL (PostgreSQL) Need massive scale (billions)? → NoSQL (MongoDB, Cassandra) Read-heavy (e.g. 10:1 ratio)? → Add a Redis cache Write-heavy (e.g. near 1:1 ratio)? → Message queue (Kafka) Need instant, real-time updates? → WebSockets + Redis pub/sub Need to handle 10x growth? → Horizontal scaling + sharding Need &lt;100ms latency? → In-memory cache + CDN Need ACID guarantees? → SQL with transactions Can tolerate occasional data loss? → Redis alone is fine Cannot afford any data loss? → Persistent storage + backups </code></pre> </div> <p>👦 <strong>Nephew:</strong> Can I just say "MongoDB" every time to sound modern?</p> <p>👨‍🦳 <strong>Uncle:</strong> Please don't. Saying "MongoDB because it scales" without being able to explain <em>why</em> your specific access pattern needs that scale, and what you're giving up (joins, strong consistency by default) to get it, is exactly the technology-first thinking we started this whole conversation warning you against. Always justify the choice with the actual requirement it satisfies.</p> <h3> Step 7 — High Level Design (HLD): Draw the Big Boxes First </h3> <p>👨‍🦳 <strong>Uncle:</strong> Now, and only now, draw the big picture. Don't explain internals yet — that's the next step. HLD is about showing the overall shape of the system.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code> Client (Web / Mobile) │ ▼ Load Balancer (Nginx) │ ┌──────┼──────┐ ▼ ▼ ▼ [API-1] [API-2] [API-3] │ ▼ Redis Cache (Hot URLs) │ ▼ Primary DB ⇄ Replica DB │ ▼ Backup (Daily Snapshot) </code></pre> </div> <p>👨‍🦳 <strong>Uncle:</strong> Every box on this diagram exists to answer one specific number from Step 4 — the Load Balancer exists because peak QPS exceeds what one server can handle; Redis exists because read QPS is high and reads are repeated (hot URLs); the Replica exists because availability requirements demand no single point of failure; the Backup exists because durability requirements demand recoverability. <strong>If you can't point to the number that justifies a box, that box probably shouldn't be on your diagram yet.</strong></p> <h3> Step 8 — Deep Dive Each Component </h3> <p>👨‍🦳 <strong>Uncle:</strong> Now, explain how each box actually works internally.</p> <p><strong>Load Balancer:</strong></p> <ul> <li>Distributes requests, typically round-robin or least-connections</li> <li>If a server goes down, it auto-detects (via health checks) and stops routing to it — automatic failover</li> <li>Can use hash-based routing for session stickiness when needed</li> </ul> <p><strong>Cache (Redis):</strong></p> <ul> <li>Stores hot (frequently accessed) URLs</li> <li>TTL (time-to-live) around 1 day, tuned to your data's actual staleness tolerance</li> <li>Eviction policy: LRU (Least Recently Used) when memory is full</li> <li> <strong>Request flow:</strong> check Redis first → found? return it. Not found? → query the database → store the result in Redis → return it. This exact pattern is called <strong>cache-aside</strong>, and it's the most common caching pattern you'll use.</li> </ul> <p><strong>Database:</strong></p> <ul> <li>The permanent source of truth — Redis can lose data (it's a cache), the database cannot</li> <li>Indexes on <code>shortCode</code>, <code>userId</code>, <code>expiresAt</code> </li> <li>A background job periodically deletes expired records</li> </ul> <p><strong>Replication:</strong></p> <ul> <li>Primary handles all writes; Replicas handle reads</li> <li>If the primary fails, a replica is automatically promoted to become the new primary</li> <li>Uses binary-log (bin-log) replication under the hood to keep replicas consistent with the primary</li> </ul> <h2> Part 5: Phase 3 — Robustify (Steps 9-12) </h2> <h3> Step 9 — Scaling Strategy </h3> <p>👨‍🦳 <strong>Uncle:</strong> Interviewers expect this conversation without being asked. Two categories:</p> <p><strong>Vertical Scaling</strong> — add more CPU/RAM to the existing machine. Simple, but limited — you hit a ceiling fast, and the cost curve is brutal ($10K server → $50K server for incremental gains).</p> <p><strong>Horizontal Scaling</strong> — add more machines instead. Preferred, because it scales close to indefinitely, and commodity servers are cheap: "10 servers instead of 1 huge server."</p> <p><strong>Database scaling specifically has its own two levers:</strong></p> <ul> <li> <strong>Read replicas</strong> — scale read QPS by adding more replicas that only serve reads</li> <li> <strong>Sharding</strong> — scale write QPS by splitting data across multiple independent databases, typically by a hash of some key (like <code>userId</code>). Each shard commonly runs its own primary + replicas setup internally.</li> </ul> <p>👦 <strong>Nephew:</strong> When do I reach for sharding versus just adding more read replicas?</p> <p>👨‍🦳 <strong>Uncle:</strong> Read replicas solve <strong>read</strong> bottlenecks — they don't help at all if your <em>writes</em> are the problem, because all replicas still ultimately depend on one primary absorbing every write. When writes themselves exceed what a single primary can handle (like our WhatsApp example — 57,870 peak write QPS), replicas can't save you; you need sharding, splitting the write load itself across multiple independent primaries.</p> <h3> Step 10 — Reliability &amp; Failure Handling </h3> <p>👨‍🦳 <strong>Uncle:</strong> Most candidates forget this entirely. Senior engineers ask, unprompted: <strong>"what if this specific component crashes?"</strong> — for every component on the diagram.</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Failure Scenario</th> <th>Solution</th> </tr> </thead> <tbody> <tr> <td>API server crashes</td> <td>Load balancer detects it via health checks, reroutes to healthy servers</td> </tr> <tr> <td>Cache (Redis) dies</td> <td>Fall back directly to the database (slower, but the system still works)</td> </tr> <tr> <td>Primary DB fails</td> <td>A replica is automatically promoted to primary</td> </tr> <tr> <td>Network partition</td> <td>Message queue buffers requests; retry once connectivity returns</td> </tr> <tr> <td>Data corruption</td> <td>Point-in-time recovery from backup</td> </tr> <tr> <td>Cascading failure</td> <td>Circuit breaker stops sending requests to the already-struggling component, giving it room to recover instead of piling on more load</td> </tr> </tbody> </table></div> <p>👦 <strong>Nephew:</strong> What's a "cascading failure," actually, in plain terms?</p> <p>👨‍🦳 <strong>Uncle:</strong> Imagine your database is struggling and responding slowly. Every API server, instead of backing off, keeps sending it <em>more</em> requests (because users keep retrying), which makes the database even slower, which makes users retry even more — a downward spiral that takes the <em>entire</em> system down over one struggling component. A <strong>circuit breaker</strong> watches for repeated failures from a component and, past a threshold, temporarily stops sending it traffic entirely — like a real household circuit breaker cutting power before a small fault becomes a fire.</p> <h3> Step 11 — Monitoring &amp; Observability, Including Logging and Tracing </h3> <p>👨‍🦳 <strong>Uncle:</strong> You wanted specific depth here, so let's actually go deep — this is the step that turns "it works on my laptop" into "we know exactly what's happening in production, right now, and can find any bug within minutes."</p> <p><strong>Metrics — the numbers, tracked over time (Prometheus):</strong></p> <ul> <li>QPS (requests/sec)</li> <li>Latency — and crucially, <strong>percentiles</strong>, not just averages: p50, p95, p99. A p99 of 3 seconds means 1% of your users are waiting 3+ seconds — a real, painful problem an <em>average</em> number would completely hide.</li> <li>Error rate</li> <li>Cache hit ratio</li> <li>Database connection pool usage</li> <li>CPU, Memory, Disk</li> </ul> <p><strong>Visualization (Grafana):</strong> turns those raw metrics into real-time dashboards, with alert thresholds — "page someone if error rate exceeds 2% for more than 5 minutes."</p> <p><strong>Logs — the detailed, per-event record of what actually happened (the ELK stack):</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Application logs (e.g. via Pino) │ ▼ Logstash (collects and processes logs from everywhere) │ ▼ Elasticsearch (stores and indexes logs for fast search) │ ▼ Kibana (search and visualize logs instantly) </code></pre> </div> <p>👦 <strong>Nephew:</strong> How is this different from metrics? Aren't they both "monitoring"?</p> <p>👨‍🦳 <strong>Uncle:</strong> Metrics tell you <strong>that</strong> something's wrong and roughly <strong>where</strong> — "error rate on <code>/shorten</code> jumped to 8% at 3:14 PM." Logs tell you <strong>exactly what happened</strong>, to which specific request, with which specific data — you search Elasticsearch for all logs tied to that endpoint in that time window and read the actual error messages and stack traces. Metrics narrow the search; logs solve the case — this is the exact same principle from our earlier debugging conversations, just now applied at the system-design level instead of the single-request level.</p> <p><strong>Tracing (Jaeger) — following one request across many services:</strong></p> <p>👨‍🦳 <strong>Uncle:</strong> In a system with multiple services (API server → cache → database → maybe a downstream notification call), a single slow user request might touch five different components. <strong>Distributed tracing</strong> assigns one unique trace ID to that request at the very start, and every service it touches logs its own timing against that same trace ID. This lets you reconstruct the <em>entire</em> journey of one specific slow request and see exactly which one of the five components was the actual bottleneck — instead of guessing by staring at five separate dashboards trying to correlate timestamps by eye.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Trace ID: abc-123 ├─ API Server: 12ms ├─ Redis lookup: 2ms ├─ Database query: 340ms ← the bottleneck, immediately obvious └─ Response sent: 4ms </code></pre> </div> <p>👨‍🦳 <strong>Uncle:</strong> In an interview, mentioning tracing specifically (not just "logs and metrics") is a strong signal — it shows you've actually operated a real multi-service system in production, not just read about monitoring in theory.</p> <h3> Step 12 — Security </h3> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Concern</th> <th>Tool/Approach</th> </tr> </thead> <tbody> <tr> <td>Authentication</td> <td>JWT, OAuth, or Session-based</td> </tr> <tr> <td>Authorization</td> <td>Role-Based Access Control</td> </tr> <tr> <td>Rate Limiting</td> <td>e.g. 100 requests/minute per user</td> </tr> <tr> <td>Encryption in transit</td> <td>HTTPS (TLS)</td> </tr> </tbody> </table></div> <p>👨‍🦳 <strong>Uncle:</strong> Always mention this, even briefly — interviewers specifically listen for it, and its absence is noticed even if everything else was excellent.</p> <h3> Step 13 (Really Step 12) — Bottlenecks &amp; Tradeoffs: The Senior-Level Closer </h3> <p>👨‍🦳 <strong>Uncle:</strong> This is the single most important discussion of the entire interview, and it's where junior and senior answers diverge the most sharply.</p> <p><strong>Junior engineer thinking:</strong> "Use Redis. Use MongoDB. Done."</p> <p><strong>Senior engineer thinking:</strong> "What happens if Redis fails? What happens if MongoDB fails? What happens if an entire region goes down? How will we monitor it? How will we scale it? How will we recover the data? How much traffic can it actually handle before it breaks?"</p> <p><strong>Where bottlenecks typically appear, and their remedies:</strong></p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Component</th> <th>Problem</th> <th>Solution</th> </tr> </thead> <tbody> <tr> <td>Database</td> <td>Too many reads</td> <td>Add cache + read replicas</td> </tr> <tr> <td>Database</td> <td>Too many writes</td> <td>Sharding</td> </tr> <tr> <td>Database</td> <td>Large result sets</td> <td>Pagination</td> </tr> <tr> <td>Network</td> <td>Large payloads</td> <td>Compression (gzip) + CDN for geographically distributed users</td> </tr> <tr> <td>Network</td> <td>Bandwidth exceeds capacity</td> <td>Rate limiting</td> </tr> <tr> <td>CPU</td> <td>Heavy processing</td> <td>Background jobs (Kafka-driven workers)</td> </tr> <tr> <td>CPU</td> <td>Auth overhead</td> <td>JWT (no database lookup needed per request, unlike sessions)</td> </tr> </tbody> </table></div> <p><strong>And always, always mention the CAP theorem tradeoff explicitly:</strong></p> <p>👨‍🦳 <strong>Uncle:</strong> In any distributed system, during a network partition, you can only guarantee two of three: <strong>C</strong>onsistency, <strong>A</strong>vailability, <strong>P</strong>artition tolerance — and partition tolerance isn't really optional in a real distributed system, so in practice you're choosing between consistency and availability. Most systems lean toward <strong>availability</strong>, accepting eventual consistency, because a slightly stale response is usually better for the user than no response at all. Banking systems are the notable exception — they lean toward consistency, because a stale balance is genuinely dangerous, and they accept the availability cost.</p> <p>👦 <strong>Nephew:</strong> So ending my answer with "here's where this breaks first, and here's the tradeoff we're accepting" is actually the strongest way to finish?</p> <p>👨‍🦳 <strong>Uncle:</strong> Exactly — it shows the interviewer you're not just capable of building the happy path; you already know where the cracks will appear before they've asked. That's the entire senior-versus-mid-level difference, distilled into one closing statement.</p> <h2> Part 6: Caching, Properly — The Piece Every Design Leans On </h2> <p>👦 <strong>Nephew:</strong> We used Redis in almost every example. Can we go deeper on caching itself?</p> <p>👨‍🦳 <strong>Uncle:</strong> Should have known you'd ask — caching deserves its own deep dive, since it's the single highest-leverage tool in nearly every system design answer.</p> <p><strong>What to cache:</strong> hot data — things read far more often than they're written or changed. A hot URL redirect, a user's profile info, a product page. Never cache things that change every request or are highly personalized and rarely re-read.</p> <p><strong>Caching patterns:</strong></p> <ul> <li> <strong>Cache-aside (the one we used above)</strong> — application checks cache first; on a miss, reads from the database and populates the cache. Simple, and the most common pattern by far.</li> <li> <strong>Write-through</strong> — every write goes to the cache <em>and</em> the database simultaneously, keeping them always in sync, at the cost of slightly slower writes.</li> <li> <strong>Write-behind (write-back)</strong> — writes go to the cache immediately and are asynchronously flushed to the database later. Fast writes, but a real risk of data loss if the cache fails before the flush happens — use only when that risk is genuinely acceptable.</li> </ul> <p><strong>TTL (Time To Live) strategy:</strong> every cached item should expire. Too short, and you lose most of the caching benefit (constant cache misses). Too long, and users see stale data. Tune it to how frequently the underlying data actually changes.</p> <p><strong>Eviction policy:</strong> when the cache is full, something has to go. <strong>LRU (Least Recently Used)</strong> is the most common default — it evicts whatever hasn't been touched in the longest time, on the reasonable assumption that recently-used data is more likely to be needed again soon.</p> <p>👦 <strong>Nephew:</strong> What if the cache and the database disagree — the classic stale data problem we talked about with React and RTK Query?</p> <p>👨‍🦳 <strong>Uncle:</strong> Exact same root problem, one level down the stack — same discipline applies. Whenever the underlying data changes, explicitly invalidate (delete or update) the corresponding cache key at that moment, rather than only relying on TTL expiry to eventually catch up. This is the system-design-level version of the "invalidate the tag when it changes" lesson from RTK Query — the specific tool changes, but the underlying discipline of the caching layer is identical everywhere you find one.</p> <h2> Part 7: Choosing the Right Async Tool — Redis vs RabbitMQ vs Kafka </h2> <p>👨‍🦳 <strong>Uncle:</strong> These three all solve "asynchronous communication between parts of your system," but with genuinely different guarantees — and choosing wrong here is a very common interview mistake.</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Feature</th> <th>Redis Pub/Sub</th> <th>RabbitMQ</th> <th>Kafka</th> </tr> </thead> <tbody> <tr> <td>Stores messages?</td> <td>No</td> <td>Temporary</td> <td>Persistent (forever, by default retention policy)</td> </tr> <tr> <td>Real-time delivery?</td> <td>Excellent</td> <td>Good</td> <td>Good</td> </tr> <tr> <td>Replay past events?</td> <td>No</td> <td>Limited</td> <td>Yes — excellent</td> </tr> <tr> <td>Great for background jobs?</td> <td>No</td> <td>Excellent</td> <td>Possible, but not its primary strength</td> </tr> <tr> <td>Throughput</td> <td>Medium</td> <td>Medium</td> <td>Excellent</td> </tr> <tr> <td>Typical use</td> <td>Presence, typing indicators</td> <td>Email, SMS, task queues</td> <td>Event streaming, analytics, audit logs</td> </tr> </tbody> </table></div> <p><strong>Three simple mental pictures that make this permanently stick:</strong></p> <ul> <li> <strong>Redis Pub/Sub = FM Radio.</strong> If you're tuned in right now, you hear it. If you weren't listening, it's gone forever — no replay.</li> <li> <strong>RabbitMQ = A Post Office.</strong> Your letter (message) is held temporarily. Once it's delivered and signed for, it's discarded.</li> <li> <strong>Kafka = A Library Book.</strong> The story stays on the shelf indefinitely. Anyone — including a brand-new team joining six months from now — can read it anytime, from the beginning if they want.</li> </ul> <p><strong>Real-world example — how WhatsApp actually uses all three, for different jobs:</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Typing Indicator → Redis Pub/Sub (not critical if occasionally lost) Email Verification → RabbitMQ (must eventually be delivered, reliably) Message Analytics → Kafka (millions of events; need persistence AND replay) </code></pre> </div> <p>👦 <strong>Nephew:</strong> So it's never "which one is best" — it's "which guarantee does <em>this specific job</em> actually need"?</p> <p>👨‍🦳 <strong>Uncle:</strong> Precisely — asking "does this message need to survive a crash? does it need to be replayed later by a system that doesn't exist yet? does losing an occasional message matter at all?" leads you to the right tool every time, far more reliably than memorizing "Kafka is for big companies."</p> <h2> Part 8: A Complete Worked Example — Designing a Notification System </h2> <p>👨‍🦳 <strong>Uncle:</strong> Let's run the <em>entire</em> framework, start to finish, on one real design, exactly the way you'd do it live in an interview. This is one of the best practice questions there is — it touches queues, async processing, retries, and failure handling all at once.</p> <p><strong>1. Understand the problem:</strong> "Design a Notification System." First question to ask: what channels — email, SMS, push, real-time, scheduled? Assume: Email, SMS, Push.</p> <p><strong>2. Functional Requirements:</strong> send email, send SMS, send push notification, support bulk notifications, retry failed notifications, view delivery status.</p> <p><strong>3. Non-Functional Requirements — the single most important question here:</strong> <em>can we afford to lose a notification?</em> Answer: <strong>No.</strong> This one answer shapes the entire architecture that follows — it immediately tells you a queue with persistence and retries is mandatory, not optional.</p> <p><strong>4. Scale Estimation:</strong></p> <ul> <li>10M total users, 1M DAU, 5 notifications/user/day → 1M × 5 = <strong>5M notifications/day</strong> </li> <li>QPS: 5,000,000 / 86,400 ≈ <strong>58 QPS</strong>, Peak: 58 × 5 ≈ <strong>290 QPS</strong> </li> <li> <strong>The critical edge case — bulk campaigns:</strong> marketing sends 1 million emails in a 10-minute window → 1,000,000 / 600 = <strong>1,666 notifications/sec</strong>. This is where the queue genuinely earns its place — a burst 5-6x higher than even your "peak" steady-state number.</li> <li>Storage: 5M × 500 bytes = 2.5 GB/day → × 365 = <strong>912 GB/year</strong> </li> </ul> <p><strong>5. Why not just call the email provider directly from the API?</strong><br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Bad: Order Service → Send Email directly → wait 2 seconds → user is stuck waiting Better: Order Service → Kafka → return success immediately → email sent asynchronously, later </code></pre> </div> <p>👦 <strong>Nephew:</strong> So the queue exists purely to stop the user from waiting on something they don't need to wait on?</p> <p>👨‍🦳 <strong>Uncle:</strong> That's one huge reason — but combined with our "cannot lose a notification" requirement from Step 3, there's a second, equally important reason: if the email provider is temporarily down, a direct call just fails and the notification is gone. With Kafka, the event sits safely in the queue regardless of whether the provider is currently up, and a worker can retry it whenever the provider recovers. The queue isn't just a performance optimization here — it's the actual mechanism satisfying your non-negotiable non-functional requirement.</p> <p><strong>8. Scaling:</strong> if 5M notifications/day becomes 500M, add more Kafka partitions and more workers per channel — they consume in parallel, so throughput scales roughly linearly with worker count.</p> <p><strong>9. Reliability &amp; Failure Handling — the heart of this design:</strong></p> <ul> <li>If the Email Provider (SendGrid) fails: retry 1, retry 2, retry 3 — still failing after that? → send to a <strong>Dead Letter Queue (DLQ)</strong> for manual inspection later, rather than silently dropping it. Kafka retains the event, so a worker can always come back and retry later.</li> <li>If a Worker crashes mid-processing: Kafka still has the event — it was never removed until successfully processed — so a new worker instance simply picks it up. No loss.</li> <li>If the Database goes down: Primary + Replica setup with automatic failover.</li> </ul> <p><strong>10. Monitoring:</strong> notifications sent (counter), failed notifications (counter), retry count (histogram), queue size (gauge), latency (histogram), and critically — <strong>Kafka consumer lag</strong> (how far behind the workers are from the incoming event stream; a rapidly growing lag number is often the very first sign of trouble, well before anything else looks wrong).</p> <p><strong>11. Security:</strong> JWT auth, authorization so only the notification's owner can view its status, rate limiting to prevent spam abuse, HTTPS encryption throughout.</p> <p>👦 <strong>Nephew:</strong> Every single answer in this walkthrough traced back to one of the twelve steps, in order, without me ever feeling lost.</p> <p>👨‍🦳 <strong>Uncle:</strong> That was the entire exercise. With tcpdump and Wireshark I look at the inter-packet delta first — a 30 fps stream should sit near 33 ms, not 100 ms — and then at gaps in the sequence number. Decide what you actually need to repair </h2> <p>Most of the time the diagnosis above points to one of three fixes: tweak the encoder profile, change the transport from UDP to TCP (or vice versa), or move the camera off a congested switch port. <blockquote> <p>The Linux DMA mapping API exists to solve two problems for driver authors: translating a CPU buffer into a bus address the device can use, and inserting the cache maintenance operations needed on non-coherent architectures. Coherent mappings (dma_alloc_coherent) are for small, long-lived control structures accessed by both CPU and device without explicit syncing, while streaming mappings (dma_map_single, dma_map_page, dma_map_sg) are for bulk data mapped just before a transfer and unmapped after, with explicit sync calls if the CPU touches the buffer in between. In the Linux 7.x source, every mapping with no custom DMA ops and no IOMMU takes the dma-direct fast path through dma_map_phys(), and CONFIG_DMA_API_DEBUG can validate that a driver's map and unmap calls are used correctly.</p> </blockquote> <p>If you write drivers for embedded Linux, the <strong>DMA mapping API</strong> is one of the interfaces you cannot avoid for long. The moment a device moves data into or out of memory on its own, without the CPU copying each byte, your driver has to tell the kernel how that memory should be prepared. Get it wrong and the symptoms are some of the hardest to debug in kernel work: data that is correct on a desktop x86 board but corrupted on an ARM target, or a buffer that reads back stale values only under load. This Deep Dive comes in two parts. First it covers the concepts and rules every driver author needs: coherent versus streaming mappings, the DMA mask, directions, and syncing. Then it goes one layer down and traces the kernel source that implements them, from the dispatch in <code>kernel/dma/mapping.c</code> to the arm64 cache hooks and the <code>CONFIG_DMA_API_DEBUG</code> facility. The source shown is from Linux 7.1, whose series reworked the DMA core to be physical-address based.</p> <h2> Three kinds of addresses </h2> <p>The first source of confusion is that DMA involves three different address spaces, and they are not interchangeable. The kernel works with virtual addresses, the kind returned by <code>kmalloc()</code> and stored in a <code>void *</code>. The memory management unit translates those into CPU physical addresses, the values you see in <code>/proc/iomem</code>. A device, however, sees a third kind of address called a bus address. On simple systems the bus address equals the physical address, but when an IOMMU or a host bridge sits between the device and memory, the two diverge.</p> <p>This matters because a device performing DMA uses bus addresses, and it has no access to the CPU's virtual memory system. You cannot hand a device a pointer from <code>kmalloc()</code> and expect it to work. The job of the DMA mapping API is to take a buffer the CPU can see and return a <code>dma_addr_t</code> value the device can use, setting up any IOMMU translation along the way. Every driver that touches DMA must include the header that defines this type:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight c"><code><span class="cp">#include</span> <span class="cpf">&lt;linux/dma-mapping.h&gt;</span><span class="cp"> </span></code></pre> </div> <h2> Why the DMA mapping API exists </h2> <p>Beyond address translation, the API solves a second problem: cache coherency. Many embedded SoCs have CPU caches that are not kept coherent with DMA traffic. If the CPU writes a buffer, the data may still be sitting in the cache when the device reads main memory, so the device sees old contents. In the other direction, the device writes main memory while the CPU still holds a cached copy, so the CPU reads stale data. The DMA mapping API is the single place where the kernel inserts the cache maintenance operations needed to handle this, in an architecture-independent way. On a fully coherent platform those operations compile down to almost nothing; on a non-coherent ARM board they become real cache flushes and invalidations. Your driver code stays the same either way.</p> <h2> Tell the kernel your addressing limits </h2> <p>Before mapping anything, a driver must declare how many address bits the device can drive. By default the kernel assumes 32-bit DMA addressing. You change that with a single call that covers both the streaming and coherent interfaces:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight c"><code><span class="k">if</span> <span class="p">(</span><span class="n">dma_set_mask_and_coherent</span><span class="p">(</span><span class="n">dev</span><span class="p">,</span> <span class="n">DMA_BIT_MASK</span><span class="p">(</span><span class="mi">64</span><span class="p">)))</span> <span class="p">{</span> <span class="n">dev_warn</span><span class="p">(</span><span class="n">dev</span><span class="p">,</span> <span class="s">"No suitable 64-bit DMA availablen"</span><span class="p">);</span> <span class="cm">/* fall back or refuse to probe */</span> <span class="p">}</span> </code></pre> </div> <p>The kernel saves this mask and uses it later when it allocates DMA addresses, so it never hands the device an address it cannot reach. Note that <code>dma_set_mask_and_coherent()</code> will not fail for masks of 32 bits or larger, so the common pattern is to set 64 bits when the device supports it and 32 bits otherwise, rather than trying a 64-bit call and falling back to 32. If the device has different limits for descriptors and for data, you can set the streaming and coherent masks separately with <code>dma_set_mask()</code> and <code>dma_set_coherent_mask()</code>.</p> <h2> Coherent mappings: allocate once, keep for the device's lifetime </h2> <p>A coherent mapping is memory for which a write by either the CPU or the device is immediately visible to the other, with no explicit flushing in your driver. Think of it as synchronous. You allocate it once, usually at probe time, and free it at removal. The classic uses are control structures the device polls continuously: network card ring descriptors, command mailboxes, or firmware microcode run out of main memory.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight c"><code><span class="n">dma_addr_t</span> <span class="n">dma_handle</span><span class="p">;</span> <span class="kt">void</span> <span class="o">*</span><span class="n">cpu_addr</span><span class="p">;</span> <span class="n">cpu_addr</span> <span class="o">=</span> <span class="n">dma_alloc_coherent</span><span class="p">(</span><span class="n">dev</span><span class="p">,</span> <span class="n">size</span><span class="p">,</span> <span class="o">&amp;</span><span class="n">dma_handle</span><span class="p">,</span> <span class="n">GFP_KERNEL</span><span class="p">);</span> <span class="k">if</span> <span class="p">(</span><span class="o">!</span><span class="n">cpu_addr</span><span class="p">)</span> <span class="k">return</span> <span class="o">-</span><span class="n">ENOMEM</span><span class="p">;</span> </code></pre> </div> <p>The call returns two things: a CPU virtual address you use to read and write the buffer, and a <code>dma_handle</code> of type <code>dma_addr_t</code> that you program into the device. When you are done, release both with the matching free call:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight c"><code><span class="n">dma_free_coherent</span><span class="p">(</span><span class="n">dev</span><span class="p">,</span> <span class="n">size</span><span class="p">,</span> <span class="n">cpu_addr</span><span class="p">,</span> <span class="n">dma_handle</span><span class="p">);</span> </code></pre> </div> <p>One subtlety that surprises people: coherent does not mean the CPU stops reordering writes. Think of it as asynchronous, outside the coherency domain. Network packets being transmitted or received, and filesystem buffers, are the standard examples. First, always check <code>dma_mapping_error()</code> on the returned address; mapping can fail when DMA address space is exhausted or an IOMMU mapping cannot be created, and using an unchecked address can lead to silent corruption. Second, never use the CPU buffer while it is mapped for the device. The buffer belongs to the device between map and unmap. The same applies to <code>dma_map_page()</code>, which takes a page and offset instead of a CPU pointer so it can map HIGHMEM memory, and to <code>dma_map_sg()</code> for scatter-gather lists.</p> <h2> Synchronising a buffer you reuse </h2> <p>Sometimes you need the CPU to look at a streaming buffer between transfers without fully unmapping it. That is what the sync calls are for. Before the CPU reads a buffer the device just wrote, give ownership back to the CPU; before handing it to the device again, return ownership to the device:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight c"><code><span class="n">dma_sync_single_for_cpu</span><span class="p">(</span><span class="n">dev</span><span class="p">,</span> <span class="n">dma_handle</span><span class="p">,</span> <span class="n">size</span><span class="p">,</span> <span class="n">DMA_FROM_DEVICE</span><span class="p">);</span> <span class="cm">/* CPU may now safely read the buffer */</span> <span class="n">dma_sync_single_for_device</span><span class="p">(</span><span class="n">dev</span><span class="p">,</span> <span class="n">dma_handle</span><span class="p">,</span> <span class="n">size</span><span class="p">,</span> <span class="n">DMA_FROM_DEVICE</span><span class="p">);</span> <span class="cm">/* device may now use the buffer again */</span> </code></pre> </div> <p>If you never touch the data between <code>dma_map_*()</code> and <code>dma_unmap_*()</code>, you do not need the sync calls at all. They exist precisely for the reuse case, and skipping them on a non-coherent platform is a frequent cause of intermittent corruption.</p> <h2> Alignment and cache lines </h2> <p>One rule deserves special attention on embedded targets. You may DMA to memory from <code>kmalloc()</code> or the page allocator, but not from <code>vmalloc()</code> memory, kernel stack, or static (data, text, bss) addresses. On a CPU with DMA-incoherent caches, a DMA buffer must also not share a cache line with other data, or a CPU write to one word and a DMA write to a neighbouring word in the same line can overwrite each other. Architectures set <code>ARCH_DMA_MINALIGN</code> so that <code>kmalloc()</code> buffers are aligned safely, but if you embed a DMA buffer inside a larger structure next to fields the CPU writes, you are responsible for keeping them on separate cache lines.</p> <h2> Inside the DMA mapping API: three back ends </h2> <p>Everything above is the contract your driver works to, and it is stable across kernel versions. The allocator <code>dma_alloc_attrs()</code> dispatches the same three ways, and on the direct path calls <code>dma_direct_alloc()</code> in <code>direct.c</code>. On a non-coherent architecture it allocates pages with <code>__dma_direct_alloc_pages()</code>, prepares them with <code>arch_dma_prep_coherent()</code> (on arm64 a cache clean over the region), and then, when <code>CONFIG_DMA_DIRECT_REMAP</code> is set, remaps them uncached with <code>dma_common_contiguous_remap()</code>; on architectures with <code>CONFIG_ARCH_HAS_DMA_SET_UNCACHED</code> it instead calls <code>arch_dma_set_uncached()</code>.</p> <p>So on a non-coherent SoC, the buffer returned by <code>dma_alloc_coherent()</code> is mapped uncached. That is how coherency is achieved without per-access cache maintenance, and it is also why coherent memory is the wrong choice for large data buffers: every CPU access bypasses the cache and is slow. This is the implementation reason behind the rule from part one to keep coherent memory for small control structures and use streaming maps for bulk data.</p> <h2> When the device cannot reach the buffer: swiotlb </h2> <p>Return to the <code>dma_capable()</code> check. When a device with a narrow DMA mask is handed a buffer above its reach, the direct path calls <code>swiotlb_map()</code>. The software I/O TLB keeps a low, device-addressable memory pool reserved at boot. <code>swiotlb_map()</code> copies (bounces) the buffer into that pool and returns a bus address the device can use. For a transfer toward the device, the bounce copy happens at map time; for a transfer from the device, it happens when you sync or unmap, which is one more reason the unmap and sync calls are mandatory rather than advisory. Bouncing is transparent to your driver, but it costs a copy, so a correct DMA mask is what lets the kernel skip it whenever the hardware can address the buffer directly.</p> <h2> Streaming sync, in the source </h2> <p>The sync calls from part one map onto <code>dma_direct_sync_single_for_cpu()</code> and <code>dma_direct_sync_single_for_device()</code>. the DMA mapping API converts a CPU buffer into a <code>dma_addr_t</code> the device can use, handling IOMMU and cache maintenance.</li> <li>Declare the device's addressing limits with <code>dma_set_mask_and_coherent()</code> before mapping; the mask feeds <code>dma_capable()</code> and decides whether the kernel must bounce through swiotlb.</li> <li>Use coherent mappings (<code>dma_alloc_coherent()</code>) for small long-lived control structures; they are uncached on non-coherent SoCs, so use streaming maps (<code>dma_map_single()</code>, <code>dma_map_page()</code>, <code>dma_map_sg()</code>) with the correct direction and a <code>dma_mapping_error()</code> check for bulk data.</li> <li>Call <code>dma_sync_*()</code> when the CPU touches a streaming buffer between transfers; on arm64 the direction selects the cache op, <code>dcache_clean_poc_nosync()</code> before the device reads and <code>dcache_inval_poc_nosync()</code> after it writes.</li> <li>The 7.x DMA core is physical-address based: with no custom ops and no IOMMU every map takes the dma-direct path (<code>dma_map_phys()</code> then <code>dma_direct_map_phys()</code>); build with <code>CONFIG_DMA_API_DEBUG</code> and read <code>/sys/kernel/debug/dma-api/</code> to catch misuse.</li> </ul> <h2> Frequently asked questions </h2> <p><strong>What is the difference between a coherent and a streaming DMA mapping?</strong><br /> A coherent mapping, made with dma_alloc_coherent(), is allocated once and kept for the device's lifetime with no explicit flushing needed; it suits small control structures like ring descriptors. A streaming mapping, made with dma_map_single() or similar, is set up just before one transfer and torn down after, and is the right choice for bulk data such as network packets.</p> <p><strong>Why does a DMA buffer that works on x86 get corrupted on an ARM target?</strong><br /> Many embedded SoCs have CPU caches that are not automatically kept coherent with DMA traffic, so a CPU write can sit in cache while the device reads stale main memory, or vice versa. On a coherent platform the DMA API's cache operations compile down to almost nothing, but on a non-coherent ARM board they become real cache flushes and invalidations, so skipping the required sync calls causes intermittent corruption there but not on coherent x86 hardware.</p> <p><strong>Do I need to call the dma_sync_* functions for every streaming mapping?</strong><br /> No. The sync calls are only needed if the CPU touches the buffer data between dma_map_<em>() and dma_unmap_</em>() while reusing the same mapping. If you never touch the data in that window, the map and unmap calls alone are sufficient.</p> <p><strong>What does CONFIG_DMA_API_DEBUG check for?</strong><br /> It shadows every DMA mapping your driver makes in a hash table and checks that unmaps match the corresponding maps, that the CPU does not touch memory currently owned by a device, and that drivers do not free memory with the wrong function.
