Managing i18n keys in React without losing your mind (without leaving your IDE)翻訳スキップ
原文説明
原文説明（英語）

<p><strong><code>t('common:user.profile.title')</code>… and now you just <em>hope</em> that key actually exists.</strong></p> <p>On any serious i18next project, you end up typing keys blind. Does the key exist? In which namespace? What's its value? Is it translated in <em>every</em> language, or just English? To find out, you open <code>en/common.json</code>, hit Ctrl+F, scroll, come back. Multiply by 200 keys and 5 locales and it's a daily tax on your focus.</p> <p>The real problem isn't i18next. It's that <strong>your keys and your translations live in files your editor treats as dead text.</strong> A string in the code, a JSON next door, and zero link between the two.</p> <p>I've been maintaining and heavily extending a JetBrains plugin (<strong>I18n Support Plus</strong>) to restore that link. The idea fits in one sentence: <strong>your i18n keys become first-class references in the IDE</strong>, exactly like code symbols. Here's concretely what that changes.</p> <h2> 1. See a key's status without opening a single file </h2> <p>The plugin colors each key by its resolution state, right in the code:</p> <ul> <li>key resolved in <strong>every</strong> locale,</li> <li>a segment that doesn't exist (<code>user.profil</code> instead of <code>user.profile</code>),</li> <li>namespace file not found,</li> <li>key pointing to an object instead of a value,</li> <li>key translated in <strong>some</strong> locales only (partial translation).</li> </ul> <p>A gutter badge (✅ / ⚠️ / ❌) gives the verdict at a glance. A red key is a bug caught <em>before</em> the commit, not in QA.</p> <h2> 2. Ctrl+Click → the translation. And back again. </h2> <p>Ctrl+Click on a key takes you <strong>straight to the value</strong> in the right JSON/YAML file, even if the key is only partially resolved (it jumps to the deepest resolved node).</p> <p>And it works both ways: from a translation entry, you find where it's used in the code. No more "is this key still used anywhere?".</p> <h2> 3. Autocomplete knows your keys </h2> <p>As you type <code>t('</code>, the plugin suggests the project's real keys, <strong>namespaces included</strong>. You no longer guess, you pick. Suggestions come from all your translation files, not a stale index.</p> <p>Both <code>useTranslation('common')</code> and <code>useTranslation(['common', 'auth'])</code> are handled: completion knows which namespace you're in. Even <strong>dynamic keys</strong> like <code>t(`common:role.${role}`)</code> are resolved through wildcards instead of being flagged as broken.</p> <h2> 4. Read the code with the actual sentences </h2> <p>Two ways to stop reading cryptic keys:</p> <ul> <li> <strong>Inlay hints</strong>: the translated value shows up in grey right after the key, without touching the file.</li> <li> <strong>Code folding</strong> (<code>Ctrl+Alt+Shift+T</code>): replaces the key with its translation inline. Your JSX reads like the real UI.</li> </ul> <p>And on hover (Ctrl+hover), a tooltip shows <strong>all locales side by side</strong>, missing ones marked <code>—</code>, with a direct link to the file.</p> <h2> 5. Extract a hardcoded string in one keystroke </h2> <p>Wrote <code>&lt;h1&gt;User profile&lt;/h1&gt;</code> hardcoded? <strong>Alt+Enter → Extract i18n key.</strong> The plugin:</p> <ul> <li>creates the key in your translation file(s),</li> <li>lets you pick/create the namespace on the fly,</li> <li>replaces the string with the matching <code>t()</code> call,</li> <li>can insert the key alphabetically (option).</li> </ul> <p>This is the gesture that turns "I'll deal with i18n later" into "it's already done".</p> <h2> 6. The translation dashboard </h2> <p>A dedicated tool window pulls it all together:</p> <ul> <li> <strong>Tree view</strong>: the key hierarchy, color-coded nodes (red = missing, orange = empty), double-click to edit.</li> <li> <strong>Table view</strong>: one column per locale + a <strong>usage</strong> counter. The <strong>Scan Orphans</strong> button lists keys no code uses anymore: right-click to delete them.</li> <li> <strong>Keys Synchronizer</strong>: a key exists in <code>en.json</code> but not <code>fr.json</code>? One click propagates it across all locales, with a dialog to fill in the missing values in bulk.</li> <li> <strong>Stats</strong>: completion rate per locale (total, translated, missing, %).</li> </ul> <p>Two built-in inspections keep the files clean: <strong>empty value</strong> (flag a key that was created but never translated) and <strong>duplicate value</strong> (spot copy-paste mistakes within a locale).</p> <p>This is where you go from "editing files" to <strong>driving the actual state of your localization</strong>.</p> <h2> 7. Refactor keys like real code </h2> <p>i18n refactoring becomes as safe as refactoring code:</p> <ul> <li> <strong>Rename</strong> (<code>Shift+F6</code>): renames a key across <strong>all</strong> translation files <strong>and</strong> every reference in the source at once.</li> <li> <strong>Move to another namespace</strong> (<code>Ctrl+Alt+Shift+M</code>): moves a key from one namespace to another atomically. It copies the value into the target file for <strong>every</strong> locale (creating it if missing, so no locale is dropped), deletes the old entry, and rewrites every code reference with the right <code>ns:</code> prefix, all in a single undo step.</li> <li> <strong>Sort alphabetically</strong>: reorders a JSON translation file (recursively), or auto-sorts on every key insertion.</li> </ul> <h2> It works with your stack </h2> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th></th> <th></th> </tr> </thead> <tbody> <tr> <td><strong>Frameworks</strong></td> <td>i18next / react-i18next, vue-i18n, lingui</td> </tr> <tr> <td><strong>Languages</strong></td> <td>JS, TS, JSX, TSX, Vue SFC, PHP</td> </tr> <tr> <td><strong>Formats</strong></td> <td>JSON / JSON5, YAML, PO/POT (gettext), i18next TS config</td> </tr> </tbody> </table></div> <p>On first launch, a <strong>setup wizard</strong> auto-detects your framework and scans your <code>locales/</code>, <code>i18n/</code>, <code>translations/</code> folders. Three steps and you're ready.</p> <h2> Wrapping up </h2> <p>The day-to-day pain of i18n isn't <em>translating</em>: it's that the IDE draws <strong>no link</strong> between your keys and your translations. The whole plugin follows from a single idea:</p> <blockquote> <p><strong>An i18n key should behave like a code symbol: you resolve it, navigate to it, complete it, rename it, and see its value.</strong></p> </blockquote> <p>If you touch i18next/vue-i18n/lingui in a JetBrains IDE, give it a try → <strong><a href="https://plugins.jetbrains.com/plugin/30971-i18n-support-plus" rel="noopener noreferrer">I18n Support Plus on the Marketplace</a></strong>. It's free, and the code is on <a href="https://github.com/IBRAHIMDANS/i18nSupportPlus" rel="noopener noreferrer">GitHub</a> (a ⭐ helps if it saves you time).</p> <p>The real question isn't <em>"which i18n library should I pick?"</em>. It's:</p> <p><em><strong>"How many times a day do I open a JSON file just to check whether a key exists?"</strong></em></p> <p>If the answer isn't <em>"never"</em>, this plugin is for you.</p> <p><em>I build tools that cut the daily friction of front-end work. Let's talk on <a href="https://linkedin.com/in/ibrahimadansoko" rel="noopener noreferrer">LinkedIn</a>.</em></p>

Why Your AI Agent Needs an Audit Trail (And How to Build One)翻訳スキップ

原文説明（英語）

<p>The rapid shift from passive conversational chatbots to active operational agents has changed the software engineering landscape. Modern systems powered by Large Language Models (LLMs) do not simply display answers. They actively query databases, execute API calls, manage background workflows, and make critical operational decisions.</p> <p>Because autonomous AI agents act non-deterministically, traditional debugging and application monitoring practices fail. When a standard application breaks, a developer can follow a deterministic trace through structured code pathways. But when an autonomous agent behaves unexpectedly, standard flat logs are insufficient. They only capture what happened, failing to capture the underlying reasoning or why the agent chose a specific tool.</p> <p>For enterprise environments, maintaining a complete audit trail is a necessity. Organizations must meet strict regulatory standards including the EU AI Act (Article 12 and 14), SOC 2 compliance, ISO 42001, and CCPA Automated Decision-Making Technology rules. Designing a compliant, secure, and privacy-preserving audit trail for autonomous agents requires a completely new logging architecture.</p> <p>This guide details the technical requirements for auditing autonomous AI agents and shows how the <a href="https://volidator.com" rel="noopener noreferrer">Volidator</a> architecture solves these challenges.</p> <h2> 1. The Architectural Failure of Flat Logs </h2> <p>Traditional application logging frameworks (such as Winston, Winston-Loki, or Datadog) write sequential, flat strings of plaintext. This model is built on the assumption that software follows linear execution paths.</p> <p>Autonomous agents operate in recursive loops. An agent execution begins with a user instruction, but that instruction initiates a cascade of sub-processes:</p> <ul> <li>The model evaluates the prompt.</li> <li>The model selects and executes a tool.</li> <li>The model ingests the tool output, updates its internal state, and generates another reasoning path.</li> <li>The model triggers sub-agents to execute secondary tasks.</li> <li>The model evaluates safety guardrails before returning the final response.</li> </ul> <p>If you write these events as flat logs, the relationship between them is lost. An auditor looking at a database edit log cannot link it back to the specific LLM reasoning step or tool execution that triggered it. To audit an AI agent, you must trace the complete causality chain.</p> <h2> Tracing Causality with Hierarchical Spans </h2> <p>To capture the execution tree, logs must use parent-child relationship tokens. Every agent event must carry a trace identifier (traceId), a unique span identifier (spanId), and a parent span identifier (parentSpanId).</p> <p>The Volidator SDK constructs these relationships natively, building interactive execution trees that allow compliance auditors to drill down from a high-level user request to the exact prompt, model confidence score, and raw tool output that caused a specific state change.</p> <h2> Resolving Edge Clock Drift with Lamport Clocks </h2> <p>In distributed, edge-computed agent architectures, logs are ingested asynchronously from various serverless environments. Relying on system clocks (NTP) to order these events is dangerous. Minor clock drift between edge nodes can cause tool responses to be logged with a timestamp prior to the tool call, ruining the audit trail.</p> <p>Volidator resolves this by implementing Lamport Logical Clocks. The SDK propagates a logical clock header (x-volidator-clock) across all asynchronous context boundaries. When an event is logged, its clock value is synchronized using a deterministic formula:</p> <p>localClock = max(localClock, incomingClock) + 1</p> <p>This ensures that the sequential ordering of LLM thinking steps, tool invocations, and database responses remains chronologically consistent, regardless of NTP variations at the edge.</p> <h2> 2. Zero-Knowledge Cryptographic Privacy in Agent Logging </h2> <p><strong>Auditing AI agents presents a major compliance paradox:</strong></p> <p>Compliance standards require logs to contain the exact prompts, tools, and outputs to prove safe execution.</p> <p>Privacy regulations (such as GDPR, HIPAA, and CCPA) prohibit storing unencrypted customer data, PII, or PHI on third-party telemetry servers.<br /> Standard monitoring services store telemetry in plaintext. If an agent ingests a customer prompt containing medical history or financial data, that data is transmitted and stored in plaintext, exposing the organization to major liabilities.</p> <h2> Local Client-Side Encryption </h2> <p>Volidator solves this conflict using a zero-knowledge architecture. The SDK performs local, symmetric encryption inside your application runtime before transmitting any telemetry.</p> <p>Using AES-256-GCM, the SDK encrypts the sensitive fields of the log payload (such as prompts, thoughts, and tool inputs) using a private encryption key stored in your environment. The decryption key never leaves your server. The Volidator database ingestion worker receives only randomized ciphertext, rendering the logging service blind to your proprietary prompts and customer PII.</p> <h2> Non-Reversible Blind Indexing </h2> <p>To allow search and filter functionality on the dashboard without exposing raw data, Volidator generates non-reversible HMAC-SHA-256 blind indexes locally.</p> <p>When you query logs for a specific customer identifier or actor, the SDK hashes the search term using a private key and sends the hash. The dashboard matches this against the blind index stored in the database, enabling quick filtering of encrypted logs without the database ever seeing the unencrypted search criteria.</p> <h2> The Claim Check Pattern for Large Metadata </h2> <p>Compliance logs for LLM agents are heavy. A single prompt loop containing retrieved context blocks (RAG payloads) or sandboxed execution outputs can easily exceed 30KB. Storing large payloads directly inside the primary database creates ingestion bottlenecks.</p> <p>To handle this, the Volidator SDK executes the Claim Check Pattern. When the serialized, encrypted ciphertext of a log exceeds 30KB, the SDK streams the payload to a secure object storage proxy (built on Cloudflare R2). It then writes a content-addressed SHA-256 hash pointer to the database row with the isClaimCheck parameter enabled.</p> <p>During an audit, when a developer opens the Volidator dashboard, the browser client pulls the encrypted chunk using the hash pointer and decries it locally in-browser using the private key residing in the URL hash fragment. This keeps the database fast while preserving absolute zero-knowledge privacy.</p> <h2> 3. Cryptographic Actor Attribution and WebAuthn Attestation </h2> <p>One of the most complex challenges in auditing autonomous systems is determining accountability. If an agent executes a high-risk operation, was the action triggered by an automated bot or a human supervisor?</p> <h2> Key Prefix Taxonomy </h2> <p>To enforce strict accountability at the ingestion layer, Volidator categorizes API keys using prefixes:</p> <ul> <li>val_human_ : Assigned to interactive human operators.</li> <li>val_agent_ : Assigned to autonomous models, scripts, or LLM agents.</li> <li>val_service_ : Assigned to static background cron processes or system daemons.</li> </ul> <p>These prefixes are checked instantaneously at Layer 0 in the ingestion worker.</p> <h2> Automatic Credential Handover Detection </h2> <p>If an API key belonging to a human caller is used within an autonomous context propagation, or if it executes requests carrying agent rationales, Volidator flags this automatically. The worker sets the credentialHandoverDetected value to true and updates the actor type to agent. This prevents automated scripts from using human credentials to bypass compliance tracking.</p> <h2> Biometric Action Attestation </h2> <p>For high-risk operations (such as manual overrides or large financial payouts), compliance frameworks like the EU AI Act (Article 14) demand human verification. Volidator implements cryptographic attestation using WebAuthn:</p> <ul> <li> <strong>Challenge:</strong> The SDK generates a rolling 15-minute challenge matching the specific action payload.</li> <li> <strong>Canonicalization:</strong> To ensure payload consistency, the client and server serialize the JSON payload alphabetically using a zero-dependency canonical serializer, preventing whitespace or key order differences from invalidating the signature.</li> <li> <strong>Biometric Assertion:</strong> The SDK invokes the browser WebAuthn API (navigator.credentials.get) to sign the composite challenge (sha256(challenge + colon + payloadHash)) inside the user's hardware secure enclave (TouchID, FaceID, or YubiKey).</li> <li> <strong>Edge Verification:</strong> The ingestion worker fetches the user's public key from the database, validates the signature, and invalidates the challenge atomically using a database query to prevent replay attacks.</li> </ul> <h2> 4. This loose coupling is one of the biggest strengths of event-driven systems, but it is also one of their biggest challenges.</p> <p>The producer no longer knows who depends on the event, which makes changing the event significantly more complicated than changing an internal Java object.</p> <ul> <li><strong>Events Are Public APIs</strong></li> </ul> <p>Most teams treat REST APIs very carefully. Before removing a field, they consider existing clients, API versions, backward compatibility, and migration plans. Events deserve exactly the same level of discipline.</p> <p>Once an event is published, it becomes part of the public interface of the service. Removing a field from an event can break downstream systems just as easily as removing a field from a REST response. The difference is that consumers are often invisible. A REST API usually has documented clients, while an event may have consumers owned by completely different teams, some of which may not even exist when the producer is originally developed.</p> <p>Thinking of events as contracts fundamentally changes how they should be designed.</p> <p><strong>2. Many event-driven projects begin with something like this:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight java"><code><span class="kd">public</span> <span class="kd">class</span> <span class="nc">Order</span> <span class="o">{</span> <span class="kd">private</span> <span class="nc">Long</span> <span class="n">id</span><span class="o">;</span> <span class="kd">private</span> <span class="nc">Customer</span> <span class="n">customer</span><span class="o">;</span> <span class="kd">private</span> <span class="nc">List</span><span class="o">&lt;</span><span class="nc">OrderItem</span><span class="o">&gt;</span> <span class="n">items</span><span class="o">;</span> <span class="kd">private</span> <span class="nc">Address</span> <span class="n">shippingAddress</span><span class="o">;</span> <span class="o">}</span> </code></pre> </div> <p>The easiest approach is to serialize the entire object and publish it.</p> <p><code>eventPublisher.publish(order);</code></p> <p>It works—until the domain model changes. A field gets renamed, an object is restructured, or a new relationship is introduced. Every consumer now receives a different payload. The producer evolved, but the contract changed accidentally.</p> <p>This is one of the most common mistakes in event-driven systems. Events should not expose internal domain models; they should communicate business facts. Those are two very different things.</p> <ul> <li><strong>An Event Describes Something That Already Happened</strong></li> </ul> <p>A useful mental model is that a command asks for something to happen, while an event states that something already happened. For example:</p> <p><code>OrderConfirmed</code></p> <p>This is a business fact. It cannot be rejected because it has already occurred. Similarly:</p> <p><code>PaymentCompleted<br /> InventoryReserved<br /> ShipmentCreated<br /> CustomerRegistered</code></p> <p>All describe completed business actions. </p> <p>Consumers should be able to trust that these events represent facts. This makes event names extremely important. Good names communicate completed business outcomes, while poor names often expose implementation details. Compare the following examples:</p> <p>Good: <code>OrderConfirmed</code><br /> Poor: <code>UpdateOrderStatus</code></p> <p>The first describes something that happened, while the second sounds like an <em>internal method call</em>. This distinction becomes increasingly important as systems grow.</p> <p><strong>3. Events Are Immutable</strong></p> <p>One characteristic separates events from many other forms of communication: events are immutable. Once published, an event represents history, and history cannot be rewritten.</p> <p>Imagine the following event:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight json"><code><span class="p">{</span><span class="w"> </span><span class="nl">"orderId"</span><span class="p">:</span><span class="w"> </span><span class="s2">"ORD-1001"</span><span class="p">,</span><span class="w"> </span><span class="nl">"status"</span><span class="p">:</span><span class="w"> </span><span class="s2">"CONFIRMED"</span><span class="w"> </span><span class="p">}</span><span class="w"> </span></code></pre> </div> <p>Tomorrow, the customer cancels the order. The producer should not update the previous event. Instead, it publishes a new one:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight json"><code><span class="p">{</span><span class="w"> </span><span class="nl">"orderId"</span><span class="p">:</span><span class="w"> </span><span class="s2">"ORD-1001"</span><span class="p">,</span><span class="w"> </span><span class="nl">"status"</span><span class="p">:</span><span class="w"> </span><span class="s2">"CANCELLED"</span><span class="w"> </span><span class="p">}</span><span class="w"> </span></code></pre> </div> <p>The event stream now tells a complete story:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>OrderCreated | OrderConfirmed | OrderCancelled </code></pre> </div> <p>Consumers joining later can reconstruct what happened. This is one of the reasons event-driven systems are valuable—events become an immutable business history. Changing previously published events destroys that history.</p> <ul> <li><strong>Events Represent Facts, Not State</strong></li> </ul> <p>Another common misunderstanding is treating events as snapshots of current state. Consider this event:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight json"><code><span class="p">{</span><span class="w"> </span><span class="nl">"orderId"</span><span class="p">:</span><span class="w"> </span><span class="s2">"ORD-1001"</span><span class="p">,</span><span class="w"> </span><span class="nl">"status"</span><span class="p">:</span><span class="w"> </span><span class="s2">"PROCESSING"</span><span class="w"> </span><span class="p">}</span><span class="w"> </span></code></pre> </div> <p><em>Does it mean the order is currently processing, or that the order entered processing</em>? These are different meanings.</p> <p>A better event would be: <code>OrderProcessingStarted</code></p> <p>The name clearly communicates a business fact. Consumers no longer need to interpret the payload because the event itself explains what happened. As event catalogs grow, this principle becomes increasingly important. Well-designed event names reduce ambiguity, while poorly named events force consumers to infer business meaning.</p> <p><strong>4. Events and Commands Are Not the Same</strong></p> <p>This is one of the most misunderstood topics in event-driven architecture. Many teams use commands and events interchangeably, which creates tightly coupled systems. Understanding the difference changes how services communicate.</p> <ul> <li><strong>Commands Express Intent</strong></li> </ul> <p>A command represents a request where the sender expects another service to perform an action. </p> <p>For example:<br /> <code>ReserveInventory</code></p> <p>The sender is asking, “Please reserve inventory.” The receiving service can accept it, reject it, validate it, or return an error. </p> <p>Commands imply responsibility—someone is expected to do something.</p> <ul> <li><strong>Events Express Facts</strong></li> </ul> <p>An event communicates that something has already happened. </p> <p>For example:<br /> <code>InventoryReserved</code></p> <p>The reservation has already completed. Consumers cannot reject it; they simply react. This difference may appear subtle, but architecturally it is enormous. Commands influence behavior, while events communicate history.</p> <ul> <li><strong>Visualizing the Difference</strong></li> </ul> <p>Command<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Order Service | Reserve Inventory | v Inventory Service </code></pre> </div> <p>The Order Service knows exactly who should process the request, making the communication directed.</p> <ul> <li> <strong>Event</strong> </li> </ul> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>InventoryReserved | +----+----+---------+ | | | v v v Billing Shipping Analytics </code></pre> </div> <p>The Inventory Service simply publishes a business fact. It does not know who reacts, and it does not need to. That is the essence of loose coupling.</p> <p><strong>A Practical Rule of Thumb</strong></p> <p>One guideline that helps during system design reviews is simple: use a <strong>command</strong> when <em>one specific service is responsible for performing an action</em>, and use an <strong>event</strong> when <em>informing any interested service that a business fact has already occurred</em>.</p> <p>This distinction keeps responsibilities clear and prevents event-driven systems from gradually turning into distributed RPC systems disguised as messaging.</p> <p>In this part, we've explored why event-driven systems fail in production and events vs commands. </p> <p>In the next part, we will build on these foundations by designing event schemas that survive years of system evolution.</p> <p>Assisted AI to paraphrase. </p>
