<td>✅</td> </tr> <tr> <td>Retrieved RAG Results</td> <td>✅</td> </tr> <tr> <td>Tool Outputs</td> <td>✅</td> </tr> <tr> <td>Function Call Results</td> <td>✅</td> </tr> <tr> <td>AI Response</td> <td>✅</td> </tr> </tbody> </table></div> <p>Even if you only typed:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Summarize this. </code></pre> </div> <p>The model might actually receive thousands of tokens.</p> <p>Imagine asking an AI coding assistant:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Why is this function failing? </code></pre> </div> <p>Behind the scenes, the application might automatically include:</p> <ul> <li>System instructions</li> <li>Current source file</li> <li>Nearby functions</li> <li>Package imports</li> <li>Build configuration</li> <li>Previous conversation</li> <li>Compiler errors</li> </ul> <p>Your four-word prompt becomes a request containing thousands of tokens.</p> <p>This is another example of <strong>Context Engineering</strong> in action.</p> <h1> 🔄 Every New Request Starts Again </h1> <p>This surprises many engineers.</p> <p>Suppose your conversation looks like this:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight go"><code><span class="n">User</span><span class="o">:</span> <span class="n">Explain</span> <span class="n">JWT</span> <span class="n">Authentication</span><span class="o">.</span> <span class="n">Assistant</span><span class="o">:</span> <span class="o">...</span> <span </code></pre> </div> <p>Many people imagine the model simply remembers everything internally.</p> <p>That's not how most stateless LLM API interactions work.</p> <p>For each new request, the application typically sends the conversation history (or a selected portion of it) again so the model has the necessary context to continue the conversation coherently.</p> <p>Conceptually, it looks something like this:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Request 1 System Prompt User Message ↓ LLM ------------------------ Request 2 System <strong>Context Window?</strong></p> <p>A context window is the maximum amount of information (measured in tokens) that a model can consider at one time while generating a response.</p> <p>Think of it like a whiteboard.</p> <p>Imagine you're solving a difficult problem with a teammate.</p> <p>Everything written on the whiteboard is available to both of you.</p> <p>You can read it.</p> <p>Reference notes.</li> <li>Replace them with a summary.</li> </ul> <p>Large Language Models work in a very similar way.</p> <p>The context window is that whiteboard.</p> <p>Everything currently inside it is visible to the model while it generates its next response.</p> <p>Everything outside it is invisible unless your application decides to bring it back.</p> <h1> 📖 What's Actually Inside the Context Window? </h1> <p>One common misconception is that the context window contains only your prompt.</p> <p>In reality, modern AI applications often build a much richer context before sending the request.</p> <p>A simplified request might look like this:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>System Instructions ↓ Conversation History ↓ Retrieved Knowledge (RAG) ↓ Uploaded Files ↓ Tool Results ↓ Current User Prompt ↓ LLM </code></pre> </div> <p>Notice something interesting.</p> <p>Your latest message may only be a tiny fraction of everything the model receives.</p> <p>The application has already assembled the rest of the context behind the scenes.</p> <p>This is why two AI tools using the same underlying model can produce very different answers.</p> <p>The difference often isn't the model itself.</p> <p>It's <strong>how effectively each application constructs the context before making the request.</strong></p> <h1> 🤔 Does a Bigger Context Window Always Mean Better AI? </h1> <p>At first glance, it sounds obvious.</p> <p>If an AI model can process more information, shouldn't the answers always improve?</p> <p>Not necessarily.</p> <p>Imagine you're preparing for an interview.</p> <p>Someone gives you:</p> <ul> <li>Your resume</li> <li>The job description</li> <li>Company documentation</li> <li>Engineering handbook</li> <li>API documentation</li> <li>Meeting notes</li> <li>Internal wiki</li> <li>Product roadmap</li> <li>Last year's planning documents</li> <li>Random Slack conversations</li> </ul> <p>Could you answer the interview questions?</p> <p>Probably.</p> <p>Would all that information actually help?</p> <p>Probably not.</p> <p>In fact, it might make things worse because you'd spend time filtering out irrelevant information before focusing on what matters.</p> <p>Large Language Models face a similar challenge.</p> <p>A larger context window allows the model to <strong>see more information</strong>, but it doesn't mean every piece of that information is equally useful.</p> <p>Good Context Engineering isn't about maximizing context.</p> <p>It's about <strong>optimizing context</strong>.</p> <h1> 🎯 More Context ≠ Better Context </h1> <p>One of the biggest mistakes in AI applications is thinking:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>If some context helps, more context must help even more. </code></pre> </div> <p>That's rarely true.</p> <p>Imagine building an AI coding assistant.</p> <p>A developer asks:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Why is this API returning 500? </code></pre> </div> <p>Option A:</p> <p>Provide only:</p> <ul> <li>The API handler</li> <li>Relevant service</li> <li>Error log</li> </ul> <p>Option B:</p> <p>Provide:</p> <ul> <li>Entire repository</li> <li>All dependencies</li> <li>Every README</li> <li>CI/CD configuration</li> <li>Kubernetes manifests</li> <li>Previous conversations</li> <li>All database schemas</li> </ul> <p>Technically, the model has more information.</p> <p>Practically, you've introduced noise.</p> <p>Finding the relevant signal becomes harder.</p> <p>This is why mature AI systems spend significant effort deciding <strong>what not to include</strong>.</p> <p>Context Engineering is often about removing information rather than adding it.</p> <h1> 🔍 The "Lost in the Middle" Problem </h1> <p>Researchers have observed an interesting behavior in long-context language models.</p> <p>When a prompt becomes very large, information buried in the middle of the context may receive less attention than information closer to the beginning or end.</p> <p>This phenomenon is commonly referred to as the <strong>"Lost in the Middle"</strong> effect.</p> <p>Imagine reading a 300-page document.</p> <p>You usually remember:</p> <ul> <li>The introduction</li> <li>The conclusion</li> </ul> <p>But details from page 147?</p> <p>Much harder.</p> <p>Language models can exhibit similar behavior under certain conditions.</p> <p>This doesn't mean long-context models are ineffective.</p> <p>It means that <strong>how information is organized inside the context matters</strong>, not just how much information is included.</p> <h1> 🧠 Context Is NOT Memory </h1> <p>This is probably the biggest misconception about modern AI.</p> <p>People often say:</p> <blockquote> <p>"ChatGPT remembered what I told it last week."</p> </blockquote> <p>What actually happened depends on the application.</p> <p>It's important to distinguish between <strong>context</strong> and <strong>memory</strong>.</p> <h2> Context </h2> <p>Context is the information available <strong>only for the current request</strong>.</p> <p>Once the request is complete, the model itself doesn't retain that context for future API calls.</p> <p>Think of it as a whiteboard used during a meeting.</p> <p>Erase the whiteboard and the discussion is gone.</p> <h2> Memory </h2> <p>Memory is information that an application intentionally stores and later reintroduces into future conversations.</p> <p>For example:</p> <p>You tell an AI assistant:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>I primarily write Go and React. </code></pre> </div> <p>The application might choose to save that preference.</p> <p>Next week, when you ask:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Generate an API example. </code></pre> </div> <p>The application retrieves your stored preference and injects it into the new request.</p> <p>The model appears to "remember."</p> <p>In reality, the application supplied that information again.</p> <p>The memory lives in the application—not inside the language model itself.</p> <h1> 🏗️ A Simple Mental Model </h1> <p>Think of an AI application as three separate components:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code> User │ ▼ ┌─────────────────────┐ │ Memory Store │ └─────────────────────┘ │ ▼ </code></pre> </div> <p>The LLM only sees the final assembled context.</p> <p>It doesn't know where each piece of information came from.</p> <h1> 💰 Why Tokens Matter for Cost and Performance </h1> <p>We've already learned that everything consumes tokens.</p> <p>But token usage affects much more than pricing.</p> <p>It also impacts:</p> <ul> <li>Request latency</li> <li>Throughput</li> <li>Infrastructure cost</li> <li>User experience</li> </ul> <p>Consider these two requests.</p> <h3> Efficient Request </h3> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>System Prompt Relevant API documentation Current User Question </code></pre> </div> <h3> Inefficient Request </h3> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>System Prompt Entire conversation Complete codebase Multiple PDFs Large JSON payloads Old logs Current User Question </code></pre> </div> <p>The second request requires the model to process significantly more information before generating a response.</p> <p>Even if the answer only depends on a small portion of that data.</p> <p>More tokens generally mean:</p> <ul> <li>More computation</li> <li>Higher latency</li> <li>Higher cost</li> </ul> <p>This is why production AI systems invest heavily in efficient context management.</p> <h1> ⚙️ How Production AI Systems Manage Context </h1> <p>Well-designed AI applications rarely send everything they know to the model.</p> <p>Instead, they carefully construct the context.</p> <p>Some common strategies include:</p> <h3> ✅ Retrieve Only What's Relevant </h3> <p>Instead of sending an entire knowledge base,</p> <p>retrieve only the documents related to the user's question.</p> <h3> ✅ Summarize Long Conversations </h3> <p>Rather than sending hundreds of previous messages,</p> <p>summarize earlier parts while preserving important facts.</p> <p>For example:</p> <p>Instead of:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>150 previous messages </code></pre> </div> <p>Send:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Conversation Summary: • User uses Go • Building REST APIs • Uses PostgreSQL • Currently debugging authentication </code></pre> </div> <p>Far fewer tokens.</p> <p>Much easier for the model.</p> <h3> ✅ Keep System Prompts Focused </h3> <p>System instructions should define behavior,</p> <p>not contain an entire company handbook.</p> <h3> ✅ Separate Long-Term Memory from Current Context </h3> <p>Store reusable preferences separately.</p> <p>Retrieve them only when needed.</p> <h3> ✅ Measure Token Usage </h3> <p>Don't optimize blindly.</p> <p>Most AI SDKs expose token usage statistics.</p> <p>Monitoring them helps identify unnecessary context and control costs.</p> <h1> 🚫 Common Context Engineering Mistakes </h1> <p>Here are some patterns that often reduce AI quality rather than improve it.</p> <h3> ❌ Dumping entire documents into every request </h3> <p>More information isn't automatically better.</p> <h3> ❌ Sending the full conversation forever </h3> <p>Long conversations should be summarized or trimmed when appropriate.</p> <h3> ❌ Treating memory as conversation history </h3> <p>They're different concepts.</p> <h3> ❌ Ignoring token usage </h3> <p>Unexpected token growth often leads to higher costs and slower responses.</p> <h3> ❌ Assuming bigger context windows solve everything </h3> <p>Context quality matters far more than context quantity.</p> <h1> 🎯 Best Practices </h1> <p>If you're building AI-powered applications, keep these principles in mind:</p> <p>✅ Send only the information required for the current task.</p> <p>✅ Prefer relevant context over larger context.</p> <p>✅ Separate memory from conversation history.</p> <p>✅ Summarize older interactions instead of continuously expanding prompts.</p> <p>✅ Monitor token usage in production.</p> <p>✅ Design context deliberately instead of simply appending more text.</p> <h1> 🏁 Final Thoughts </h1> <p>When people first start working with AI, they often focus on prompts.</p> <p>Then they discover Context Engineering.</p> <p>Eventually, they realize another important truth:</p> <p><strong>Not all context is equally valuable.</strong></p> <p>Modern AI systems succeed because they carefully decide:</p> <ul> <li>What to include</li> <li>What to exclude</li> <li>What to summarize</li> <li>What to retrieve</li> <li>What to remember</li> </ul> <p>That decision-making process is often more important than the prompt itself.</p> <p>The best AI applications aren't the ones with the biggest context windows.</p> <p>They're the ones that provide <strong>the right information at the right time</strong>.</p> <p>And that's exactly what Context Engineering is about.</p> <h1> 📌 Key Takeaways </h1> <ul> <li>Tokens are the fundamental units processed by language models—they are not the same as words.</li> <li>Every request consumes tokens, including system prompts, conversation history, retrieved documents, tool outputs and the model's response.</li> <li>A context window defines how much information a model can consider at once.</li> <li>Larger context windows are useful, but more context doesn't automatically lead to better answers.</li> <li>Context and memory are different: context exists for the current request, while memory is information intentionally stored and reintroduced by the application.</li> <li>Effective Context Engineering is about sending <strong>relevant</strong> information, not <strong>all</strong> available information.</li> <li>Good context management improves response quality, reduces costs and lowers latency.</li> </ul> <h1> 🚀 Coming Up in Part 3 </h1> <p>So far, we've answered:</p> <ul> <li>What is Context Engineering?</li> <li>What are tokens?</li> <li>How do context windows work?</li> <li>Why isn't bigger context always better?</li> <li>What's the difference between context and memory?</li> </ul> <p>But another important question remains:</p> <blockquote> <p><strong>Where does all this context actually come from?</strong></p> </blockquote> <p>How do AI applications automatically search documentation, query databases, call APIs, execute tools and decide what information should be sent to the model?</p> <p>That's where concepts like <strong>Retrieval-Augmented Generation (RAG)</strong>, <strong>Tool Calling</strong>, <strong>Model Context Protocol (MCP)</strong> and <strong>AI Agents</strong> come into the picture.</p> <p>We'll break each of them down in <strong>Part 3</strong> with practical examples and real-world engineering scenarios.</p>

class="n">Can</span> <span class="n">you</span> <span class="nb">make</span> <span class="n">it</span> <span class="n">use</span> <span class="n">middleware</span><span class="err">?</span> </code></pre> </div> <p>Many people imagine the model simply remembers everything internally.</p> <p>That's not how most stateless LLM API interactions work.</p> <p>For each new request, the application typically sends the conversation history (or a selected portion of it) again so the model has the necessary context to continue the conversation coherently.</p> <p>Conceptually, it looks something like this:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Request 1 System Prompt User Message ↓ LLM ------------------------ Request 2 System Prompt Conversation History New User Message ↓ LLM ------------------------ Request 3 System Prompt Conversation History Latest User Message ↓ LLM </code></pre> </div> <p>As the conversation grows longer, more tokens are often included with each request unless the application trims, summarizes or otherwise manages the context.</p> <p>This is one reason token usage can increase significantly during long conversations.</p> <h1> 🪟 What Is a Context Window? </h1> <p>Now we can finally answer the question from the beginning.</p> <p>If tokens are the building blocks…</p> <p>Then what is the <strong>Context Window?</strong></p> <p>A context window is the maximum amount of information (measured in tokens) that a model can consider at one time while generating a response.</p> <p>Think of it like a whiteboard.</p> <p>Imagine you're solving a difficult problem with a teammate.</p> <p>Everything written on the whiteboard is available to both of you.</p> <p>You can read it.</p> <p>Reference it.</p> <p>Connect ideas.</p> <p>Reason about it.</p> <p>But the whiteboard isn't infinite.</p> <p>Eventually, it fills up.</p> <p>To continue working, you have two choices:</p> <ul> <li>Erase older notes.</li> <li>Replace them with a summary.</li> </ul> <p>Large Language Models work in a very similar way.</p> <p>The context window is that whiteboard.</p> <p>Everything currently inside it is visible to the model while it generates its next response.</p> <p>Everything outside it is invisible unless your application decides to bring it back.</p> <h1> 📖 What's Actually Inside the Context Window? </h1> <p>One common misconception is that the context window contains only your prompt.</p> <p>In reality, modern AI applications often build a much richer context before sending the request.</p> <p>A simplified request might look like this:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>System Instructions ↓ Conversation History ↓ Retrieved Knowledge (RAG) ↓ Uploaded Files ↓ Tool Results ↓ Current User Prompt ↓ LLM </code></pre> </div> <p>Notice something interesting.</p> <p>Your latest message may only be a tiny fraction of everything the model receives.</p> <p>The application has already assembled the rest of the context behind the scenes.</p> <p>This is why two AI tools using the same underlying model can produce very different answers.</p> <p>The difference often isn't the model itself.</p> <p>It's <strong>how effectively each application constructs the context before making the request.</strong></p> <h1> 🤔 Does a Bigger Context Window Always Mean Better AI? </h1> <p>At first glance, it sounds obvious.</p> <p>If an AI model can process more information, shouldn't the answers always improve?</p> <p>Not necessarily.</p> <p>Imagine you're preparing for an interview.</p> <p>Someone gives you:</p> <ul> <li>Your resume</li> <li>The job description</li> <li>Company documentation</li> <li>Engineering handbook</li> <li>API documentation</li> <li>Meeting notes</li> <li>Internal wiki</li> <li>Product roadmap</li> <li>Last year's planning documents</li> <li>Random Slack conversations</li> </ul> <p>Could you answer the interview questions?</p> <p>Probably.</p> <p>Would all that information actually help?</p> <p>Probably not.</p> <p>In fact, it might make things worse because you'd spend time filtering out irrelevant information before focusing on what matters.</p> <p>Large Language Models face a similar challenge.</p> <p>A larger context window allows the model to <strong>see more information</strong>, but it doesn't mean every piece of that information is equally useful.</p> <p>Good Context Engineering isn't about maximizing context.</p> <p>It's about <strong>optimizing context</strong>.</p> <h1> 🎯 More Context ≠ Better Context </h1> <p>One of the biggest mistakes in AI applications is thinking:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>If some context helps, more context must help even more. </code></pre> </div> <p>That's rarely true.</p> <p>Imagine building an AI coding assistant.</p> <p>A developer asks:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Why is this API returning 500? </code></pre> </div> <p>Option A:</p> <p>Provide only:</p> <ul> <li>The API handler</li> <li>Relevant service</li> <li>Error log</li> </ul> <p>Option B:</p> <p>Provide:</p> <ul> <li>Entire repository</li> <li>All dependencies</li> <li>Every README</li> <li>CI/CD configuration</li> <li>Kubernetes manifests</li> <li>Previous conversations</li> <li>All database schemas</li> </ul> <p>Technically, the model has more information.</p> <p>Practically, you've introduced noise.</p> <p>Finding the relevant signal becomes harder.</p> <p>This is why mature AI systems spend significant effort deciding <strong>what not to include</strong>.</p> <p>Context Engineering is often about removing information rather than adding it.</p> <h1> 🔍 The "Lost in the Middle" Problem </h1> <p>Researchers have observed an interesting behavior in long-context language models.</p> <p>When a prompt becomes very large, information buried in the middle of the context may receive less attention than information closer to the beginning or end.</p> <p>This phenomenon is commonly referred to as the <strong>"Lost in the Middle"</strong> effect.</p> <p>Imagine reading a 300-page document.</p> <p>You usually remember:</p> <ul> <li>The introduction</li> <li>The conclusion</li> </ul> <p>But details from page 147?</p> <p>Much harder.</p> <p>Language models can exhibit similar behavior under certain conditions.</p> <p>This doesn't mean long-context models are ineffective.</p> <p>It means that <strong>how information is organized inside the context matters</strong>, not just how much information is included.</p> <h1> 🧠 Context Is NOT Memory </h1> <p>This is probably the biggest misconception about modern AI.</p> <p>People often say:</p> <blockquote> <p>"ChatGPT remembered what I told it last week."</p> </blockquote> <p>What actually happened depends on the application.</p> <p>It's important to distinguish between <strong>context</strong> and <strong>memory</strong>.</p> <h2> Context </h2> <p>Context is the information available <strong>only for the current request</strong>.</p> <p>Once the request is complete, the model itself doesn't retain that context for future API calls.</p> <p>Think of it as a whiteboard used during a meeting.</p> <p>Erase the whiteboard and the discussion is gone.</p> <h2> Memory </h2> <p>Memory is information that an application intentionally stores and later reintroduces into future conversations.</p> <p>For example:</p> <p>You tell an AI assistant:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>I primarily write Go and React. </code></pre> </div> <p>The application might choose to save that preference.</p> <p>Next week, when you ask:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Generate an API example. </code></pre> </div> <p>The application retrieves your stored preference and injects it into the new request.</p> <p>The model appears to "remember."</p> <p>In reality, the application supplied that information again.</p> <p>The memory lives in the application—not inside the language model itself.</p> <h1> 🏗️ A Simple Mental Model </h1> <p>Think of an AI application as three separate components:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code> User │ ▼ ┌─────────────────────┐ │ Memory Store │ └─────────────────────┘ │ ▼ ┌─────────────────────┐ │ Context Builder │ │ - System Prompt │ │ - History │ │ - RAG Results │ │ - Tool Outputs │ └─────────────────────┘ │ ▼ ┌─────────────────────┐ │ LLM │ └─────────────────────┘ </code></pre> </div> <p>The LLM only sees the final assembled context.</p> <p>It doesn't know where each piece of information came from.</p> <h1> 💰 Why Tokens Matter for Cost and Performance </h1> <p>We've already learned that everything consumes tokens.</p> <p>But token usage affects much more than pricing.</p> <p>It also impacts:</p> <ul> <li>Request latency</li> <li>Throughput</li> <li>Infrastructure cost</li> <li>User experience</li> </ul> <p>Consider these two requests.</p> <h3> Efficient Request </h3> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>System Prompt Relevant API documentation Current User Question </code></pre> </div> <h3> Inefficient Request </h3> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>System Prompt Entire conversation Complete codebase Multiple PDFs Large JSON payloads Old logs Current User Question </code></pre> </div> <p>The second request requires the model to process significantly more information before generating a response.</p> <p>Even if the answer only depends on a small portion of that data.</p> <p>More tokens generally mean:</p> <ul> <li>More computation</li> <li>Higher latency</li> <li>Higher cost</li> </ul> <p>This is why production AI systems invest heavily in efficient context management.</p> <h1> ⚙️ How Production AI Systems Manage Context </h1> <p>Well-designed AI applications rarely send everything they know to the model.</p> <p>Instead, they carefully construct the context.</p> <p>Some common strategies include:</p> <h3> ✅ Retrieve Only What's Relevant </h3> <p>Instead of sending an entire knowledge base,</p> <p>retrieve only the documents related to the user's question.</p> <h3> ✅ Summarize Long Conversations </h3> <p>Rather than sending hundreds of previous messages,</p> <p>summarize earlier parts while preserving important facts.</p> <p>For example:</p> <p>Instead of:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>150 previous messages </code></pre> </div> <p>Send:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Conversation Summary: • User uses Go • Building REST APIs • Uses PostgreSQL • Currently debugging authentication </code></pre> </div> <p>Far fewer tokens.</p> <p>Much easier for the model.</p> <h3> ✅ Keep System Prompts Focused </h3> <p>System instructions should define behavior,</p> <p>not contain an entire company handbook.</p> <h3> ✅ Separate Long-Term Memory from Current Context </h3> <p>Store reusable preferences separately.</p> <p>Retrieve them only when needed.</p> <h3> ✅ Measure Token Usage </h3> <p>Don't optimize blindly.</p> <p>Most AI SDKs expose token usage statistics.</p> <p>Monitoring them helps identify unnecessary context and control costs.</p> <h1> 🚫 Common Context Engineering Mistakes </h1> <p>Here are some patterns that often reduce AI quality rather than improve it.</p> <h3> ❌ Dumping entire documents into every request </h3> <p>More information isn't automatically better.</p> <h3> ❌ Sending the full conversation forever </h3> <p>Long conversations should be summarized or trimmed when appropriate.</p> <h3> ❌ Treating memory as conversation history </h3> <p>They're different concepts.</p> <h3> ❌ Ignoring token usage </h3> <p>Unexpected token growth often leads to higher costs and slower responses.</p> <h3> ❌ Assuming bigger context windows solve everything </h3> <p>Context quality matters far more than context quantity.</p> <h1> 🎯 Best Practices </h1> <p>If you're building AI-powered applications, keep these principles in mind:</p> <p>✅ Send only the information required for the current task.</p> <p>✅ Prefer relevant context over larger context.</p> <p>✅ Separate memory from conversation history.</p> <p>✅ Summarize older interactions instead of continuously expanding prompts.</p> <p>✅ Monitor token usage in production.</p> <p>✅ Design context deliberately instead of simply appending more text.</p> <h1> 🏁 Final Thoughts </h1> <p>When people first start working with AI, they often focus on prompts.</p> <p>Then they discover Context Engineering.</p> <p>Eventually, they realize another important truth:</p> <p><strong>Not all context is equally valuable.</strong></p> <p>Modern AI systems succeed because they carefully decide:</p> <ul> <li>What to include</li> <li>What to exclude</li> <li>What to summarize</li> <li>What to retrieve</li> <li>What to remember</li> </ul> <p>That decision-making process is often more important than the prompt itself.</p> <p>The best AI applications aren't the ones with the biggest context windows.</p> <p>They're the ones that provide <strong>the right information at the right time</strong>.</p> <p>And that's exactly what Context Engineering is about.</p> <h1> 📌 Key Takeaways </h1> <ul> <li>Tokens are the fundamental units processed by language models—they are not the same as words.</li> <li>Every request consumes tokens, including system prompts, conversation history, retrieved documents, tool outputs and the model's response.</li> <li>A context window defines how much information a model can consider at once.</li> <li>Larger context windows are useful, but more context doesn't automatically lead to better answers.</li> <li>Context and memory are different: context exists for the current request, while memory is information intentionally stored and reintroduced by the application.</li> <li>Effective Context Engineering is about sending <strong>relevant</strong> information, not <strong>all</strong> available information.</li> <li>Good context management improves response quality, reduces costs and lowers latency.</li> </ul> <h1> 🚀 Coming Up in Part 3 </h1> <p>So far, we've answered:</p> <ul> <li>What is Context Engineering?</li> <li>What are tokens?</li> <li>How do context windows work?</li> <li>Why isn't bigger context always better?</li> <li>What's the difference between context and memory?</li> </ul> <p>But another important question remains:</p> <blockquote> <p><strong>Where does all this context actually come from?</strong></p> </blockquote> <p>How do AI applications automatically search documentation, query databases, call APIs, execute tools and decide what information should be sent to the model?</p> <p>That's where concepts like <strong>Retrieval-Augmented Generation (RAG)</strong>, <strong>Tool Calling</strong>, <strong>Model Context Protocol (MCP)</strong> and <strong>AI Agents</strong> come into the picture.</p> <p>We'll break each of them down in <strong>Part 3</strong> with practical examples and real-world engineering scenarios.</p>

<p>Most engineers default to TTL for cache invalidation. But a simple Time-To-Live guarantees <em>stale reads</em> for a duration, which is unacceptable for critical data that demands immediate read-after-write consistency.</p> <p>Relying solely on TTL can lead to frustrating production outages and customer trust issues. Imagine a user updating their profile picture, only for their friends to see the old one for another five minutes. Or worse, a customer sees their bank balance after a transfer and it shows the old amount, leading them to believe the transaction failed. These aren't edge cases; they're common failures when caching is poorly integrated with data updates, directly impacting user experience and potentially costing real money.</p> <h2> Beyond TTL: Explicit Cache Invalidation </h2> <p>For data that demands read-after-write consistency, you cannot rely on a probabilistic, time-based expiry. You need <em>explicit invalidation</em>. The application must actively remove or update a cache entry whenever the underlying data changes in the source of truth (typically a database).</p> <p>The most robust and common pattern is "Cache Aside with explicit invalidation." Here’s the flow:</p> <ol> <li> <strong>Read:</strong> Application tries to read data from the cache. <ul> <li> <strong>Cache Hit:</strong> Return data immediately.</li> <li> <strong>Cache Miss:</strong> Application reads data from the database, stores it in the cache, then returns it.</li> </ul> </li> <li> <strong>Write (Update/Delete):</strong> Application writes (updates or deletes) data directly to the database.</li> <li> <strong>Invalidate:</strong> <em>Immediately after</em> the successful database write, the application explicitly deletes the corresponding entry from the cache. This ensures the next read will be a cache miss, fetching the fresh data from the database.</li> </ol> <p>This strategy ensures that any subsequent read after a write will either hit an empty cache (triggering a fresh database read) or hit a cache that still holds the old value <em>briefly</em> until the invalidation propagates. For true read-after-write consistency, the invalidation must happen <em>before</em> any subsequent read.</p> <p>Here's an ASCII diagram for a single application instance interacting with a single cache instance:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>+--------+ +----------+ +----------+ | Client | &lt;--&gt; | App | &lt;--&gt; | Database | +--------+ +----------+ +----------+ | ^ | | (1. Cache Miss -&gt; Read DB) v | (2. Store in Cache) +-----+ |Cache| +-----+ ^ | (3. Invalidate on Write) </code></pre> </div> <p>In a distributed system with multiple application instances and multiple cache nodes (e.g., a Redis cluster or a Memcached fleet), explicit invalidation becomes more complex. You can't just delete from one cache node; you need to ensure all relevant cache nodes invalidate that entry. This is typically handled via a distributed messaging system (like Kafka, RabbitMQ, or Redis Pub/Sub) or by making the cache invalidation a part of a distributed transaction or event stream.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>+--------+ +----------+ +----------+ | Client | &lt;--&gt; | App 1 | &lt;--&gt; | Database | +--------+ +----------+ +----------+ | ^ | | | | (DB Write) v | v +-----+ +----------------+ |Cache| | Invalidation | | Node| &lt;------- | Message Bus | | A | | (e.g., Kafka) | +-----+ +----------------+ ^ ^ | | | (Publishes invalidation) | | (Subscribes to invalidation) v v +-----+ +-----+ |Cache| |Cache| | Node| &lt;------- | Node| | B | | C | +-----+ +-----+ </code></pre> </div> <p>When <code>App 1</code> writes to the database, it also publishes an invalidation message (e.g., "invalidate key 'user:123'") to the Message Bus. All cache nodes, or services that manage cache nodes, subscribe to this bus and delete the specific key from their local cache. This ensures eventual consistency across cache replicas for invalidation, which is typically fast enough (tens of milliseconds latency for most message buses).</p> <h2> Uber's Profile Service Invalidation </h2> <p>Consider Uber's profile service. When a driver updates their vehicle registration or a rider changes their payment method, this critical information needs to be instantly consistent across various downstream services (e.g., matching engine, billing, support).</p> <p>Uber likely employs a pattern similar to "Cache Aside with explicit invalidation" for such high-priority data. When a write occurs to the underlying persistent storage (e.g., a sharded MySQL database or a NoSQL store), the service responsible for that write explicitly invalidates the relevant key in their distributed cache (e.g., a massive Redis cluster). This invalidation isn't just a local operation; it's propagated across regions and cache replicas.</p> <p>For example, a <code>Profile Update Service</code> might:</p> <ol> <li> Persist data to MySQL (typically takes 5-10ms).</li> <li> On successful commit, publish an event to a Kafka topic like <code>profile_updates</code> (adds &lt;1ms to latency for publishing).</li> <li> A dedicated <code>Cache Invalidation Service</code> (or individual services with their own caches) subscribes to <code>profile_updates</code>.</li> <li> Upon receiving an event for <code>user:123</code>, it issues a <code>DELETE user:123</code> command to the Redis cluster (typically &lt;1ms for a local delete).</li> </ol> <p>This ensures that within tens of milliseconds, even across geographically dispersed data centers, stale data is purged from caches. For reads hitting other services, if they cache user profiles, they also subscribe to the <code>profile_updates</code> topic, ensuring they too invalidate their local caches. This pattern helps Uber achieve a consistently high cache hit ratio (&gt;95%) for frequently accessed data while guaranteeing strong read-after-write consistency for critical updates.</p> <h2> Common Mistakes </h2> <ol> <li> <strong>Assuming TTL is enough for everything.</strong> TTL is great for less critical data where eventual consistency is acceptable (e.g., trending topics, non-essential recommendations). But for user-generated content, financial data, or critical configuration, TTL is a recipe for stale data and customer complaints. Engineers often apply a blanket TTL policy without differentiating data consistency needs.</li> <li> <strong>Forgetting about Race Conditions during invalidation.</strong> A classic race condition: <ul> <li> App writes to DB.</li> <li> App tries to invalidate cache.</li> <li> <em>Before invalidation happens</em>, another App instance reads from the cache, gets old data, and stores it back in the cache (if it's a "miss and populate" strategy, or if the invalidation fails).</li> <li> The initial App instance successfully invalidates.</li> <li> Result: the cache now contains the old data again. The safest way to avoid this is to DELETE from cache <em>after</em> a successful DB write, and if a read hits a stale cache <em>before</em> invalidation, it will get stale data <em>once</em>. The subsequent read will correctly fetch fresh data. To mitigate the race, some systems use a "delete-then-write" pattern, but this can lead to temporary cache misses and increased DB load. A more robust solution involves versioning (e.g., write data to cache <em>with</em> a version, only update if versions match or newer) or ensuring cache writes are atomic with DB writes (e.g., transactional outbox pattern).</li> </ul> </li> <li> <strong>No retry mechanism for invalidation.</strong> Network issues, cache server failures, or transient errors can cause an invalidation request to fail. If your application doesn't retry invalidation, the cache will remain stale indefinitely, leading to hard-to-debug issues. Always build in retry logic with exponential backoff for cache operations.</li> <li> <strong>Blindly invalidating everything.</strong> When a complex object changes, some engineers invalidate the entire object or even all related objects. This can lead to a drastic drop in cache hit ratio, increased load on the database, and degraded performance. Design your cache keys and invalidation strategy to be as granular as possible. If only a small field within a large user profile changes, invalidate only that specific user's profile, not all profiles.</li> </ol> <h2> Interview Angle </h2> <p>When an interviewer asks about caching, especially about invalidation, they're looking for your understanding of consistency models and practical trade-offs beyond simple TTL.</p> <p><strong>Common Follow-up Questions:</strong></p> <ul> <li> <strong>"How do you achieve read-after-write consistency in a distributed cache?"</strong> <ul> <li> <strong>Strong Answer:</strong> "We'd use a 'Cache Aside' pattern. For extremely critical data, you might even consider a transactional outbox pattern where the DB write and cache invalidation message are part of an atomic transaction."</li> </ul> </li> <li> <strong>"How do you handle this with multiple cache replicas/regions?"</strong> <ul> <li> <strong>Strong Answer:</strong> "We'd use a publish/subscribe mechanism. The service performing the write publishes an invalidation event to a message queue (e.g., Kafka). All cache instances (or cache management services) in different regions or data centers subscribe to this queue and, upon receiving the event, explicitly delete the corresponding key from their local cache. This ensures eventual consistency for cache invalidation across the distributed system, typically within tens of milliseconds."</li> </ul> </li> <li> <strong>"What are the trade-offs of explicit invalidation vs. TTL?"</strong> <ul> <li> <strong>Strong Answer:</strong> "Explicit invalidation offers strong read-after-write consistency, crucial for critical data, but adds complexity to the write path (extra cache operation, retry logic, distributed messaging). The latency incurred during leader election directly dictates your system's Mean Time To Recovery (MTTR) for critical control plane operations, often measured in hundreds of milliseconds of complete unavailability.</p> <p>Imagine a payment processing service where a single "controller" node is responsible for assigning transaction IDs and routing requests to different payment gateways. While this is crucial for data consistency, a common and critical application is agreeing on a <em>leader</em> node. This leader is often responsible for coordinating other nodes, managing metadata, or being the sole write-endpoint to simplify consistency models.</p> <p>The election process typically involves a set of nodes (a quorum) voting to select a new leader when the current one is perceived to have failed.</p> <p>Here's a simplified flow for a leader election in a 3-node cluster:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code> +-----------------+ | Leader A | (Active) +--------+--------+ | Heartbeats (e.g., every 100ms) | +----------------V------------------+ | | +-----+-----+ +-----+-----+ | Follower B| | Follower C| +-----------+ +-----------+ Scenario: Leader A fails. They become "candidates," increment an election term, and send <code>RequestVote</code> messages to other nodes. The first candidate to secure a majority of votes (e.g., 2 out of 3 nodes, including its own implied vote) becomes the new leader. This process can take anywhere from tens of milliseconds to several seconds depending on network conditions, timeout settings, and system load.</p> <h2> Real-World Application: Apache Kafka's Controller Election </h2> <p>Apache Kafka, a distributed streaming platform, relies heavily on a leader election mechanism for its "controller" node. The Kafka controller is a special broker responsible for critical cluster-wide operations: managing partition leader elections, orchestrating replica assignments, creating/deleting topics, and managing broker membership.</p> <p>Historically, Kafka used Apache ZooKeeper for controller election and metadata management. When the Kafka controller broker crashes:</p> <ol> <li> Other brokers detect its failure (via ZooKeeper session expiration).</li> <li> A new controller election is triggered in ZooKeeper. ZooKeeper, running its own consensus algorithm (like ZAB for atomic broadcast, similar to Paxos/Raft), facilitates this election.</li> <li> The election latency is dominated by ZooKeeper's consensus protocol and network round-trip times (RTT) between ZooKeeper ensemble members. In a typical 5-node ZooKeeper ensemble spread across availability zones, an election might take <strong>200ms to 2 seconds</strong>.</li> <li> During this period, the Kafka cluster is effectively "headless." This means <strong>writes to partitions without a leader fail, reads might serve stale data or fail, and scaling operations are impossible</strong>.</li> </ol> <p>While a 2-second recovery time might seem acceptable, for high-throughput, low-latency systems, this is a significant availability hit for the control plane. A 5-node quorum pushes this higher. This isn't just a one-off hit; it's the baseline for your control plane's MTTR.</li> <li> <strong>Ignoring the impact of leader election on data plane operations:</strong> It's common to only think about how consensus ensures data consistency. What most people get wrong is that a stalled leader election can completely block critical data plane operations indirectly. If a service depends on the leader to fetch configurations, assign work, or route requests, a leaderless period means the data plane also stalls. Kafka is a prime example: no controller means no new partition leaders, blocking writes and reads to affected partitions.</li> <li> <strong>Configuring aggressive timeouts without proper network analysis:</strong> Setting very short heartbeats or election timeouts (e.g., 100ms) might seem good for fast recovery. However, in an overloaded system or one with transient network jitter, this can lead to "flapping" – constant re-elections as nodes prematurely declare the leader dead, causing instability and consuming resources for fruitless elections. This leads to higher CPU usage, increased network traffic, and <em>longer</em> overall downtime as the system struggles to stabilize.</li> </ol> <h2> Interview Angle </h2> <p>When discussing distributed consensus in an interview, especially concerning leader election, expect these follow-up questions:</p> <ul> <li> <strong>"How does leader election impact overall system availability and latency?"</strong> <ul> <li> <strong>Strong Answer:</strong> "Leader election directly affects the system's Mean Time To Recovery (MTTR) for control-plane operations.
