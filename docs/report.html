There's no standard status page, and other lessons from tracking 96 providers
原文説明
原文説明（英語）

<p>Your Slack won't load. Is it you, your VPN, the office wifi, or Slack? You open Slack's status page and it's green. It's often green while you're clearly down, because a company updates its own status page <em>after</em> it confirms an incident — not the moment you feel it.</p> <p>I spent the last few months building <a href="https://outagedeck.com" rel="noopener noreferrer">OutageDeck</a>, which reads the official status source of 96 cloud and SaaS providers and folds them into one place. I assumed the hard part would be the frontend. It wasn't. The hard part was that "official status" is a swamp. Here's what I found in it.</p> <p><strong>The short version:</strong></p> <ul> <li>There is no standard status format. Statuspage is common, but the tail is bespoke — Google, Slack, Heroku, Azure over RSS, and AWS in UTF-16.</li> <li>Providers' own timestamps lag, sometimes by days, so you can't compute uptime from them.</li> <li>Some official feeds are stale fossils, or model thousands of instances, so "is it up?" has no single answer.</li> <li>Official status is authoritative but late — which is the whole reason crowd-sourced "is it down?" sites exist.</li> <li>All of this pushed me toward a deliberately tiny backend: one Next.js app and a Postgres database, no queue, about $55/month.</li> </ul> <h2> There is no standard </h2> <p>Atlassian Statuspage is the closest thing to a common language. Around 89 of my 96 providers expose a Statuspage-style <code>summary.json</code> and <code>incidents.json</code>, and one adapter reads all of them. Some providers don't even run Statuspage but copy its shape anyway — incident.io serves a Statuspage-compatible endpoint on each customer's own domain, so Notion and Linear parse with the exact same code.</p> <p>Then the long tail starts. Google publishes an <code>incidents.json</code> in its own format (shared across Cloud, Workspace, and Firebase). Slack has a bespoke API. Heroku is on the fourth version of its own. Azure only offers RSS. AWS ships an event feed encoded in <strong>UTF-16 with a byte-order mark</strong>, which silently breaks any parser that assumes UTF-8. "Just read the status page" turned into six adapters and a pile of encoding checks.</p> <h2> The timestamps lie </h2> <p>My first uptime numbers were nonsense, and it took me a while to see why: I was computing them from each provider's own <code>captured_at</code> timestamp, and those lag. One provider's was <strong>days</strong> behind at one point. If a feed says a state was "captured three days ago," you can't tell whether the service was down for three days or the timestamp is just stale.</p> <p>The fix was to stop trusting upstream time entirely. Every observation is stamped with <em>my</em> check time, and uptime is computed from when I saw a state, not when the provider says it happened. The corollary matters too: if my own poller has a gap — a deploy, say — I don't backfill it as "up." Any gap longer than 30 minutes is recorded as <em>no data</em>, because inventing green is worse than admitting I wasn't looking.</p> <h2> Some official sources are fossils </h2> <p>A few providers technically have a machine-readable status endpoint that you still can't use. Stripe's legacy status JSON has been effectively frozen since 2024; it will cheerfully report that everything is fine, forever. Salesforce's status API models nearly 3,900 separate instances, so "is Salesforce up?" depends entirely on which pod your org sits on.</p> <p>For sources like these, the honest move is to <em>not</em> ship them until I have something I'd trust myself. That's why they're still missing from the list. I'd rather explain a gap than fake a green light.</p> <h2> What the feeds did to the backend </h2> <p>Two properties of these feeds shaped the whole system.</p> <p><strong>They re-send their full recent history on every poll,</strong> and an unchanged upstream just re-stamps the same row. So the raw data can't reconstruct history — you can't diff yesterday back out of it. Instead of trying to rederive it later, I roll each poll's observations into daily per-provider buckets at write time. I also skip writing rows that didn't change; an early version rewrote identical rows every run and burned the database's IO budget for nothing.</p> <p><strong>I run two schedulers on purpose:</strong> a 10-minute cron as the primary, and a 30-minute job as a backstop. So runs overlap. Rather than add a queue or a lock service, I let Postgres be the referee. Alert deliveries are claimed with a unique constraint before anything goes out:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight sql"><code><span class="c1">-- alert_deliveries has: unique (subscription_id, event_key)</span> <span class="k">insert</span> <span class="k">into</span> <span class="n">alert_deliveries</span> <span class="p">(</span><span class="n">subscription_id</span><span class="p">,</span> <span class="n">event_key</span><span class="p">,</span> <span class="n">payload</span><span class="p">)</span> <span class="k">values</span> <span class="p">(</span><span class="err">$</span><span class="mi">1</span><span class="p">,</span> <span class="err">$</span><span class="mi">2</span><span class="p">,</span> <span class="err">$</span><span class="mi">3</span><span class="p">)</span> <span class="k">on</span> <span class="n">conflict</span> <span class="p">(</span><span class="n">subscription_id</span><span class="p">,</span> <span class="n">event_key</span><span class="p">)</span> <span class="k">do</span> <span class="k">nothing</span><span class="p">;</span> <span class="c1">-- 1 row inserted -&gt; I won the claim, send the alert</span> <span class="c1">-- 0 rows inserted -&gt; another run already has it, skip</span> </code></pre> </div> <p>The insert that survives the unique index is the one that sends; the overlapping run writes nothing and moves on. No queue, no exactly-once framework, no dead-letter topic. The uptime writes take a per-entity advisory lock for the same reason. The whole thing is one Next.js app and a Postgres database, no background workers, running on about <strong>$55 a month</strong>.</p> <h2> So I built the thing I wanted </h2> <p>OutageDeck is the result: one place to check whether 96 major providers are up, sourced from each one's official status feed rather than from user complaints. It's free to use and there's no signup to check a provider. Under the hood it's the pipeline above — official-only, honest about timestamps, and clear when it just doesn't know yet.</p> <p>If you want more than a webpage, there's a public JSON API, embeddable SVG status badges you can drop in a README, RSS feeds per provider, and an OpenAPI 3.1 doc.</p> <p>Two honest caveats: incident history only goes as deep as each upstream feed exposes, so this isn't a system of record for ancient outages; and I'm still adding providers as I find sources I actually trust.</p> <h2> Try it, and stay in the loop </h2> <ul> <li> <strong>Check any provider, free, no signup:</strong> <a href="https://outagedeck.com" rel="noopener noreferrer">outagedeck.com</a> </li> <li> <strong>Want a heads-up when something breaks?</strong> Free email alerts for up to 5 providers; paid plans add Slack, Discord, and webhooks.</li> <li> <strong>Prefer a digest?</strong> I send a short weekly rundown, <em>This week in cloud outages</em> — you can subscribe on the site.</li> </ul> <p>I'd genuinely like to know two things in the comments: <strong>which providers you'd want tracked next</strong>, and <strong>what would make the API useful inside your own monitoring</strong>. I'll be writing up more of the internals (the uptime rollups, the fail-open API) — follow along if that's your thing.</p>
