<h1> ClickHouse® Data Sampling: Querying Billions of Rows Fast </h1> <p><em>Day 53 of #100DaysOfClickHouse</em></p> <p>Modern analytics platforms routinely manage billions—or even trillions—of rows of data. Although ClickHouse® is built to execute analytical queries at incredible speed, there are many situations where an approximate answer is more valuable than waiting for an exact one.</p> <p>Imagine you're monitoring application metrics, exploring user behavior, or checking whether a marketing campaign is performing as expected. In these scenarios, getting an answer in a few hundred milliseconds can be far more useful than waiting several seconds for perfect accuracy.</p> <p>This is exactly where <strong>data sampling</strong> becomes valuable.</p> <p>Instead of reading every row in a table, ClickHouse® can process only a portion of the data and generate approximate results. This reduces disk I/O, CPU usage, and query execution time, making exploratory analytics significantly faster.</p> <p>However, sampling is not a universal optimization. It comes with a trade-off between speed and accuracy, and it only works efficiently when your table is designed to support it.</p> <p>In this article, we'll explore how data sampling works in ClickHouse®, when it should be used, and the best practices for designing tables that take advantage of it.</p> <h1> What Is Data Sampling? </h1> <p>Data sampling is the process of executing a query against only a subset of the available data rather than scanning every row.</p> <p>Consider a table containing <strong>10 billion events</strong>.</p> <p>Without sampling, ClickHouse® reads all 10 billion rows to produce an exact result.</p> <p>With sampling enabled, ClickHouse® may read only 10% of the data and return an approximate answer based on that subset.</p> <p>Because less data is processed, sampling typically provides several benefits:</p> <ul> <li>Faster query execution</li> <li>Lower CPU utilization</li> <li>Reduced disk I/O</li> <li>Lower overall resource consumption</li> </ul> <p>The trade-off is straightforward:</p> <p><strong>The results are approximate rather than exact.</strong></p> <p>Whether this approximation is acceptable depends entirely on your use case.</p> <h1> How Sampling Works in ClickHouse® </h1> <p>One common misconception is that ClickHouse® randomly selects rows every time a sampled query runs.</p> <p>That's not how sampling works.</p> <p>Instead, efficient sampling depends on how the table was designed.</p> <p>For MergeTree tables, you specify a sampling key using the <code>SAMPLE BY</code> clause during table creation.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight sql"><code><span class="k">CREATE</span> <span class="k">TABLE</span> <span class="n">events</span> <span class="p">(</span> <span class="n">user_id</span> <span class="n">UInt64</span><span class="p">,</span> <span class="n">event_time</span> <span class="nb">DateTime</span><span class="p">,</span> <span class="n">event_name</span> <span class="n">String</span> <span class="p">)</span> <span class="n">ENGINE</span> <span class="o">=</span> <span class="n">MergeTree</span> <span class="k">ORDER</span> <span class="k">BY</span> <span class="p">(</span><span class="n">user_id</span><span class="p">,</span> <span class="n">event_time</span><span class="p">)</span> <span class="n">SAMPLE</span> <span class="k">BY</span> <span class="n">user_id</span><span class="p">;</span> </code></pre> </div> <p>The sampling key determines how data is organized for efficient sampling.</p> <p>Later, you can query a portion of the table using:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight sql"><code><span class="k">SELECT</span> <span class="k">count</span><span class="p">()</span> <span class="k">FROM</span> <span class="n">events</span> <span class="n">SAMPLE</span> <span class="mi">0</span><span class="p">.</span><span class="mi">1</span><span class="p">;</span> </code></pre> </div> <p>Rather than randomly inspecting rows, ClickHouse® deterministically reads approximately 10% of the data based on the sampling key.</p> <p>This deterministic behavior provides an important advantage:</p> <p>Running the same sampled query multiple times returns the same subset of data, provided the underlying data has not changed.</p> <p>This makes sampled queries consistent and predictable.</p> <h1> Why the <code>SAMPLE BY</code> Clause Matters </h1> <p>The <code>SAMPLE</code> clause only performs efficiently when the table has been created with an appropriate sampling expression.</p> <p>Without a sampling key, ClickHouse® has no efficient way to determine which portions of the data should be read.</p> <p>Simply adding:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight sql"><code><span class="n">SAMPLE</span> <span class="mi">0</span><span class="p">.</span><span class="mi">1</span> </code></pre> </div> <p>to every query will not automatically improve performance.</p> <p>Sampling is a table design feature rather than a query optimization that can be added later.</p> <p>If you expect to use sampling regularly, it should be considered during schema design.</p> <h1> Understanding the <code>SAMPLE</code> Clause </h1> <p>The most common syntax is:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight sql"><code><span class="k">SELECT</span> <span class="o">*</span> <span class="k">FROM</span> <span class="n">events</span> <span class="n">SAMPLE</span> <span class="mi">0</span><span class="p">.</span><span class="mi">1</span><span class="p">;</span> </code></pre> </div> <p>Here, <code>0.1</code> instructs ClickHouse® to process approximately 10% of the data.</p> <p>Other examples include:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight sql"><code><span class="n">SAMPLE</span> <span class="mi">0</span><span class="p">.</span><span class="mi">25</span> </code></pre> </div> <p>which processes roughly 25% of the data.</p> <p>Or:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight sql"><code><span class="n">SAMPLE</span> <span class="mi">0</span><span class="p">.</span><span class="mi">5</span> </code></pre> </div> <p>which reads approximately half the table.</p> <p>Larger sampling ratios generally provide more accurate results but require reading more data.</p> <p>Smaller ratios improve speed while increasing estimation error.</p> <p>There is no universally correct sampling ratio—the ideal choice depends on your workload and acceptable level of approximation.</p> <h1> Choosing a Good Sampling Key </h1> <p>Selecting the right sampling key is one of the most important design decisions.</p> <p>An effective sampling key should generally have:</p> <ul> <li>High cardinality</li> <li>Even distribution</li> <li>Stable values</li> </ul> <p>Common examples include:</p> <ul> <li><code>user_id</code></li> <li><code>session_id</code></li> <li><code>device_id</code></li> </ul> <p>These columns usually distribute rows evenly across the dataset, producing representative samples.</p> <p>In contrast, low-cardinality columns such as:</p> <ul> <li><code>country</code></li> <li><code>status</code></li> <li><code>gender</code></li> </ul> <p>may create biased or uneven samples depending on the underlying data distribution.</p> <p>Choosing the right sampling key is often just as important as selecting the sampling ratio.</p> <h1> Speed vs Accuracy </h1> <p>Sampling always involves a trade-off.</p> <div class="table-wrapper-paragraph"><table> <thead> <tr> <th>Full Scan</th> <th>Sampled Query</th> </tr> </thead> <tbody> <tr> <td>Exact results</td> <td>Approximate results</td> </tr> <tr> <td>Reads all matching rows</td> <td>Reads only a subset</td> </tr> <tr> <td>Higher resource usage</td> <td>Lower resource usage</td> </tr> <tr> <td>Best for reporting</td> <td>Best for exploration</td> </tr> </tbody> </table></div> <p>The quality of sampled results depends on several factors:</p> <ul> <li>Sampling ratio</li> <li>Sampling key</li> <li>Data distribution</li> <li>Aggregation being performed</li> </ul> <p>For example, calculating an average from a representative sample often produces reliable estimates.</p> <p>However, estimating very rare events using a small sample may lead to significant inaccuracies.</p> <h1> When Should You Use Data Sampling? </h1> <p>Sampling is most valuable when approximate answers are sufficient.</p> <p>Common use cases include:</p> <ul> <li>Interactive analytics</li> <li>Dashboard exploration</li> <li>Product analytics</li> <li>Monitoring event streams</li> <li>Trend analysis</li> <li>Investigating large datasets before running exact queries</li> </ul> <p>In these situations, reducing query latency often provides more value than perfect precision.</p> <h1> When Should You Avoid Sampling? </h1> <p>Some workloads require exact answers.</p> <p>Sampling should generally be avoided for:</p> <ul> <li>Financial reporting</li> <li>Billing systems</li> <li>Compliance reporting</li> <li>Tax calculations</li> <li>Exact customer counts</li> <li>Regulatory reporting</li> </ul> <p>Whenever correctness is more important than execution speed, a full scan remains the better choice.</p> <h1> Performance Expectations </h1> <p>It's tempting to assume that sampling always produces dramatic speed improvements.</p> <p>In reality, the performance gain depends on several factors, including:</p> <ul> <li>Table design</li> <li>Sampling ratio</li> <li>Query complexity</li> <li>Data distribution</li> <li>Storage layout</li> <li>Available hardware</li> </ul> <p>Queries that already read only a small portion of the table because of highly selective filters may gain little from sampling.</p> <p>Similarly, very small tables often don't benefit enough for sampling to justify the loss in accuracy.</p> <p>Instead of assuming a fixed improvement, benchmark sampled and non-sampled queries against your own workload.</p> <h1> Best Practices </h1> <p>If you plan to use data sampling in production, consider these recommendations:</p> <ul> <li>Design tables with an appropriate <code>SAMPLE BY</code> expression from the beginning.</li> <li>Choose high-cardinality sampling keys whenever possible.</li> <li>Periodically compare sampled results against full scans to measure estimation error.</li> <li>Increase the sampling ratio when higher accuracy is required.</li> <li>Clearly label sampled metrics as <strong>approximate</strong> in dashboards and reports.</li> </ul> <p>These practices help users understand the limitations of sampled data while still benefiting from faster query execution.</p> <h1> Conclusion </h1> <p>ClickHouse® data sampling is a powerful feature for accelerating analytical queries when approximate answers are acceptable.</p> <p>By processing only a subset of the data, ClickHouse® can significantly reduce disk I/O, CPU usage, and query latency, making exploratory analytics much more responsive. When combined with a well-designed <code>SAMPLE BY</code> expression and an appropriate high-cardinality sampling key, sampling becomes an effective tool for working with massive datasets.</p> <p>However, sampling is not a replacement for exact analytics. It introduces estimation error, requires thoughtful schema design, and should not be used for workloads where every record matters.</p> <p>Like many performance optimization techniques, sampling is most effective when applied intentionally. Understanding its strengths, limitations, and design considerations allows you to make informed decisions about when fast approximate answers are sufficient—and when a full scan is the better choice.</p> <p>If you're following along, this is <strong>Day 53 of #100DaysOfClickHouse</strong>, where I'm exploring one ClickHouse® concept every day and sharing practical examples, best practices, and real-world use cases.</p> <p>Read more... <a href="https://www.quantrail-data.com/clickhouse-data-sampling-querying-billions-of-rows-fast" rel="noopener noreferrer">https://www.quantrail-data.com/clickhouse-data-sampling-querying-billions-of-rows-fast</a></p>

Agents, models, and tools all become custom resources you apply with <code>kubectl</code>.</p> <p>👉 The big idea: kagent puts one clean abstraction between your agents and your models, so you can point the same agent at Claude in the cloud or at Ollama running in your own cluster, and switch by changing a single field.</p> <p>In this guide we will build a small but real inference stack in a homelab cluster, wired to both.</p> <h2> 🧱 What we will cover </h2> <ul> <li>✅ What the kagent inference stack actually looks like</li> <li>✅ Installing kagent with Helm</li> <li>✅ Wiring up Claude for cloud inference (the ModelConfig resource)</li> <li>✅ Deploying Ollama inside the cluster for fully local, on prem inference</li> <li>✅ Building agents that use each model with real Kubernetes tools</li> <li>✅ Swapping models with a one line change</li> <li>✅ Adding a human approval gate so your agent never nukes the wrong namespace</li> </ul> <h2> 🗺️ The inference stack, layer by layer </h2> <p>Before we touch YAML, here is the mental model. In kagent your "inference stack" is just a few layers, each a Kubernetes resource or component:</p> <ul> <li>✅ Model provider: where the tokens come from. Claude via the Anthropic API, or Ollama serving a local model.</li> <li>✅ ModelConfig (CRD): tells kagent which provider, which model, and which secret holds the key.</li> <li>✅ Agent (CRD): a system prompt plus a set of tools plus a reference to a ModelConfig.</li> <li>✅ Tools over MCP: kagent ships a tool server that exposes Kubernetes actions as tools your agent can call.</li> <li>✅ Controller and runtime: the Go controller watches the CRDs, and the runtime (Agent Development Kit) runs the agent loop.</li> <li>✅ Storage: a bundled PostgreSQL instance for agent state (swap it for your own in production).</li> </ul> <p>The nice part is that the Agent resource does not care where the model lives. It just references a ModelConfig by name. Cloud or local, the agent spec looks the same.</p> <h2> 🧰 Prerequisites </h2> <p>Nothing exotic, just homelab basics:</p> <ul> <li>✅ A running Kubernetes cluster (k3s, kind, minikube, or your bare metal cluster all work)</li> <li>✅ <code>kubectl</code> and <code>helm</code> installed and pointing at that cluster</li> <li>✅ An Anthropic API key for the Claude part (grab one from the Anthropic console)</li> <li>✅ Enough CPU and RAM on a node for a local model (an 8B model like <code>llama3.1</code> is happy with a couple of cores and around 8Gi of memory)</li> </ul> <blockquote> <p>Heads up on versions: everything below was verified against kagent v0.9 (CRD apiVersion <code>kagent.dev/v1alpha2</code>), current Claude model ids <code>claude-haiku-4-5</code> and <code>claude-sonnet-4-5</code>, and the Ollama <code>llama3.1</code> model, all checked in July 2026. kagent moves fast, so if a field looks different, check the release notes.</p> </blockquote> <h2> 📦 Step 1: Install kagent </h2> <p>kagent installs as two Helm charts: one for the CRDs, one for the controller and friends.
