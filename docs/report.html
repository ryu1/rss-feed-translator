# ColibriのおかげでLLMの運営方法を再考することになった ― そして次のステップがあるかもしれない
# Colibri Made Me Rethink How We Run LLMs — And There Might Be a Next Step
原文説明
原文説明（英語）

<p>Over the past few days, I came across a project that made me stop everything to understand how it worked. It wasn't just another framework for running language models, nor another GPU-based optimization. It was something different.</p> <p>The project is called <strong>Colibri</strong>.</p> <p>At first glance, it almost seems impossible. The idea is simple, yet remarkably bold: run a massive Mixture of Experts (MoE) model on an ordinary computer using nothing more than a CPU and an SSD, loading into memory only the experts that are actually needed at each step.</p> <p>When I first read that, my reaction was probably the same as everyone else's:</p> <p><em>"How could that possibly work?"</em></p> <p>After studying the architecture, I realized that Colibri's real innovation isn't simply that it can run an enormous model. The real breakthrough is that it completely changes the way we think about inference.</p> <h1> A Shift in Perspective </h1> <p>For years, we've been taught that running a model means loading the entire model into memory.</p> <p>More VRAM.</p> <p>More RAM.</p> <p>More GPUs.</p> <p>Colibri challenges that assumption.</p> <p>Instead of keeping hundreds of billions of parameters resident in memory, it treats the model as a living system.</p> <p>The router selects which experts are needed.</p> <p>Those experts are loaded from the SSD.</p> <p>They perform their computation.</p> <p>And once they're no longer needed, they can be discarded.</p> <p>It's an incredibly elegant approach.</p> <p>In practice, storage is no longer just a place where model weights are kept—it becomes an active participant in the inference process.</p> <p>It's one of those ideas that feels both simple and brilliant.</p> <h1> The Bottleneck Isn't the Architecture </h1> <p>Once I understood how it worked, another question came to mind.</p> <p><strong>What if the real problem isn't the algorithm?</strong></p> <p>Because the architecture itself makes perfect sense.</p> <p>The bottleneck appears somewhere else.</p> <p>Every time an expert is needed, there's an unavoidable sequence:</p> <p><strong>SSD → RAM → CPU</strong></p> <p>That process happens thousands of times during a single generation.</p> <p>In other words, the real limitation is no longer computational power—it's the time spent waiting for data to move from storage into memory.</p> <p>That's when I started wondering what the next step for this architecture could be.</p> <h1> What If Colibri Could Predict the Future? </h1> <p>Modern CPUs already do this.</p> <p>They try to predict which instruction will be executed next.</p> <p>Operating systems do it.</p> <p>Databases do it.</p> <p>Web browser caches do it.</p> <p>So why couldn't an MoE inference engine do the same?</p> <p>Imagine that after processing thousands of tokens, the system discovers patterns like these:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Expert 15 ↓ 92% Expert 48 Expert 48 ↓ 87% Expert 102 Expert 102 ↓ 95% Expert 211 </code></pre> </div> <p>In this scenario, the engine wouldn't need to wait for the next token before loading the next expert.</p> <p>While the CPU is computing the current expert, another thread could already be loading the most likely next experts.</p> <p>By the time they're needed, they're already sitting in RAM.</p> <p>The waiting time nearly disappears.</p> <h1> Storage Would No Longer Be the Bottleneck </h1> <p>Today, the SSD is used reactively.</p> <p>First, the router makes a decision.</p> <p>Then, the SSD responds.</p> <p>But what if we reversed that logic?</p> <p>Instead of reacting to requests, the system begins anticipating them.</p> <p>That transforms storage from a passive component into a proactive one.</p> <p>One possible architecture could consist of four independent modules:</p> <ul> <li> <strong>An intelligent scheduler</strong>, responsible for predicting upcoming experts.</li> <li> <strong>An asynchronous I/O manager</strong>, performing disk reads in parallel.</li> <li> <strong>A smart expert cache (LRU/LFU)</strong>, keeping frequently used experts in memory.</li> <li> <strong>A dedicated inference engine</strong>, focused solely on computation.</li> </ul> <p>This separation would allow disk I/O and computation to happen simultaneously, dramatically reducing idle time.</p> <h1> Other Interesting Ideas </h1> <p>As I explored the project further, a few additional possibilities came to mind.</p> <h3> Frequency-Based Caching </h3> <p>Experts that are used hundreds of times shouldn't be evicted from memory as aggressively as experts that are rarely selected.</p> <h3> Intelligent Prefetching </h3> <p>As soon as the router makes its decision, immediately begin loading the most likely upcoming experts.</p> <h3> Memory-Mapped I/O (mmap) </h3> <p>Allow the operating system to manage page caching directly, reducing unnecessary memory copies and making better use of the existing virtual memory subsystem.</p> <h3> Physical Expert Clustering </h3> <p>Experts that are frequently activated together could be stored physically close to one another on disk, enabling much more efficient sequential reads.</p> <h3> A Fully Pipelined Engine </h3> <p>While one group of experts is executing...</p> <p>Another group is already being loaded.</p> <p>And a third group is waiting in memory.</p> <p>Ideally, the CPU would spend almost no time waiting for the SSD.</p> <h1> The Core Idea Remains the Same </h1> <p>The most interesting part is that none of these ideas changes Colibri's original philosophy.</p> <p>The entire model still isn't loaded into memory.</p> <p>Inference still runs entirely on CPUs.</p> <p>The SSD remains an active part of the inference pipeline.</p> <p>The difference is that the system stops reacting to events and starts anticipating them.</p> <p>It's an evolution of the same idea.</p> <p>Not a replacement.</p> <h1> Perhaps This Is Only the Beginning </h1> <p>Colibri demonstrates that there's still plenty of room for innovation in inference engines.</p> <p>For years, the conversation has revolved around bigger GPUs, more memory, and greater computational power.</p> <p>Maybe there's another path.</p> <p>Maybe we can build increasingly intelligent systems for moving data.</p> <p>Because in the end, the hardest problem may not be computing faster.</p> <p>It may simply be ensuring that the right data is in the right place at exactly the right time.</p> <p>That's what made Colibri so fascinating to me.</p> <p>And perhaps that's exactly where the next major step in this idea lies.</p> <p>Congratulations to the creator of the project for the creativity and the courage to challenge a paradigm that many considered unquestionable. A model asked to pick from six categories is reliable; a model asked to "summarize the customer's needs" is a liability you'll be debugging at 2 a.m. The category is what drives everything downstream — which folder it lands in, whether the agent auto-replies, and whether a human gets pulled in.</p> <h2> Route and filter with policies, rules, and lists </h2> <p>Here's the move a lot of "AI inbox" builds miss: not every message should reach your application at all. Nylas Agent Accounts give you three server-side primitives that filter and sort mail <em>before</em> your webhook ever fires, so the LLM only spends tokens on things worth classifying.</p> <ul> <li> <strong>Policies</strong> bundle limits (send quotas, storage, retention) and spam detection. One policy can govern many accounts.</li> <li> <strong>Rules</strong> match inbound or outbound mail on sender/recipient fields and run actions like <code>block</code>, <code>mark_as_spam</code>, <code>assign_to_folder</code>, <code>archive</code>, or <code>trash</code>.</li> <li> <strong>Lists</strong> are typed collections of domains, TLDs, or addresses that rules reference via the <code>in_list</code> operator — so non-engineers can update an allowlist or blocklist without a deploy.</li> </ul> <p>They form a chain: lists hold values, rules reference lists and describe conditions plus actions, policies bundle limits, and a <strong>workspace</strong> carries one <code>policy_id</code> and an array of <code>rule_ids</code>. Every Agent Account in the workspace inherits both. You don't attach anything to an individual grant.</p> <p>Start by blocking a known-bad sender at the SMTP layer, so it never hits the mailbox:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>curl <span class="nt">--request</span> POST <span class="se">\</span> <span class="nt">--url</span> <span class="s2">"https://api.us.nylas.com/v3/rules"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Authorization: Bearer &lt;NYLAS_API_KEY&gt;"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Content-Type: application/json"</span> <span class="se">\</span> <span class="nt">--data</span> <span class="s1">'{ "name": "Block spam-domain.com", "priority": 1, "trigger": "inbound", "match": { "conditions": [ { "field": "from.domain", "operator": "is", "value": "spam-domain.com" } ] }, "actions": [{ "type": "block" }] }'</span> </code></pre> </div> <p>The CLI builds the same rule and attaches it to the default workspace in one shot:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>nylas agent rule create <span class="se">\</span> <span class="nt">--name</span> <span class="s2">"Block spam-domain.com"</span> <span class="se">\</span> <span class="nt">--trigger</span> inbound <span class="se">\</span> <span class="nt">--priority</span> 1 <span class="se">\</span> <span class="nt">--condition</span> from.domain,is,spam-domain.com <span class="se">\</span> <span class="nt">--action</span> block </code></pre> </div> <p>For the allowlist/blocklist pattern, create a list and reference it from a rule. The reply has to land <em>inside</em> the customer's existing conversation, not as a fresh disconnected email. Nylas handles this through <code>reply_to_message_id</code>: pass it on the send and Nylas sets the <code>In-Reply-To</code> and <code>References</code> headers for you, so the customer's mail client groups the reply correctly.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>curl <span class="nt">--request</span> POST <span class="se">\</span> <span class="nt">--url</span> <span class="s2">"https://api.us.nylas.com/v3/grants/&lt;NYLAS_GRANT_ID&gt;/messages/send"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Authorization: Bearer &lt;NYLAS_API_KEY&gt;"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Content-Type: application/json"</span> <span class="se">\</span> <span class="nt">--data</span> <span class="s1">'{ "reply_to_message_id": "msg-abc123", "to": [{ "email": "dana@customer.example" }], "subject": "Re: Can'</span><span class="se">\'</span><span class="s1">'t log in after password reset", "body": "Hi Dana — I see the reset went through an hour ago..." }'</span> </code></pre> </div> <p>From the CLI, <code>nylas email reply</code> does the bookkeeping for you. Point it at the message ID and it fetches the original to populate the recipient and subject, then preserves threading via <code>reply_to_message_id</code> automatically:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>nylas email reply msg-abc123 <span class="nt">--body</span> <span class="s2">"Hi Dana — I see the reset went through an hour ago..."</span> </code></pre> </div> <p>By default the reply goes only to the original sender; add <code>--all</code> if you need to keep the rest of the To/Cc on the thread. This is the command I lean on most when I'm testing an agent by hand — it's the shortest path from "a message landed" to "a threaded reply went out."</p> <h2> Don't reply twice </h2> <p>This is where naive builds fall over in production. Webhooks are delivered <strong>at least once</strong> — if your endpoint is slow to <code>200</code> or there's a network blip, you'll get the same <code>message.created</code> again. Add concurrent workers and two of them can grab the same notification at the same millisecond. Either way: two replies, one angry customer.</p> <p>The fix is idempotency, and it lives entirely in your own store — not in Nylas. Custom metadata tagging on Agent Account resources isn't supported yet, so you can't stash dedup state on the message or grant; keep it in Redis or Postgres. Track the message IDs you've already processed and refuse to handle one twice, using an atomic check-and-set:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight javascript"><code><span class="kd">const</span> <span class="nx">messageId</span> <span class="o">=</span> <span class="nx">event</span><span class="p">.</span><span class="nx">data</span><span class="p">.</span><span class="nx">object</span><span class="p">.</span><span class="nx">id</span><span class="p">;</span> <span class="c1">// Atomic insert. Returns truthy ONLY when the key was newly inserted:</span> <span class="c1">// Redis -&gt; SET messageId 1 NX EX 86400 (truthy if it didn't exist)</span> <span class="c1">// Postgres -&gt; INSERT ... ON CONFLICT DO NOTHING, then check rowCount === 1</span> <span class="kd">const</span> <span class="nx">wasInserted</span> <span class="o">=</span> <span class="k">await</span> <span class="nx">db</span><span class="p">.</span><span class="nx">processedMessages</span><span class="p">.</span><span class="nf">setIfAbsent</span><span class="p">(</span><span class="nx">messageId</span><span class="p">,</span> <span class="p">{</span> <span class="na">receivedAt</span><span class="p">:</span> <span class="nb">Date</span><span class="p">.</span><span class="nf">now</span><span class="p">(),</span> <span class="p">});</span> <span class="k">if </span><span class="p">(</span><span class="o">!</span><span class="nx">wasInserted</span><span class="p">)</span> <span class="k">return</span><span class="p">;</span> <span class="c1">// insert did nothing -&gt; already seen, bail</span> </code></pre> </div> <p>Watch the polarity here — it's the easiest bug to ship. <code>setIfAbsent</code> succeeds (returns truthy) only the <em>first</em> time it sees a message ID; every redelivery finds the key already present and returns falsy. So you proceed when <code>wasInserted</code> is true and exit when it's false. Invert that test and you'd drop every genuine first message while happily processing the duplicates.</p> <p>Give the dedup record a TTL of about 24 hours — long enough that a webhook redelivered hours later still gets caught, short enough that the table doesn't grow forever. For the concurrent-worker race, layer a per-thread lock on top, and inside it double-check the thread's latest message: if the most recent message is already from the agent, a prior worker beat you to it and you skip.</p> <p>Dedup catches redelivered events; locking catches simultaneous ones. You want both. The full lifecycle is on the command line, which makes the CLI handy for managing webhooks across environments without writing a single request.</p> <h2> Answer the challenge before it activates </h2> <p>Here's the step that trips up almost everyone the first time: before a webhook goes live, Nylas verifies your endpoint with a challenge. Until it does, the webhook doesn't activate.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight javascript"><code><span class="nx">app</span><span class="p">.</span><span class="nf">get</span><span class="p">(</span><span class="dl">"</span><span class="s2">/webhooks/nylas</span><span class="dl">"</span><span class="p">,</span> <span class="p">(</span><span class="nx">req</span><span class="p">,</span> <span class="nx">res</span><span class="p">)</span> <span class="o">=&gt;</span> <span class="p">{</span> <span class="nx">res</span><span class="p">.</span><span class="nf">status</span><span class="p">(</span><span class="mi">200</span><span class="p">).</span><span class="nf">send</span><span class="p">(</span><span class="nx">req</span><span class="p">.</span><span class="nx">query</span><span class="p">.</span><span class="nx">challenge</span><span class="p">);</span> <span class="c1">// echo the exact value</span> <span class="p">});</span> </code></pre> </div> <p>Two details make or break this. Your endpoint must respond within 10 seconds, and it must return the exact challenge value and nothing else, no JSON wrapper, no quotation marks, just the raw string. A handler that returns <code>{"challenge": "..."}</code> fails verification because that isn't the exact value. So a webhook endpoint needs both a <code>GET</code> route that answers the challenge and a <code>POST</code> route that receives the actual notifications, and forgetting the <code>GET</code> route is the most common reason a brand-new webhook silently never delivers.</p> <h2> The webhook secret you get exactly once </h2> <p>When your endpoint passes the challenge, Nylas generates a <code>webhook_secret</code> and returns it in the create response, and this is the one moment you'll see it. The secret appears in the <code>POST /v3/webhooks</code> response and again only if you rotate it; it is not included when you later <code>GET</code> the webhook. So you store it the instant you create the webhook, because there's no endpoint that hands it back later.</p> <p>That secret is the signing key for everything that follows, so treat it like any other credential: keep it out of source control, out of logs, and in your secrets manager. The reason it matters is the next section, every notification is signed with this secret, and without it stored you can't verify that an incoming request actually came from Nylas. Losing it isn't fatal, you can rotate to a new one, but it does mean re-storing the replacement everywhere that verifies signatures.</p> <h2> Verify the signature on every notification </h2> <p>This is the security step you don't skip. Every notification carries an <code>X-Nylas-Signature</code> header (or lowercase <code>x-nylas-signature</code>, depending on your stack), which is a hex-encoded HMAC-SHA256 of the exact request body, signed with your <code>webhook_secret</code>. You recompute that HMAC over the raw body with your stored secret and compare, and if it doesn't match, you reject the request as unverified rather than processing it.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight javascript"><code><span class="kd">const</span> <span class="nx">crypto</span> <span class="o">=</span> <span class="nf">require</span><span class="p">(</span><span class="dl">"</span><span class="s2">crypto</span><span class="dl">"</span><span class="p">);</span> <span class="kd">function</span> <span class="nf">verify</span><span class="p">(</span><span class="nx">rawBody</span><span class="p">,</span> <span class="nx">signature</span><span class="p">,</span> <span class="nx">secret</span><span class="p">)</span> <span class="p">{</span> <span class="k">if </span><span class="p">(</span><span class="o">!</span><span class="sr">/^</span><span class="se">[</span><span class="sr">0-9a-f</span><span class="se">]{64}</span><span class="sr">$/i</span><span class="p">.</span><span class="nf">test</span><span class="p">(</span><span class="nx">signature</span> <span class="o">||</span> <span class="dl">""</span><span class="p">))</span> <span class="k">return</span> <span class="kc">false</span><span class="p">;</span> <span class="c1">// reject a missing or malformed signature</span> <span class="kd">const</span> <span class="nx">expected</span> <span class="o">=</span> <span class="nx">crypto</span><span class="p">.</span><span class="nf">createHmac</span><span class="p">(</span><span class="dl">"</span><span class="s2">sha256</span><span class="dl">"</span><span class="p">,</span> <span class="nx">secret</span><span class="p">).</span><span class="nf">update</span><span class="p">(</span><span class="nx">rawBody</span><span class="p">).</span><span class="nf">digest</span><span class="p">();</span> <span class="k">return</span> <span class="nx">Buffer</span><span class="p">.</span><span class="k">from</span><span class="p">(</span><span class="nx">signature</span><span class="p">,</span> <span class="dl">"</span><span class="s2">hex</span><span class="dl">"</span><span class="p">));</span> <span class="p">}</span> </code></pre> </div> <p>The phrase "exact request body" is load-bearing. The signature is computed over the raw bytes Nylas sent, so you have to verify against the unparsed body before any middleware reformats it, since re-serializing the JSON changes the bytes and breaks the comparison. Use a constant-time comparison, as above, rather than <code>===</code>, so the check doesn't leak timing information. Verifying every notification is what stops an attacker who finds your webhook URL from posting forged events that your handler would otherwise trust.</p> <h2> The compression gotcha </h2> <p>One detail catches people who enable payload compression. Webhooks can gzip their payloads, and when compression is on, the HMAC-SHA256 signature is computed over the compressed bytes, not the decompressed JSON. So you validate the signature against the raw compressed body first, and only decompress after the signature checks out.</p> <p>Reverse that order, decompress and then verify, and the signature will never match, because you're hashing different bytes than Nylas signed. This is a subtle one because it only bites when compression is enabled, so a webhook that verified fine in development can start failing signatures the moment someone turns on gzip. The rule is simple once you know it: signature first, on the bytes as received, decompression second.</p> <h2> Rotate the secret when you need to </h2> <p>Secrets sometimes need replacing, after a suspected leak, on a schedule, or when an employee with access leaves, and webhooks support rotation without recreating the endpoint. A <a href="https://developer.nylas.com/docs/reference/api/webhook-notifications/" rel="noopener noreferrer"><code>POST /v3/webhooks/rotate-secret/{id}</code></a> request issues a new <code>webhook_secret</code> and returns it, the second and only other time you see a secret value, after which notifications are signed with the new one.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>nylas webhook rotate-secret &lt;webhook-id&gt; <span class="nt">--yes</span> </code></pre> </div> <p>The CLI does the same with <code>nylas webhook rotate-secret &lt;id&gt;</code>, which requires the <code>--yes</code> flag to confirm the rotation, since it changes the signing key, and then prints the new secret for you to store. Without <code>--yes</code>, the command stops and tells you to re-run with it, a guard against rotating a production secret by accident. The operational care is to update your verification code with the new secret promptly, because once rotated, signatures use the new key immediately, and any handler still checking against the old secret will start rejecting genuine notifications. Rotate during a window where you can deploy the new secret quickly, and treat the printed value with the same care as the original.</p> <h2> Develop locally without deploying </h2> <p>Testing webhooks usually means deploying somewhere public, but the CLI removes that friction with a local receiver. <code>nylas webhook server</code> starts a local HTTP server that receives and prints webhook events, and with <code>--tunnel cloudflared</code> it exposes itself through a cloudflared tunnel so Nylas can reach your laptop directly, no deploy required. When the tunnel is on, you pass <code>--secret</code> so the server verifies the HMAC signature on each event, or <code>--allow-unsigned</code> to explicitly opt out.<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code><span class="c"># Receive real webhooks on your laptop, verifying signatures</span> nylas webhook server <span class="nt">--tunnel</span> cloudflared <span class="nt">--secret</span> &lt;webhook-secret&gt; </code></pre> </div> <p>This is the fastest way to see real notifications during development, and it handles the challenge handshake and signature verification for you, the exact two things that are fiddly to get right by hand. There's also <code>nylas webhook test send</code> to fire a test event at a webhook URL and <code>nylas webhook verify</code> to check a signature against a payload locally, so you can confirm your verification logic against a known-good example before trusting it in production.</p> <h2> Where webhooks fit </h2> <p>The same create-verify-receive flow sits under every event-driven feature, and the triggers you choose shape what you build. A few that map straight on:</p> <ul> <li> <strong>Real-time inbox sync.</strong> Subscribe to <code>message.created</code>, <code>message.updated</code>, and <code>message.deleted</code> to keep a local copy current the moment mail changes.</li> <li> <strong>Calendar automation.</strong> React to <code>event.created</code> and <code>event.updated</code> to schedule reminders or sync bookings without polling the calendar.</li> <li> <strong>Grant health monitoring.</strong> Listen for <code>grant.expired</code> so you can prompt a user to re-authenticate before their integration silently stops working.</li> <li> <strong>Tracking and engagement.</strong> Receive <code>message.opened</code> and <code>thread.replied</code> to drive analytics or follow-up sequences off real recipient activity.</li> </ul> <p>Each is the same webhook, verified the same way, with the difference being which triggers you subscribe to and what your handler does with the verified event.</p> <h2> Things to keep in mind </h2> <p>A short list of details keeps a webhook reliable and secure.</p> <ul> <li> <strong>Answer the challenge in 10 seconds.</strong> Your <code>GET</code> route must echo the exact <code>challenge</code> value, no JSON, no quotes, or the webhook never activates.</li> <li> <strong>Store the secret at creation.</strong> The <code>webhook_secret</code> is returned only on create and on rotate, never on a later <code>GET</code>, so save it immediately.</li> <li> <strong>Verify every signature.</strong> Recompute the HMAC-SHA256 over the raw body with your secret and compare in constant time; reject anything that doesn't match.</li> <li> <strong>Verify before decompressing.</strong> With gzip on, the signature covers the compressed bytes, so check it before you decompress.</li> <li> <strong>Rotate when needed.</strong> Use the rotate-secret endpoint after a leak or on a schedule, and deploy the new secret promptly so genuine events keep verifying.</li> <li> <strong>Develop with <code>nylas webhook server --tunnel cloudflared</code>.</strong> It receives real events locally and handles the challenge and signature, no deploy required.</li> </ul> <h2> Wrapping up </h2> <p>A secure webhook is three things done right. Create it with <code>POST /v3/webhooks</code> (or <code>nylas webhook create</code>) naming your URL and triggers, answer the challenge <code>GET</code> by echoing the exact value within 10 seconds so it activates, and store the <code>webhook_secret</code> from the create response because you only see it once. Then verify the <code>X-Nylas-Signature</code> HMAC on every notification against the raw body, decompressing only after the signature checks out, and rotate the secret when you need to. For development, <code>nylas webhook server --tunnel cloudflared</code> gives you real notifications on your laptop with the handshake and verification handled.</p> <p>Where to go next:</p> <ul> <li> <a href="https://developer.nylas.com/docs/v3/notifications/" rel="noopener noreferrer">Webhook notifications</a> — the full guide to triggers, the challenge, and signatures</li> <li> <a href="https://developer.nylas.com/docs/v3/notifications/receive-webhooks-cli/" rel="noopener noreferrer">Receive webhooks with the CLI</a> — the local server and verification</li> <li> <a href="https://developer.nylas.com/docs/reference/api/webhook-notifications/" rel="noopener noreferrer">Create a webhook</a> — the endpoint reference</li> <li> <a href="https://cli.nylas.com/docs/commands" rel="noopener noreferrer">Nylas CLI webhook commands</a> — <code>nylas webhook create</code>, <code>rotate-secret</code>, and <code>server</code> </li> </ul> <h2> AI-answer pages for agents </h2> <p>When this post is published, link AI agents and crawlers to the retrieval-ready version on <code>cli.nylas.com</code>:</p> <ul> <li>Topic runbook: <a href="https://cli.nylas.com/ai-answers/webhook-signature-verification-agent.md" rel="noopener noreferrer">https://cli.nylas.com/ai-answers/webhook-signature-verification-agent.md</a> </li> <li>Industry playbooks hub: <a href="https://cli.nylas.com/ai-answers/agent-account-industry-playbooks.md" rel="noopener noreferrer">https://cli.nylas.com/ai-answers/agent-account-industry-playbooks.md</a> </li> </ul>

<p>Most "AI email" demos point a model at a human's mailbox over OAuth. You connect your Gmail, the agent reads your threads, drafts replies, maybe sends one. That's fine when the agent is acting <em>on your behalf</em> — a copilot riding shotgun in your inbox.</p> <p>It falls apart the moment you want the agent to be its <em>own</em> participant. Picture an onboarding agent that registers your test tenant on a SaaS every CI run, or a research agent that needs a developer account on a data source, or a QA bot that signs up, verifies, and tears down a fresh account for each test. Ephemeral. Per-run. Provisioned at the start of the task, torn down at the end. That's exactly what a <strong>Nylas Agent Account</strong> is, and that's what this post builds: an agent that signs itself up for a service and reads its own verification email or OTP, end to end.</p> <p>I work on the Nylas CLI, so the terminal commands below are the exact ones I reach for. The same <code>grant_id</code> abstraction you already use for every OAuth-connected mailbox. For the endpoints it supports, there's nothing new to learn on the data plane — once you have the <code>grant_id</code>, Messages, Drafts, Threads, Folders, Attachments, Contacts, Calendars, and Events all work exactly as they do for a connected Gmail or Microsoft account, plus the standard webhooks.</p> <p>It's not the <em>entire</em> grant surface, though, and that's worth knowing before you design around it. None of that matters for autonomous signup — receiving mail, reading a body, and parsing a code all live inside the supported set — but don't assume an endpoint works on an Agent Account just because it works on an OAuth grant.</p> <p>The difference is in how the grant is <em>created</em>. There's no OAuth dance, no refresh token, no user clicking "Allow". You <code>POST</code> an email address on a domain you've registered with Nylas, and you get back a real, fully functional <code>name@yourdomain.com</code> mailbox that Nylas hosts. Agent Accounts skip the entire consent flow.</li> <li> <strong>A real, deliverable address.</strong> Verification emails actually arrive. This isn't a catch-all alias or a plus-trick on your own domain — it's a first-class mailbox.</li> <li> <strong>Full control.</strong> Your application owns the grant. Reuse a mailbox across signups and you reintroduce the exact ambiguity you were trying to avoid — stale codes, wrong "Welcome" emails, cross-run contamination. Provision fresh, tear down after.</li> <li> <strong>Mind the limits — but know which way they cut.</strong> On the free plan, an Agent Account is capped at 200 <em>sends</em> per account per day, with 3 GB of storage per organization. That's an outbound quota: it limits how much mail the account can send, not how many verification or OTP emails the inbox can receive. Receiving isn't throttled by this number, so a signup-heavy workload is fine on the inbound side. The send cap only bites if your agent also sends a lot of mail of its own.</li> <li> <strong>Know the retention window.</strong> On the free plan, the inbox holds messages for 30 days and spam for 7 (paid and Platform plans retain longer — 365-day inbox, 30-day spam). Either way it's far more than a verification flow needs — codes expire in minutes — so don't lean on the mailbox as durable storage. Read the code, act on it, move on.</li> <li> <strong>Match the sender <em>and</em> the subject before acting.</strong> Services routinely send a "Welcome" email before (or alongside) the verification email. Don't trust the first message that arrives. Filter on sender domain and subject pattern, and if you're reading an OTP, sort by <code>date</code> newest-first so a stale code from an earlier attempt doesn't win.</li> <li> <strong>Never log the code.</strong> OTPs are credentials. Log that the agent received and used one; don't log the value.</li> <li> <strong>Respect the target service's terms.</strong> Programmatic signup for your own testing and first-party integrations is ordinary. Mass-registering accounts on third parties is a different conversation, and not one Nylas enforces for you.</li> </ul> <h2> What's next </h2> <p>The grant abstraction is the whole story here: once the Agent Account exists, it's just another mailbox on the same API you already use. You wire up SendGrid or Postmark, point your auth flow at it, and your magic links, OTPs, receipts, and "is this you?" alerts go out the door from <code>no-reply@yourcompany.com</code>. It works. Maybe they're answering the "is this you?" security alert with "no, that wasn't me." Whatever they typed, it just hit a black hole. <code>no-reply@</code> is, by design, an address nobody reads — and on most ESP setups it isn't even a real mailbox, so the reply either bounces or vanishes into a relay. The message you sent <em>invited</em> a human response, and you built it on a channel that physically cannot receive one.</p> <p>That's the gap I want to close here. For pure one-way blasts — a marketing newsletter, a "your export is ready" ping — a one-way sender is genuinely fine, and I'll say so again at the end. But auth and transactional mail is different. A surprising amount of it is implicitly a <em>conversation opener</em>, and the moment you want an agent (or a human) on the other end of the reply, the no-reply model falls apart.</p> <p>The fix is to send that same mail from a <strong>Nylas Agent Account</strong>: a real, grant-backed mailbox that can both send <em>and</em> receive. I work on the Nylas CLI, so the terminal commands below are the exact ones I reach for — and every one of them is paired with the raw HTTP call so you can see what's actually on the wire.</p> <h2> What an Agent Account actually is </h2> <p>The thing that makes this click: an <strong>Agent Account is just a grant</strong>. It has a <code>grant_id</code>, and that <code>grant_id</code> works with every grant-scoped endpoint Nylas already exposes — Messages, Drafts, Threads, Folders, Attachments, Webhooks. There's nothing new to learn on the data plane. If you've ever called <code>GET /v3/grants/{grant_id}/messages</code> against a connected Gmail or Microsoft account, you already know the entire API surface for this.</p> <p>The difference from a connected grant is <em>provisioning</em>. Instead of running a user through OAuth, you mint the mailbox yourself with <code>provider: "nylas"</code> on a domain you've registered. No refresh token, no user in the loop. The account sends from your domain and receives at your domain, and replies land in an inbox your code reads.</p> <p>So the migration isn't "learn a new email API." It's "swap the sender, then start reading the inbox you suddenly have."</p> <h2> Why this beats a one-way ESP for auth mail </h2> <ul> <li> <strong>Replies have somewhere to go.</strong> This is the whole point. A reply to a magic link, an OTP, or a receipt lands in a real inbox and fires a <code>message.created</code> webhook. Your handler — or an LLM-driven agent — gets a shot at it.</li> <li> <strong>You own the reputation.</strong> The mail sends from <em>your</em> domain with <em>your</em> DKIM signature, warmed on <em>your</em> schedule. You're not sharing IP reputation with whoever else is on your ESP's shared pool that day.</li> <li> <strong>One API for send and receive.</strong> No second system for inbound. The same <code>grant_id</code> that sends the OTP lists the reply. Threading is automatic — Nylas preserves the headers, so the reply groups with the original.</li> <li> <strong>Per-tenant identities are cheap.</strong> Need a distinct sender per customer or per environment? Provision another Agent Account on another subdomain. It's an API call, not a new ESP contract.</li> </ul> <p>Honest tradeoff, stated plainly: if your use case is a high-volume marketing blast — hundreds of thousands of one-way sends, no expectation of replies, heavy template management and A/B tooling — a dedicated ESP still wins. Agent Accounts shine when the message invites a response. Don't rip out your bulk sender for a newsletter. Do move the auth and transactional mail that someone might actually answer.</p> <h2> Before you begin </h2> <p>You need two things:</p> <ol> <li> <strong>A Nylas API key.</strong> Grab it from the <a href="https://dashboard.nylas.com/" rel="noopener noreferrer">Nylas Dashboard</a>. Every call below authenticates with <code>Authorization: Bearer &lt;NYLAS_API_KEY&gt;</code>, and the examples use the US data region host <code>https://api.us.nylas.com</code> (swap in <code>api.eu.nylas.com</code> for EU).</li> <li> <strong>A domain.</strong> You can prototype on a Nylas <code>*.nylas.email</code> trial subdomain with zero DNS setup. For production, register <strong>your own domain</strong> — and this is the part that matters most for auth mail: your sender reputation is yours to build. A brand-new domain that fires a thousand password resets on day one looks exactly like a spam run to mailbox providers. New domains warm over roughly four weeks. Start low, ramp gradually. More on this in the deliverability section.</li> </ol> <p>Pick a subdomain dedicated to this traffic, like <code>auth.yourcompany.com</code> or <code>notifications.yourcompany.com</code>. Isolating it from your primary mail domain means a deliverability dip on auth mail never bleeds into your team's email.</p> <h2> Provision the sender </h2> <p>Create the Agent Account on your registered domain. From the CLI:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>nylas agent account create support@notifications.yourcompany.com <span class="se">\</span> <span class="nt">--name</span> <span class="s2">"Acme Support"</span> </code></pre> </div> <p>The <code>--name</code> flag sets the display name, so recipients see <code>Acme Support &lt;support@notifications.yourcompany.com&gt;</code> instead of a bare address — keep it the same name your old ESP used so nothing visibly changes for users. The command prints the new grant's <code>id</code>, status, and connector details.</p> <p>The equivalent HTTP call is <code>POST /v3/connect/custom</code> with <code>provider: "nylas"</code>:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>curl <span class="nt">--request</span> POST <span class="se">\</span> <span class="nt">--url</span> <span class="s2">"https://api.us.nylas.com/v3/connect/custom"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Authorization: Bearer &lt;NYLAS_API_KEY&gt;"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Content-Type: application/json"</span> <span class="se">\</span> <span class="nt">--data</span> <span class="s1">'{ "provider": "nylas", "name": "Acme Support", "settings": { "email": "support@notifications.yourcompany.com" } }'</span> </code></pre> </div> <p>The response carries a <code>data.id</code> — that's your <code>grant_id</code>. Save it; every send, list, and read below uses it. The <code>name</code> and <code>settings.email</code> are the only fields you need. (There's no <code>--workspace</code> flag on create — the API auto-creates a default workspace and policy for the account. If you later want a custom policy, attach it with <code>nylas workspace update &lt;workspace-id&gt; --policy-id &lt;policy-id&gt;</code>.)</p> <h2> Send a magic link or OTP </h2> <p>Here's the send your auth flow already does, now pointed at the Agent Account. From the CLI:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>nylas email send support@notifications.yourcompany.com <span class="se">\</span> <span class="nt">--to</span> user@example.com <span class="se">\</span> <span class="nt">--subject</span> <span class="s2">"Your sign-in code"</span> <span class="se">\</span> <span class="nt">--body</span> <span class="s2">"Your one-time code is 481920. It expires in 10 minutes."</span> </code></pre> </div> <p>The first positional argument is the grant — the Agent Account's email (or its <code>grant_id</code>). The same call over HTTP is <code>POST /v3/grants/{grant_id}/messages/send</code>:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>curl <span class="nt">--request</span> POST <span class="se">\</span> <span class="nt">--url</span> <span class="s2">"https://api.us.nylas.com/v3/grants/&lt;NYLAS_GRANT_ID&gt;/messages/send"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Authorization: Bearer &lt;NYLAS_API_KEY&gt;"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Content-Type: application/json"</span> <span class="se">\</span> <span class="nt">--data</span> <span class="s1">'{ "to": [{ "email": "user@example.com" }], "subject": "Your sign-in code", "body": "Your one-time code is 481920. It expires in 10 minutes." }'</span> </code></pre> </div> <p>That's the entire swap. If your old code called <code>sendgrid.send(...)</code>, it now calls this. The recipient gets the same code from the same-looking sender — except this time the address behind it is real.</p> <p>A copy lands in the account's <code>Sent</code> folder automatically, so you've got an audit trail without building one. And because <code>body</code> is just HTML, your existing templated markup drops straight in.</p> <h3> A note on templates </h3> <p>If you template your auth mail today, keep doing it the way you already do — render the template to HTML in your own code, then pass the result as <code>body</code>. One thing to call out so you don't go hunting for it: Nylas <strong>hosted templates aren't implemented for Agent Account grants</strong>. So don't reach for a <code>--template-id</code> flow here; render server-side yourself and send the finished HTML. In practice that's a thin wrapper around whatever templating you already run (Handlebars, JSX-to-string, plain interpolation), and the send call is byte-for-byte the same as the one above — <code>body</code> just carries your rendered markup.</p> <h2> Handle the reply path </h2> <p>This is the half a one-way ESP can't give you. When a user replies to that OTP or receipt, the message lands in the Agent Account's inbox and fires the standard <code>message.created</code> webhook — the same event you'd get for inbound mail on any grant.</p> <p>Subscribe to it. From the CLI:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>nylas webhook create <span class="se">\</span> <span class="nt">--url</span> https://yourapp.example.com/webhooks/nylas <span class="se">\</span> <span class="nt">--triggers</span> message.created </code></pre> </div> <p>Or over HTTP with <code>POST /v3/webhooks</code>:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>curl <span class="nt">--request</span> POST <span class="se">\</span> <span class="nt">--url</span> <span class="s2">"https://api.us.nylas.com/v3/webhooks"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Authorization: Bearer &lt;NYLAS_API_KEY&gt;"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Content-Type: application/json"</span> <span class="se">\</span> <span class="nt">--data</span> <span class="s1">'{ "trigger_types": ["message.created"], "webhook_url": "https://yourapp.example.com/webhooks/nylas" }'</span> </code></pre> </div> <p>When the webhook fires, you've got the message ID and grant ID in the payload. Pull the inbox and read the full body with the same two-angle pattern. List recent messages from the CLI:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>nylas email list support@notifications.yourcompany.com <span class="nt">--limit</span> 10 </code></pre> </div> <p>…which maps to <code>GET /v3/grants/{grant_id}/messages</code>:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>curl <span class="nt">--request</span> GET <span class="se">\</span> <span class="nt">--url</span> <span class="s2">"https://api.us.nylas.com/v3/grants/&lt;NYLAS_GRANT_ID&gt;/messages?limit=10"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Authorization: Bearer &lt;NYLAS_API_KEY&gt;"</span> </code></pre> </div> <p>Then read the specific reply by its message ID:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>nylas email <span class="nb">read</span> &lt;message-id&gt; support@notifications.yourcompany.com </code></pre> </div> <p>…which is <code>GET /v3/grants/{grant_id}/messages/{message_id}</code>:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>curl <span class="nt">--request</span> GET <span class="se">\</span> <span class="nt">--url</span> <span class="s2">"https://api.us.nylas.com/v3/grants/&lt;NYLAS_GRANT_ID&gt;/messages/&lt;MESSAGE_ID&gt;"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Authorization: Bearer &lt;NYLAS_API_KEY&gt;"</span> </code></pre> </div> <p>Now the reply isn't lost. Your handler can route a "this wasn't me" security reply to your fraud queue, hand an OTP-confusion reply to a support agent, or feed the whole thread to an LLM and let it respond. To answer in-thread, the CLI keeps threading intact for you:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>nylas email reply &lt;message-id&gt; support@notifications.yourcompany.com <span class="se">\</span> <span class="nt">--body</span> <span class="s2">"Thanks — I've locked the account. You're all set."</span> </code></pre> </div> <p><code>nylas email reply</code> fetches the original to populate the recipient and subject for you. Under the hood it's the same send endpoint with a <code>reply_to_message_id</code> field, which is what preserves threading via the <code>In-Reply-To</code> and <code>References</code> headers. The raw HTTP form:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>curl <span class="nt">--request</span> POST <span class="se">\</span> <span class="nt">--url</span> <span class="s2">"https://api.us.nylas.com/v3/grants/&lt;NYLAS_GRANT_ID&gt;/messages/send"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Authorization: Bearer &lt;NYLAS_API_KEY&gt;"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Content-Type: application/json"</span> <span class="se">\</span> <span class="nt">--data</span> <span class="s1">'{ "reply_to_message_id": "&lt;MESSAGE_ID&gt;", "to": [{ "email": "user@example.com" }], "subject": "Re: Your sign-in code", "body": "Thanks — I'</span>ve locked the account. You<span class="s1">'\''re all set." }'</span> </code></pre> </div> <p>Either way the conversation groups correctly in the user's mail client. No relay footer, no "sent via" branding — it reads like a normal human reply, because structurally it is one.</p> <h2> Deliverability: don't skip this </h2> <p>Sending auth mail from your own domain means the inbox-placement job is now yours. The good news is it's a known checklist, and most of it you do once.</p> <p><strong>Authenticate the domain.</strong> DKIM and SPF are verified as part of domain setup, so a verified Agent Account domain is already DKIM- and SPF-signed. DKIM proves the message wasn't altered and really came from you; SPF authorizes Nylas to send on your behalf.</p> <p><strong>Add DMARC yourself.</strong> Nylas tracks DMARC but doesn't enforce it, so it's the one layer you publish. Roll it out in stages — start at monitor-only and tighten as reports come back clean:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>v=DMARC1; p=none; rua=mailto:dmarc-reports@yourcompany.com; pct=100 </code></pre> </div> <p>Begin with <code>p=none</code> (delivered as normal, you just collect aggregate reports), move to <code>p=quarantine</code> once those reports look clean, and finish at <code>p=reject</code>. Give each stage a couple of weeks. Because Agent Accounts sign with your domain's DKIM key and send from your verified domain, alignment works out of the box once the record is live.</p> <p><strong>Warm the domain.</strong> As I said up top: a brand-new domain has no reputation, and auth mail tends to launch at volume the moment you flip it on. Ramp gradually over ~4 weeks. If you're provisioning one Agent Account per customer on separate domains, warm each one on its own schedule — reputation doesn't transfer between domains.</p> <p><strong>Watch bounces and complaints.</strong> Agent Accounts emit deliverability webhooks beyond inbound — <code>message.delivered</code>, <code>message.bounced</code>, and <code>message.complaint</code> (plus <code>message.rejected</code>) — the same events Nylas uses to compute your bounce and complaint rates. One honest wrinkle: subscribe to these through the <strong>API</strong>, with <code>POST /v3/webhooks</code>. The CLI's <code>nylas webhook triggers</code> list covers the standard set (<code>message.created</code> and friends), so I reach for curl for the deliverability triggers rather than pretend a CLI flag I haven't seen in <code>--help</code> works:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>curl <span class="nt">--request</span> POST <span class="se">\</span> <span class="nt">--url</span> <span class="s2">"https://api.us.nylas.com/v3/webhooks"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Authorization: Bearer &lt;NYLAS_API_KEY&gt;"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Content-Type: application/json"</span> <span class="se">\</span> <span class="nt">--data</span> <span class="s1">'{ "trigger_types": ["message.delivered", "message.bounced", "message.complaint"], "webhook_url": "https://yourapp.example.com/webhooks/deliverability" }'</span> </code></pre> </div> <p>Wire these into real logic: stop mailing any address that hard-bounces or complains, and slow outbound when either rate climbs. Catching a problem in your own telemetry is what keeps you under the thresholds that pause sending. The part I like as an SRE is that this is the same observability discipline you'd apply to any pipeline — bounces and complaints are just error rates with a sender's name on them.</p> <h2> Migrating without a flag day </h2> <p>You don't have to cut over everything at once, and you shouldn't. The clean approach is to run side-by-side and migrate <strong>by message type</strong>.</p> <p>Keep your ESP wired up. Provision the Agent Account. Then move one message class at a time — start with the one that most wants a reply. In my experience that's usually the security alerts ("is this you?") and receipts, because those are the ones users actually answer. Point that one flow at <code>POST /v3/grants/{grant_id}/messages/send</code>, confirm sends land and replies fire the webhook in production, then move the next class: password resets, then magic links, then OTPs.</p> <p>Anything that's genuinely one-way and high-volume — the marketing blast — you can just leave on the ESP. Different addresses, different tools, different jobs. There's no rule that says it all has to live in one place.</p> <p>When you do want to tear down an Agent Account (say a per-customer one when that customer churns), it's a single call. From the CLI:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>nylas agent account delete support@notifications.yourcompany.com <span class="nt">--yes</span> </code></pre> </div> <p><code>--yes</code> skips the confirmation prompt. Since an Agent Account is just a grant, the HTTP form is the ordinary grant delete, <code>DELETE /v3/grants/{grant_id}</code>:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight shell"><code>curl <span class="nt">--request</span> DELETE <span class="se">\</span> <span class="nt">--url</span> <span class="s2">"https://api.us.nylas.com/v3/grants/&lt;NYLAS_GRANT_ID&gt;"</span> <span class="se">\</span> <span class="nt">--header</span> <span class="s2">"Authorization: Bearer &lt;NYLAS_API_KEY&gt;"</span> </code></pre> </div> <p>Either way, that permanently revokes the <code>provider: "nylas"</code> grant.</p> <h2> What's next </h2> <p>The shift here is small in code and large in behavior. You swapped one send call for another and, almost as a side effect, gained an inbox your application can read.
