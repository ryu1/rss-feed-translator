WordPress Pre-Launch Checklist for Client Handoff — What to Check, Clean Up, and Prove Before Delivery

原文説明（英語）

<p>A few years ago, at an agency I worked at, we launched a corporate WordPress site by migrating the whole database from our development environment to production. The demo had looked perfect. The client had signed off.</p> <p>The reports started coming in almost immediately — first from our own post-launch review, then from the client's side as they clicked through their brand-new site. Buttons that led nowhere. Images that wouldn't load. Our local dev URLs were hardcoded all over the site: pasted into button links, baked into image paths. Nothing had <em>looked</em> wrong, because on our machines, those URLs worked fine.</p> <p>Three of us spent the rest of that day clicking through every single page, hunting for leftovers. We fixed what we found and quietly hoped we'd found them all. No one yelled at us. The consequence was quieter than that, and heavier: the string of post-launch defects was treated as a formal incident, with a permanent entry in the company's internal incident list. That entry doesn't expire. As far as I know, it's still there.</p> <p>　</p> <h2> The handoff problem </h2> <p>Handoff is the moment <strong>responsibility changes hands — the most dangerous, and the most important, moment in the entire project.</strong></p> <p>Clients trust the agency. They take it for granted — as they should — that the site they're receiving is solid. But that trust is easier to betray than anyone likes to admit. The moment a client finds a defect, doubt creeps in. However hard the team worked, a single careless leftover is enough to make them wonder: <em>what does the quality of this site really look like?</em></p> <p>There's a structural reason this happens. While you're building, you're in the admin every day — you'd notice a stray test post, a debug flag, a weird setting. After handoff, the person in the admin is your client. They edit content, upload images, and take everything at face value. The leftovers that were invisible in the demo are now <em>their</em> discoveries.</p> <p>Most launch checklists focus on whether the site works. A handoff checklist asks a different question: <strong>what will the client find that I should have found first?</strong></p> <p>Here's the list I wish I'd had back then. Every item includes how to check it manually — no tools required.</p> <p>　</p> <h2> 1. Content cleanup </h2> <p><strong>Test posts.</strong> About an hour after one delivery, a chat message arrived from the client: <em>"What is this 'test test' page?"</em> Chat messages don't carry tone, but I could feel the reaction through the screen. Test content is harmless to the site and terrible for trust.<br /> <em>How to check:</em> In the admin, go to <strong>Posts</strong> and <strong>Pages</strong>, sort by date, and scan for anything you created as filler — "test", lorem ipsum, placeholder drafts. Don't forget custom post types and the trash.</p> <p><strong>Unused media.</strong> On another project, a stock photo was still carrying the vendor's <em>watermark</em> — we had never replaced it with the licensed version. The client spotted it right before delivery, which is one step better than after delivery, and still one step worse than us spotting it ourselves.<br /> <em>How to check:</em> Open <strong>Media Library</strong>, switch to list view, and go through everything uploaded during development. Look for placeholders, duplicates, screenshots, and anything watermarked.</p> <p><strong>Temporary pages.</strong> Style-guide pages, block catalogs, layout sandboxes — anything you built for your own reference.<br /> <em>How to check:</em> <strong>Pages → All Pages</strong>, review anything the client never asked for.</p> <p>　</p> <h2> 2. Launch settings </h2> <p><strong>Search engine visibility (the noindex trap).</strong> This one has two layers, and the second one almost got me. The obvious layer is <strong>Settings → Reading → "Discourage search engines from indexing this site"</strong> — staging environments have it on, production must not. The sneaky layer is your <em>SEO plugin</em>: on one site, the Reading setting was fine, but a noindex rule was still sitting in the SEO plugin's configuration from the staging days. Weeks of indexing time are lost this way, silently.<br /> <em>How to check:</em> Verify the Reading setting, then open your SEO plugin's settings and check its index rules. Finally, view the page source of the live homepage and search for <code>noindex</code> in the <code>&lt;meta name="robots"&gt;</code> tag — that's the ground truth.</p> <p><strong>WP_DEBUG.</strong> Debug mode on a production site can print warnings — with file paths — right onto the page for every visitor.<br /> <em>How to check:</em> Open <code>wp-config.php</code> and confirm <code>WP_DEBUG</code> is <code>false</code>.</p> <p><strong>Admin email.</strong> If the site's admin address is still yours, the client's password resets and system notices go to you forever.<br /> <em>How to check:</em> <strong>Settings → General → Administration Email Address.</strong></p> <p><strong>Permalinks / rewrite rules.</strong> After a migration, stale rewrite rules are a classic source of unexplained 404s.<br /> <em>How to check:</em> <strong>Settings → Permalinks → Save Changes</strong> (yes, just re-saving flushes the rules), then click through a few posts and archive pages.</p> <p>　</p> <h2> 3. Security loose ends </h2> <p>Four small things that take five minutes total and are exactly the kind of leftovers a security scan (or a curious visitor) finds later:</p> <ul> <li> <strong>File editor</strong> — <code>DISALLOW_FILE_EDIT</code> in <code>wp-config.php</code> disables the built-in theme/plugin code editor. A client account clicking around in there is a risk nobody needs.</li> <li> <strong>XML-RPC</strong> — if nothing uses it (Jetpack, some mobile apps), it's an open door for brute-force amplification. Disable it via filter or plugin.</li> <li> <strong>debug.log</strong> — a <code>wp-content/debug.log</code> file left web-accessible is a quiet information leak. Delete it and confirm logging is off.</li> <li> <strong>The 'admin' username</strong> — a default <code>admin</code> account is half of a credential-stuffing attempt already solved. Rename or replace it.</li> </ul> <p>　</p> <h2> 4. Site health before delivery </h2> <p><strong>Outdated plugins.</strong> Taking over or delivering a site with plugins that haven't been updated on WordPress.org in over a year means handing the client a maintenance liability without telling them.<br /> <em>How to check:</em> For each installed plugin — including inactive ones — check its "Last updated" date on its WordPress.org page. Flag anything past 6 months; treat anything past a year as a conversation you need to have with the client before handoff, not after.</p> <p><strong>Inactive plugins.</strong> They still ship code. If it's not needed, it shouldn't ship.<br /> <em>How to check:</em> <strong>Plugins → Inactive</strong>, and delete what has no reason to stay.</p> <p>　</p> <h2> 5. Beyond the basics </h2> <p>Depending on the hosting stack, a few more things deserve a look — these are harder to automate, which is exactly why you want the basics above done quickly and reliably:</p> <ul> <li> <strong>Cache purge after the final content change.</strong> A page cache that never got purged will serve the old price, the old phone number, the old everything — with zero errors logged.</li> <li> <strong>Leftover staging credentials.</strong> Test admin accounts and staging logins have a talent for surviving into production.</li> <li> <strong>Hardcoded dev URLs in the database.</strong> My opening story. A search-replace pass over the database (with a proper serialized-data-aware tool) before launch would have saved us an afternoon of page-by-page hunting — and a permanent line in an incident log. <em>How to check:</em> Don't trust the search-replace tool alone. Click through the primary navigation, footer links, and CTA buttons on the live site — and open a few image-heavy pages — right after migration.</li> </ul> <p>　</p> <h2> 6. The missing step: prove what you checked </h2> <p>Here's the part almost no checklist talks about.</p> <p>Running the checks protects the site. <strong>Showing the checks protects the relationship.</strong> A client can't see the noindex flag you flipped back or the debug log you deleted — to them, a clean launch and a lucky launch look identical. The difference only becomes visible when something breaks.</p> <p>So make the invisible work visible. Before handoff, write a short plain-text delivery note:<br /> </p> <div class="highlight js-code-highlight"> <pre class="highlight plaintext"><code>Pre-launch verification — example.com — 2026-07-02 ✓ Test content removed (posts, pages, media) ✓ Search engine indexing enabled (Reading settings + SEO plugin + meta verified) ✓ WP_DEBUG disabled ✓ Admin email set to client ✓ File editor locked, XML-RPC disabled, no debug.log present ✓ Default 'admin' username: not present ✓ Plugin update status reviewed (2 flagged, listed below) </code></pre> </div> <p>Attach it to your delivery email. <h1> How TypeScript Infers Types Through Async Generators in 2026 </h1> <p>Most async generator type errors stem from TypeScript's fundamental constraint: the compiler cannot infer what calling code will send to <code>yield</code> expressions. This limitation creates silent type holes in codebases that rely on async generators for streaming operations, pagination, or event processing.</p> <p>The async generator pattern appears deceptively simple. Developers write <code>async function*</code> expecting TypeScript to infer types from yielded values. Instead, TypeScript assigns <code>any</code> to yield expressions because the type depends on external callers, not the generator's implementation. This distinction is critical. Without explicit annotations, teams ship generators that accept arbitrary input types through <code>.next()</code> calls, bypassing type safety at the exact boundary where validation matters most.</p> <h2> Understanding AsyncGenerator Type Parameters </h2> <p>TypeScript models async generators with three type parameters that control different phases of execution. The first parameter <code>T</code> represents yielded values — what consumers receive from <code>await iterator.next()</code>. The second parameter <code>TReturn</code> specifies the final return value when the generator completes. The third parameter <code>TNext</code> controls what calling code can send back through <code>.next(value)</code>.</p> <p>This three-parameter structure maps directly to generator control flow. When a generator yields a value, TypeScript uses <code>T</code> to type the promise resolution.
